## 문제와 왜 지금인가

에이전트와 함께하는 장기 연구·개발은 세션이 바뀔 때마다 방향·결정·검증 증거를 잃는다 — 대화는 압축되고, 메모리 파일은 흩어지고, "완료했습니다"는 남지만 그것을 뒷받침한 증거는 남지 않는다. waystone은 이 intent drift와 context loss를 막기 위해 존재한다.

멀티에이전트 작업의 규모가 커지면서 네 가지 손실이 반복적으로 관찰되었다(실제 세션 로그 분석으로 확인된 패턴이다).

- main session이 탐색·구현·반복 디버깅까지 직접 떠안아 희소한 reasoning budget과 context를 소모한다. 반대로 단순한 작업의 과잉 위임은 전달 손실과 재시도로 총비용을 키운다.
- "이 작업을 누구에게 맡길 것인가"가 prose 지시와 매 순간의 모델 판단에 의존해, 역할 경계와 최종 책임이 흐려진다.
- 동일한 preset이 어떤 프로젝트에서는 유용한 guard이고 다른 프로젝트에서는 마찰이다. CUDA 연구 코드와 문서 중심 프로젝트는 요구되는 검증·환경·위험 수준이 다르다.
- session log의 성공/실패 신호는 품질을 말해주지 않는다. API 오류 없이 끝난 작업에도 심각한 결함이 있을 수 있다.

이 문제를 "더 강한 프롬프트 하나"로 풀지 않는다. 관측, 분석, delegation, evidence, review, policy adaptation을 분리하고 연결하는 운영체계로 푼다.
