# waystone

## 문제와 왜 지금인가

에이전트와 함께하는 장기 연구·개발은 세션이 바뀔 때마다 방향·결정·검증 증거를 잃는다 — 대화는 압축되고, 메모리 파일은 흩어지고, "완료했습니다"는 남지만 그것을 뒷받침한 증거는 남지 않는다. waystone은 이 intent drift와 context loss를 막기 위해 존재한다.

멀티에이전트 작업의 규모가 커지면서 네 가지 손실이 반복적으로 관찰되었다(실제 세션 로그 분석으로 확인된 패턴이다).

- main session이 탐색·구현·반복 디버깅까지 직접 떠안아 희소한 reasoning budget과 context를 소모한다. 반대로 단순한 작업의 과잉 위임은 전달 손실과 재시도로 총비용을 키운다.
- "이 작업을 누구에게 맡길 것인가"가 prose 지시와 매 순간의 모델 판단에 의존해, 역할 경계와 최종 책임이 흐려진다.
- 동일한 preset이 어떤 프로젝트에서는 유용한 guard이고 다른 프로젝트에서는 마찰이다. CUDA 연구 코드와 문서 중심 프로젝트는 요구되는 검증·환경·위험 수준이 다르다.
- session log의 성공/실패 신호는 품질을 말해주지 않는다. API 오류 없이 끝난 작업에도 심각한 결함이 있을 수 있다.

이 문제를 "더 강한 프롬프트 하나"로 풀지 않는다. 관측, 분석, delegation, evidence, review, policy adaptation을 분리하고 연결하는 운영체계로 푼다.

## 비전

**Agents forget. Projects shouldn't.** — waystone은 여러 모델과 도구가 명확한 책임 아래 협업하게 만드는 개발 운영 하네스이며, 고정된 predefined harness에서 출발해 관측과 review evidence를 근거로 사용자·프로젝트별 정책을 축적하는 **adaptive harness로 진화하는 체계**다.

이것이 의미하는 것: 프로젝트에는 세션보다 오래 사는 방향 문서(SSOT)와 검증된 task 목록이 있고, 작업은 경계 있는 round로 진행되며, 위임된 작업은 독립 검증과 기록된 verdict 없이는 인수되지 않고, 과거 세션은 다음 워크플로 개선의 근거가 된다.

이것이 의미하지 않는 것: 모델을 세세하게 통제하는 규칙 모음이 아니다. 고난도 작업에서 모델의 추론 능력과 유연성을 억누르는 hard gate의 나열은 이 비전의 실패 형태다.

## 누구를 위한 것인가

일차 사용자는 저자 자신 — 여러 머신과 두 호스트(Claude Code, Codex)에 걸쳐 장기 연구·소프트웨어 프로젝트를 병행하는 솔로 개발자이며, 고용하는 job은 "세션·에이전트·머신이 바뀌어도 프로젝트가 방향과 증거를 기억하게 하라"는 것이다.

이 위상 규정은 장식이 아니라 모든 트레이드오프의 지배 변수다: 사업의 명운을 건 제품이 아니므로 robustness 투자, readiness 기준의 보수성, 통계적 엄밀성 요구 수준은 개인 도구에 맞게 정한다. 커뮤니티에서 쓰이게 된다면 훌륭한 보너스이지, 설계를 왜곡할 목표가 아니다.

## 원칙

로드맵이 모호해질 때 판단을 가르는 값들이다. 각 원칙은 그럴듯한 무언가에 "아니오"라고 말할 수 있어야 한다.

- **품질 우선, 효율 최적화.** 정확성·연구 품질 → 검증 가능성 → 작업 지속성 → 자원 효율 순. 최종 품질을 약화시키는 절감은 성공이 아니다. 고난도 판단을 비용 때문에 하위 모델로 내리는 것도, 상위 모델이 기계적 잡무를 직접 하는 것도 모두 위반이다.
- **자기보고보다 evidence.** 완료 선언, exit code, API success는 품질의 증거가 아니다. 변경된 파일, 실행된 검증, review finding, remediation 이력이 근거다. 기능은 선언이 아니라 실제 소비 지점까지 이어져야 구현된 것이다.
- **모델 이름보다 역할.** 설계 단위는 main·orchestrator·implementer·clerk·verifier·reviewer 같은 책임이고, 모델은 profile이 binding한다. 구독이나 모델 세대가 바뀌면 binding만 바뀌어야지 워크플로를 재설계해야 한다면 설계 위반이다.
- **반복 가능한 단계는 스크립트, 판단은 모델.** 검증·렌더링·bookkeeping·로그 파싱은 결정론적 스크립트가 사실(facts)을 만들고, 모델은 그 해석만 담당한다. bookkeeping에 모델 토큰이 들어가면 설계가 잘못된 것이다.
- **점진적 강제.** 정책은 observe → recommend → warn → enforce 단계를 거치며, 승격에는 evidence와 사용자 consent가 필요하다. hard enforcement는 의미가 명확하고 false positive 가능성이 낮은 경우로 한정한다.
- **Local-first, 비파괴.** 행동 데이터(trace·evidence·overlay)는 로컬에 남고 공유 repo에 커밋되지 않는다; 공유는 consent-gated 물질화 경로뿐이다. 기존 프로젝트의 이력은 보존되고, 생성물은 검토를 위해 uncommitted로 남는다.
- **Silent fallback 금지.** 이름과 다른 동작으로 성공을 가장하지 않는다. 파싱 실패·증거 부재·불가한 지표는 조용히 0이나 생략이 되지 않고 이유와 함께 드러난다.
- **단순성 — 단 증거 무결성은 예외.** 불안정한 외부 포맷에 방어적 버저닝·어댑터 레이어를 쌓지 않고 깨지면 고친다. 그러나 기록의 무결성(silent drop 없음, 커버리지 보고, fail-loud)은 이 절약의 대상이 아니다.

## Non-goals

waystone이 의도적으로 되지 않으려는 것들 — 범위가 번질 때 되돌아올 경계선이다.

- **비용 최소화 도구가 아니다.** 가장 싼 실행자를 찾는 것이 아니라 최소 충분 역량을 배치하고 독립 검증하는 것이 목적이다.
- **범용 프로젝트 관리·CI 플랫폼이 아니다.** 이슈 트래커, merge queue, 이벤트 인박스, 자체 호스팅 러너 같은 인프라의 재발명은 하지 않는다(실증 분석에서 반복적으로 기각된 방향이다).
- **모델·호스트 종속 도구가 아니다.** 특정 모델 이름이나 단일 호스트를 전제한 기능은 만들지 않는다.
- **관측을 인과로 팔지 않는다.** 세션 로그의 상관 패턴은 후보일 뿐이며, 정책의 품질 이득 주장은 적용 후 live evidence로만 성립한다. shadow replay는 발동률·마찰 추정 도구이지 이득 증명이 아니다.
- **습관의 모방 학습이 아니다.** 관찰된 행동을 그대로 정책화하지 않는다. review finding과 검증 evidence로 실제 문제로 확인된 패턴만 delta 후보가 된다.
- **상용 수준 방어를 목표하지 않는다.** 위협 모델과 robustness는 솔로 개발자의 신뢰 경계에 맞춘다.

## 방향성 베팅

지금의 설계를 지탱하는 큰 wager들 — 반대 증거가 쌓이면 decision을 열어 재검토할 대상이다.

- **SSOT anchor bet.** 프로젝트당 하나의 살아있는 방향 문서 + 생성된 view가, 흩어진 메모리·채팅 요약보다 장기 방향 유지에 우월하다. SSOT는 binding but falsifiable — 증거가 모순되면 조용한 이탈이 아니라 decision → ADR로 개정한다.
- **3축 binding bet.** 역할 × 실행 방식 × backend의 분리가, 모델 이름 중심 설정보다 모델 교체·호스트 추가에 견고하다.
- **Adaptive overlay bet.** 강한 기본값 위의 얇은 정책 overlay(좁은 범위가 넓은 범위를 덮고, 모든 delta에 근거·범위·상태가 남는)가, preset 자체를 계속 변형하는 것보다 설명 가능하고 되돌릴 수 있다.
- **Evidence-gated autonomy bet.** 위임 루프는 인간 게이트가 아니라 하네스가 강제하는 인수 게이트(기준별 verdict, 독립 검증, 변조 봉쇄) 위에서 자율적으로 돌 수 있으며, 인간 개입은 열거된 escalation 사유로 예약하는 편이 매 단계 승인보다 안전하고 빠르다.
- **Dual-host single-state bet.** Claude Code와 Codex가 같은 프로젝트 상태를 공유하는 것이, 호스트별 독립 상태보다 워크플로의 수명을 길게 만든다.
- **Dogfooding bet.** waystone 자신의 개발을 waystone으로 운영하는 것이 가장 정직한 검증 경로다 — 자기 데이터에서 개선이 측정되지 않으면 adaptive 비전 자체를 의심해야 한다.

## 열린 질문

진짜로 미결인 것들 — 이 절은 결함이 아니라 여유 공간이다.

- **Enforce의 정당화 경계.** 어떤 증거 수준이면 관측된 check가 실제로 작업을 차단해도 되는가? waiver의 ergonomics(기록·만료·남용 방지)는 어떤 형태여야 마찰이 이득을 넘지 않는가?
- **Scale-up의 소유권 경계.** 병렬 task group을 열 때 main session이 cross-task 결정의 단일 소유자로 남는 구조가 어디까지 유지되는가?
- **자기증명.** longitudinal metrics가 실제로 "round가 쌓일수록 routing·verification·review policy가 개선된다"를 보여줄 것인가? 보여주지 못하면 무엇을 접을 것인가?
- **일반화 투자 수준.** 개인 도구 위상을 유지하면서 어디까지 타인이 쓸 수 있는 형태(문서·onboarding·안전 기본값)에 투자할 것인가?
- **멀티머신 학습.** 머신별 독립 학습을 유지할 것인가, 언젠가 evidence의 선택적 공유 경로를 열 것인가?
