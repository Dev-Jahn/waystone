## 0. 합성 기준과 판정 불일치 해소

**골격 = cross-host-first (judge 2명 일치).** 공통 결정 근거: (1) `delegate verdict`의 "기준 집합 원문 정확 일치" 검사가 판정 시점의 기준 바꿔치기를 기계적으로 차단하는 유일한 설계, (2) `--accept-add`가 기준 합성을 락이 보호하는 CLI 경로 위에 유지 — 다른 두 안은 자신이 잔여 위험으로 지목한 tasks.yaml 손편집 경로로 합성을 보내는 자기모순을 가짐, (3) 마이그레이션이 유일하게 전면 무손실(보존+전건 보고, `.pre-0.9` rename, 고아 목록 출력).

**이식 항목(judge가 명시 지정, 전부 채택):**

**판정 불일치의 명시적 해소 3건:**

- **(i) 머신 루트 컨벤션** — J0는 "XDG triple을 단일 루트로 교체"를 지시, J1은 "XDG 유지 + 단일 루트를 fallback으로 문서화"를 지시. **단일 `~/.waystone` 채택.** 근거: 두 judge 모두 XDG triple을 승자의 최대 약점으로 지목했고, 헌장 "simplest correct"가 결정 기준이다(해석 규칙 1개·마이그레이션 타깃 1개 vs 3개). J1의 실질 우려(백업/청소 구분, 경로 가시성)는 `cache/` 서브디렉토리 분리 + conventions.md의 tier별 가이드(#11) + `waystone paths` verb로 전부 충족되므로 XDG를 유지할 잔여 이유가 없다.
- **(ii) `--reason`의 범위** — J1은 apply·discard 양쪽 CLI 필수를 지시했으나, apply는 필수 verdict 아티팩트가 rationale과 기준별 판정을 이미 구조적으로 강제한다(플래그보다 강한 게이트). **채택: discard와 모든 `--override-*` 플래그에 `--reason` CLI 필수, apply는 verdict 게이트로 충족.** J1의 목적("worktree 파괴 전에 결론이 기록된다")은 두 경로 모두에서 기계 게이트로 성립한다.
- **(iii) profile 스키마 bump** — J0가 cross-host-first의 profile-2 강제 재작성을 약점으로 지목("execution-key-optional would do"). **채택: `execution` 키를 optional-deprecated로** — 부재 시 host 유도(정상 경로), 존재+유도값 일치 시 경고와 함께 수용, 불일치 시 fail-loud. 마이그레이션이 키를 자동 제거+보고하므로 사용자 재작성이 강제되지 않아 스키마 문자열은 `waystone-profile-1` 유지.

그 외 이식은 전부 가산적(상호 충돌 없음)이어서 그대로 채택했다.

## 1. 저장 구조

### 1.1 원칙: scope당 저장 위치 1개, `WAYSTONE_HOST`는 경로를 선택하지 않는다

cross-host 분열의 뿌리는 단 하나 — `common.data_dir()`의 host 분기다(scripts/common.py:19-26). 이 분기 아래 모든 상태가 매달려 있어(resume/start_here: common.py:245-257, delegations/worktrees/profile: delegate.py:167-189, overlay/exposure: overlay.py:125·543-544, registry: common.py:59, improve: improve.py:1344-1345) 내용상 host-특정이 아닌 상태 전부가 기계적으로 두 벌이 된다. 0.9 이후 `WAYSTONE_HOST`는 **동작**(verifier transport, trace 소스/파서, stanza 대상, 문구)만 선택하고 **저장 경로는 절대 선택하지 않는다.**

gitignore 전략: `.waystone/` 최초 생성 시 CLI가 내부에 `.gitignore`(내용 `*`) 생성 — self-ignore 패턴. 사용자의 최상위 `.gitignore` 무접촉, 지워져도 다음 실행에서 재생성. (autonomy-first의 "커밋되는 .gitignore 편집"은 두 judge 모두 기각)

**Tier M — 머신 전역: `~/.waystone/` (`$WAYSTONE_HOME`로 override)**

```
~/.waystone/
  projects.json     # 신규 `waystone project register`가 유일한 작성자 (§2)
  registry.lock
  profile.yml       # 공유 프로파일 (1.3)
  improve/          # sessions/delegations/reviews/evidence.jsonl + *_coverage.json + decisions.jsonl

```

근거: projects.json은 머신 절대경로의 프로젝트 목록(common.py:59; 현재는 init 스킬이 모델 손편집으로 작성 — skills/init:133-138), improve/는 전 프로젝트 스캔 결과이며 decisions.jsonl은 재생성 불가한 사용자 판단(improve.py:1287-1295). 단일 루트 선택 근거는 §0 해소 (i).

**Tier C — 캐시: `~/.waystone/cache/worktrees/<slug>/<did>/`**

worktree를 프로젝트 안에 두지 않는 이유: nested checkout 혼동, `git status`/IDE 오염, `git clean` 시 무거운 증거 소실. 캐시 경로가 공유되면서 **"Codex가 만든 delegation을 Claude가 verify"가 처음으로 성립**한다(verify는 worktree 안에서 codex를 실행 — delegate.py:913-957). slug는 프로젝트 횡단 네임스페이스라 여기서만 유지.

**Host-귀속으로 남는 것(전부 read-only 입력 또는 transport — 저장 아님):**

| 항목 | 근거 |

### 1.3 profile.yml 통합

파일 전체에서 host-발산 필드는 verifier `execution` 하나뿐이고(codex-companion vs codex-cli — skills/delegate/SKILL.md:62-63), 그 값은 `WAYSTONE_HOST`로 완전 결정되므로 저장 자체가 중복이다:

- `_resolve_verifier_binding()`(delegate.py:244-264): `execution` 부재 시 host에서 유도. 존재 시 유도값과 일치하면 "deprecated — remove" 경고와 함께 수용, 불일치하면 fail-loud(`remove the execution key to let waystone derive it`). 스키마 문자열 `waystone-profile-1` 유지 (§0 해소 iii).
- fail-loud 불변: profile 부재 시 host-맞춤 템플릿과 함께 거부(delegate.py:204-218, _profile_example 67-70), non-codex backend 거부(267-273), byte fingerprint의 exposure 기록(delegate.py:535). 파일이 하나가 되므로 fingerprint도 호스트 간 동일해져 감사가 단순해진다.

### 1.4 코드 경로 계층

`scripts/common.py`:

```python
def machine_dir() -> Path             # $WAYSTONE_HOME 또는 ~/.waystone
def project_state_dir(root) -> Path   # root/.waystone (+ self-.gitignore 보장)
def worktrees_cache_dir() -> Path     # machine_dir()/cache/worktrees
def registry_path() -> Path           # machine_dir()/projects.json — 함수
```

- `data_dir()` 해체. **`REGISTRY_PATH`의 import-time 동결(common.py:59) 동시 수정** — dashboard.py:26·116-119가 유일한 상수 소비자(improve.py:748-749는 이미 런타임 재해석 중이라 함수 호출로 통일).
- 신규 조회 verb **`waystone paths [--root .] [--json]`**: 해석된 실경로 출력. 스킬/문서는 리터럴 경로 대신 이 verb를 참조 — `waystone resume --start-here-path`(resume.py:64-80)가 증명한 "스킬은 경로를 CLI에 묻는다" 패턴의 일반화. 경로가 또 바뀌어도 스킬은 무풍.

### 1.5 마이그레이션 계획

0.8.1 `migrate_home_data()` 패턴(waystone.py:45에서 매 진입 호출; move-if-absent, 충돌 보존+보고 — common.py:33-51) 재사용·확장. 전 과정 **삭제 없음** — 레거시 루트는 `~/.claude/waystone.pre-0.9`, `~/.codex/waystone.pre-0.9`로 rename 보존.

**Phase 1 — 머신 scope (eager, 디스패처 진입 시 1회, `registry.lock` 하)**

1. 소스: `~/.claude/waystone`(primary), `($CODEX_HOME|~/.codex)/waystone`(secondary). 둘 다 없으면 no-op.
2. **projects.json**: path/repo 키 기준 결정적 union, 병합 내역 전건 stderr 출력. 0.8.1 never-merge 원칙의 명시적 완화 — 그때는 같은 데이터의 중복 루트였고 지금은 상호보완 레지스트리이며, union은 무손실·결정적·전건 보고(no-merge는 "프로젝트가 대시보드에서 조용히 사라지는" silent 결과라 오히려 헌장 위반). 두 judge 모두 승인.
3. **profile.yml**: execution 키 제거 후 투영 기준 — 한쪽만 존재 → 변환 이동, 양쪽 동일 → 이동, 양쪽 상이 → **아무것도 쓰지 않고 fail-loud**(두 경로+템플릿 출력, human-authored 파일은 추측 병합 금지 — "모델을 추측하지 않는다" 정신).
4. **improve/**: claude 쪽 이동. codex 쪽 재생성 가능 투영(sessions/delegations/reviews/evidence)은 보존 루트에 남기고 재생성 안내(`waystone improve trace --host codex`). **decisions.jsonl은 양쪽 존재 시 timestamp 순 concat**(append-only 로그의 순서 병합은 무손실·결정적, 행 수 보고).
5. 레지스트리로 매핑 불가한 slug 디렉터리는 보존 루트에 남기고 목록 출력(해시 역산 불가 → 추측 금지).
6. 버전 스큐 fail-safe: 레거시 평문 루트 존재를 매 진입 시 재검사(존재 검사 1회 비용) — 스큐 기간에 구버전 플러그인이 재생성한 상태도 다음 신버전 실행이 회수.

**Phase 2 — 프로젝트 scope (lazy, root 해석 시마다: 모든 CLI 명령 + session_context hook)** — J0·J1 공통 이식

- 트리거: `.pre-0.9` 보존 루트에 `_project_slug(root)` 하위 항목 존재 시. 이동하면 소스가 사라지므로 자기소멸적 멱등(hook 경로 비용 무시 가능). 레지스트리 미등록 프로젝트도 자연 커버(slug 역산 불가 문제의 해법).
- resume(ephemeral): newer만 취함. 그 외 파일 충돌(양쪽 존재+내용 상이): newer-mtime을 live로, 패자는 `{root}/.waystone/migration-conflicts/<host>/`에 무손실 보존, 건별 stderr.
- **예외 1**: 동일 overlay rule-id delta가 양쪽 존재 → **fail-loud, 사람이 선택**(mtime 중재는 suspend된 warn을 조용히 부활/억압할 수 있음 — 이식 #4).
- **예외 2**: 동일 did가 양쪽 존재(타임스탬프 키라 사실상 불가) → 해당 did skip+보존+경고.
- worktrees: `git -C <project> worktree move <old> <new>` 우선(gitdir back-link 네이티브 갱신), 실패 시 move+`git worktree repair`, 지속 실패 시 해당 레코드 discard-only degrade + 대문 안내(이식 #3).
- Codex 호스트도 마이그레이션 수행 가능 — 0.8.1의 "codex는 이동 금지" 제약(common.py:38)은 폐기(목적지가 host-중립이라 누가 옮겨도 결과 동일). 전 과정이 락 하라 claude+codex 동시 최초 실행도 안전.

## 2. Lock 메커니즘

### 2.1 메커니즘: `fcntl.flock` (커널 advisory lock)

concurrency-audit 결론 채택. flock은 (i) 프로세스가 죽으면 커널이 자동 해제 → **stale-lock 처리 코드가 통째로 불필요**, (ii) claude/codex가 **같은 경로**를 열기만 하면 cross-host 직렬화 — §1이 이 전제를 만들었다(현행은 data_dir 분열로 락을 공유할 수 없음이 audit의 전제 조건 1), (iii) Python `subprocess`의 `close_fds=True` 기본값으로 codex/git 자식에 락이 새지 않음, (iv) **repo에 이미 검증된 패턴 존재**(delegate.py:857-894 `_acquire_verify_lock`). `lockf`/POSIX record lock은 "아무 fd나 닫으면 해제" 풋건으로 기각, CAS(digest 재확인)는 round close 같은 다중 파일 불변식을 못 덮어 기각. NFS는 "locking assumes a local filesystem; network-mounted homes are unsupported" 한 줄 문서화로 대응하고 엔지니어링하지 않는다.

`common.hold_lock(path, timeout)` 단일 프리미티브: `open("a+")` + `flock(LOCK_EX|LOCK_NB)` 100ms 간격 재시도, 기본 10초(`WAYSTONE_LOCK_TIMEOUT` env), 획득 시 `{pid, host, verb, at}`를 진단용으로만 기록(권위는 flock 자체), 락 파일은 절대 unlink하지 않음(unlink 레이스 원천 차단).

### 2.2 락 3개 — 전부다

| 락 | 경로 | 잡는 커맨드 | 보호 대상 (audit 고위험 항목) |
|---|---|---|---|
| project lock (coarse) | `{root}/.waystone/lock` | `task add/set/drop/archive` 전체, `round close` 전 구간(preflight read→SSOT regen→rollback: round.py:174-255), `overlay add/promote/demote/suspend/retire`+replay의 load→write(overlay.py:263-309, 373-401, 244), `delegate run`의 scan→claim 구간만(delegate.py:450-471→577), exposure 기록, roadmap/tasks_guard regen | tasks.yaml RMW(#1: tasks.py:198/219), round close 다중파일 커밋+롤백 클로버(#2), archive 2-파일 이동(#3: tasks.py:230-268), delegate owner-lock check-then-act(#4), overlay 전이의 warn 부활(#7), exposure suffix 레이스 |
| record lock | `<record>/record.lock` (기존 verify.lock 개명·일반화) | `delegate verify`(현행 유지) + **apply·discard·verdict**의 check→mutate 전 구간 | apply vs discard vs verify vs verdict 상호 레이스(#5: delegate.py:986, 1009) |
| registry lock | `~/.waystone/registry.lock` | 신규 `waystone project register/unregister/list`, 마이그레이션 Phase 1 | projects.json RMW(#6) — 모델 손편집을 CLI verb로 회수하는 것이 실효 조건 |

**중요 경계**: `delegate run`은 claim 직후 project lock을 **해제**하고 나서 수 분짜리 `codex exec` runner를 돌린다(락 보유 중이면 그동안 `task set`까지 막힘). claim 이후는 per-task owner-lock 상태기계(delegate.py:416-469)가 단일 소유를 보장. 공짜 수준의 보강 2건: record dir claim을 `mkdir(exist_ok=False)`로(EEXIST → suffix 증가 재시도 — did 충돌 루프 571-573 해소), `git update-ref refs/waystone/delegations/<did> <sha> ""`(빈 old-value = must-not-exist CAS)로 ref 생성 충돌을 시끄럽게 검출.

**동반 원자화(락과 한 세트, crash-tearing 방지)**: tasks.yaml/tasks.archive.yaml/.waystone.yml/ROADMAP.md → tmp+`os.replace`; exposure round-N → `open('x')` suffix 재시도(delegate.py:897-910 패턴 재사용, 현행 plain write overlay.py:577-581 교체); resume.md → `os.replace`; review ingest(review.py:635) → `open('xb')` + `--force`(같은 라운드 재수집 거부가 애초에 올바른 의미론).

### 2.3 실패 UX

- 타임아웃 시 **락 없이 진행하지 않고**(no silent fallback) exit 1: `waystone: {root}/.waystone/lock is held by pid 4242 (codex, round close, since 12:03:11) — retry after it finishes, or raise WAYSTONE_LOCK_TIMEOUT`.
- force-break verb는 만들지 않는다 — 홀더 사망 시 커널이 자동 해제하므로 "강제 해제" 개념 자체가 불필요하고, 있으면 오용된다.
- hook(tasks_guard)만 3초 타임아웃: 초과 시 **경고 출력 후 regen 생략, exit 0** — ROADMAP은 derived·self-healing(roadmap.py:129, tasks_guard.py:83; 다음 편집에서 복구)이라 조용하지 않은 degrade가 정답.
- 락 순서 규칙(데드락 방지): `registry → project → record` 순서로만 획득. 실제로 둘을 동시에 잡는 코드는 마이그레이션뿐.

### 2.4 의도적으로 잠그지 않는 것 (audit 판정 인용)

- `warnings.jsonl`/`decisions.jsonl`: O_APPEND 소행 원자성으로 충분(overlay.py:403-412, improve.py:1293).
- improve 투영: 재생성 가능+byte-stable, 유일한 live 소비자(session_context.py:91-121)가 이미 관용적.
- SSOT views: byte-diff 검출(ssot.py:175-202)+멱등 재생성, round close 경로는 project lock 상속.
- `start-here.md`: 단일 작성자 관행(작성 트리거인 round close가 락 아래) — last-writer-wins 문서화.
- **잔여 위험 명시**: advisory lock은 모델이 Edit 도구로 tasks.yaml을 직접 고치는 경로를 못 막는다. 완화는 락이 아니라 "뮤테이션은 CLI verb로"라는 기존 방향의 지속이며, projects.json이 이번에 그 방향으로 회수되는 첫 사례.

테스트는 위험 직결분 3개만(동시성 재현 테스트는 flaky): (1) 선점된 락 하에서 `WAYSTONE_LOCK_TIMEOUT=0.2`로 `task set`이 holder 정보를 담아 exit 1, (2) 동일 task 이중 `delegate run`에서 mkdir claim 한쪽만 승리, (3) exposure `open('x')` suffix 무덮어쓰기.

## 3. Delegate 자율화

### 3.1 전제

증거 지도가 확인한 핵심: **사람 게이트는 전부 prose다** — task 선택 ask(skills/delegate/SKILL.md:22-26), verify 제안 ask(59-71), apply/discard ask(74-83)는 스킬 산문이고, CLI verb는 이미 무인 안전·증거 기록형이다(delegate.py). 주입되는 운영 계약은 이미 "Main owns routing, acceptance, and final decisions"라고 말한다(references/main-contract.md:1-8) — **자율화는 계약 위반이 아니라 계약의 이행**이며, owner 헌장("Main session owns task routing, hard decisions, and final acceptance")과도 정합. 따라서 본체는 스킬 재작성 + verb 1개 + 기계 게이트이고, 헌장 조항("no silent fallback", "verification evidence before acceptance")을 산문에서 **harness가 거부하는 전제조건으로 승격**한다.

### 3.2 새 루프 (사람 프롬프트 0회)

```
SELECT → ENSURE-CRITERIA → RUN → (실패: TRIAGE/RETRY/ESCALATE)
       → VERIFY(필수) → JUDGE(verdict 기록) → APPLY | DISCARD → REPORT
```

1. **SELECT** — Step 1의 ask 삭제. main agent가 next-actionable에서 직접 선택. 기계 검사(pending/active만, deps — delegate.py:285-289)는 불변.
2. **ENSURE-CRITERIA** — 빈 기준 거부(delegate.py:290-296)는 **불변**(harness는 여전히 바를 발명하지 않는다). agent 합성은 허용하되 경로를 하나로 고정: 신규 반복 가능 플래그 **`waystone task set <id> --accept-add "<criterion>"`** — 콤마 분리 없음(기존 accept 거부 사유였던 free-text 왜곡을 해소, tasks.py:48-52), project lock 하에서 tasks.yaml에 영속 → **커밋 diff가 감사 표면**. packet.yaml에 `accept_provenance` 기록. 합성은 owner 산출물(태스크 설명·ROADMAP·SSOT·리뷰 이력)에서 도출 가능한 경우만 — 도출 불가면 에스컬레이션. `--accept`는 사람의 ad-hoc 기준용으로 유지. (양 judge의 결정적 채택 사유: 합성이 락 보호되는 CLI 경로에 남는다.)
3. **RUN** — 기계부 무변경. warn은 "사람에게 원문 릴레이"에서 **판정 agent의 정책 입력 + 최종 보고 수록**으로 전환. warnings.jsonl 트레일(overlay.py:404-412)과 exit-code 불변성(테스트 4198)은 유지.
4. **실패 처리** — 신규 한정 표면 **`waystone delegate show <did> --failure`**(status.json error + runner.stderr tail — 이식 #8; runner.jsonl 전체 금독 규율 유지). 일시적/환경성 → `discard --reason "<진단 결론>"` 후 재시도(**worktree 파괴 전 결론 기록** 순서를 기계화). 결정적 실패 → discard하지 않고(증거 보존) 에스컬레이션. **재시도 상한: 태스크당 세션당 총 2회.** 재시도는 항상 신규 레코드(task_id로 연결)이며 `delegate run --note`가 packet에 `retry_context`(provenance: main-session)로 기록(이식 #9).
5. **VERIFY** — ask 삭제, verifier 바인딩 존재 시 **무조건 실행**. 불변식 전부 보존: read-only sandbox, `verify-N.json` provenance `independent-verifier`, 상태 무전이(테스트 4815), record lock. 바인딩 부재 시: agent가 정규화된 worktree에서 **한정된 커맨드 실행 증거**를 만들어 verdict의 `agent_checks[]`로 기록(이식 #6) — 검증 증거 없는 수락은 어느 경로로도 불가.
6. **JUDGE — 신규 verb `waystone delegate verdict <did> --file <verdict.json>`.** "contract에는 verdict가 없다" 불변식(delegate.py:630-643, conventions.md:151-156)은 유지 — 판정은 contract 밖의 제4 provenance 아티팩트다.
7. **APPLY | DISCARD** — 아래 3.4.
8. **REPORT** — 사람에게 보이는 유일한 표면. 내부 용어 없이(헌장): 무엇을 했나, 기준별 판정+증거 한 줄, verifier finding 심각도 카운트, agent가 작성한 기준 표시, warn 원문, 재시도 이력, 레코드 경로, 되돌리기 방법(`git apply -R .waystone/delegations/<did>/artifact/changes.patch` — 레코드 영구 보존이라 가능).

### 3.3 verdict 아티팩트 (스키마 `waystone-verdict-1`)

```json
{
  "schema": "waystone-verdict-1",
  "decision": "apply | discard",
  "decided_by": "main-session | user",
  "criteria": [{"criterion": "<packet accept 항목 원문>", "met": true,
                "evidence": ["verify-2.json#finding-3", "agent_checks[0]", "contract:changed_files"]}],
  "agent_checks": [{"cmd": "uv run scripts/tests/run_tests.py", "exit": 0, "summary": "310 passed"}],
  "warnings_seen": ["delegation-verification-evidence-v1 fired at delegate-run"],
  "rationale": "...", "limitations": ["..."]
}
```

`decided_by`는 에스컬레이션 후 사람이 결정한 경우에도 같은 게이트를 지나게 한다 — **단일 게이트, 두 저자**(이식 #5). harness가 **기계적으로 거부**하는 것(전부 fail-loud, record lock 하):

1. state ≠ needs-review.
2. `criteria` 집합이 packet.yaml의 accept 집합과 **원문 정확 일치**하지 않음(누락/추가/부분 판정 → 거부) — 판정 시점의 기준 바꿔치기 차단.
3. 검증 증거 게이트: verifier 바인딩 존재 또는 이 레코드의 run에서 rule1 발화(warnings.jsonl 조회 — overlay.py:465-466) 시 `verify-N.json` ≥ 1 필수; 바인딩 부재 시 `agent_checks` ≥ 1 필수. — **"verification evidence must be recorded before acceptance"의 기계화.**
4. `decision: apply`인데 최신 verify에 미해소 `blocker` finding → 거부. 통과는 인간 전용 `--override-blocker --reason`뿐(stderr 대문 + verdict `overrides[]` 기록).

통과 시 `artifact/verdict-<n>.json`을 `open('x')` 넘버링으로 기록(delegate.py:897-910 패턴) + harness 부가 필드(provenance: `main-session`, 참조 verify 번호, profile_fingerprint, timestamp). verdict는 상태를 바꾸지 않는다(verify와 동일한 기록/전이 분리).

**provenance 4종**: `harness-computed`(contract의 sha/changed_files) / `delegate-claimed`(여전히 사실 승격 금지; "보고 부재 ≠ 미검증" 뉘앙스 유지) / `independent-verifier`(verify-N.json) / **`main-session`(verdict-N.json, 신규)**.

### 3.4 apply / discard 게이트

- **apply**: 최신 verdict 존재 && `decision == apply`일 때만. 없으면 거부("run `waystone delegate verdict` first"). 인간 탈출구 `--override-no-verdict --reason`(명시적+기록 — silent fallback 아님). 역학 불변: needs-review 전용(delegate.py:987; 재적용 거부 테스트 3637), **non-3-way `git apply`, drift 시 원자적 exit 1**(991-995; 테스트 3602-3616) — 3-way/auto-stash 폴백은 영구 금지(헌장), 성공 시 cleanup(968-977) + 레코드 영구 보존.
- **discard**: `--reason` **CLI 필수**(§0 해소 ii; status.json 전이 엔트리에 기록 — delegate.py:402-412). verdict 게이트는 없음 — 크래시 잔해·corrupt 레코드 청소 경로 보존(1002-1014, 테스트 3646-3673). 판정에 의한 discard는 verdict(decision: discard) 선기록을 스킬이 지시.

### 3.5 잔존 안전 레일 (명시 체크리스트)

owner lock(태스크당 비종결 delegation 1개) + corrupt-record fail-safe(delegate.py:450-471) — 무변경 · profile/바인딩 fail-loud(204-218) — 무변경 · 빈 accept 거부 — 무변경 · verify sandbox/provenance/무전이/record lock — 무변경 · apply 원자성 + 레코드 보존 + `git apply -R` 가역성 — 무변경 · warnings/exposure 증거 트레일 — 무변경 · context discipline(skills/delegate:85-89) — 완화 3건만: 실패 진단은 `show --failure` 경유, "when the user asks" → "특정 기준 판정에 필요할 때", 바인딩 부재 시 worktree 내 한정 커맨드 실행 허용(wholesale 읽기는 계속 금지) · SessionStart의 "needs-review delegations N" 표면(session_context.py:72-88)은 유지하되 스킬 지시가 "물어볼 일"에서 "재개해 verdict까지 끝낼 일"로.

### 3.6 에스컬레이션 트리거 (전수 — 이 외에는 사람에게 묻지 않는다)

1. accept 기준을 owner 산출물에서 도출 불가(공허한 바 발명 금지)

## 5. 열린 결정사항 (owner만 결정 가능)

1. **`.waystone/`의 `git clean -fdx` 취약성 수용** — delegation 감사 레코드와 start-here 서사가 유실될 수 있다. 설계는 "문서화+수용"(refs·worktree는 생존)이지만, 감사 번들의 보존 요구 수준이 더 높다면 레코드만 머신 루트에 남기는 절충이 있다(cross-host 단순성 일부 상실).
2. **머신 루트 이름** — 설계는 `~/.waystone`(+`$WAYSTONE_HOME`). judge 근거로 단일 루트를 채택했으나 `~/.config/waystone` 선호라면 이름만 바꾸면 된다(구조 불변).
3. **agent의 accept 기준 합성 허용 범위** — 설계는 "owner 산출물에서 도출 가능할 때만, `--accept-add`로 영속+provenance 기록, 불가 시 에스컬레이션". 전면 금지(항상 owner 작성 — 자율 루프가 더 자주 멈춤)로 좁힐지, tasks.yaml 안에 작성 주체 마커까지 남길지.
4. **`--override-blocker` / `--override-no-verdict` 탈출구 존치** — 전부 `--reason` 필수+기록이지만, 수동 운용성 vs 오용 위험 사이에서 존재 자체를 없앨 수도 있다.
5. **재시도 상한(2회)과 verify 상시 실행의 토큰 비용** — 자율 apply의 전제 비용으로 수용할지, 저위험 태스크에 대한 완화 정책(추후 overlay 규칙화)을 열어둘지.
6. **마이그레이션 트리거와 잔재 정리** — 0.8.1 선례대로 첫 실행 시 자동(waystone.py:45) vs 명시적 1회 명령(이번엔 프로젝트 repo 안에 디렉터리를 만들므로 자동이 부담스러울 수 있음). `.pre-0.9` 보존 루트의 정리 시점도 포함.
7. **(이월 갭) Codex의 PreCompact/SessionEnd 발화 미검증**(hooks/hooks.json:34·46; repo 내 어떤 테스트도 증명하지 않음) — resume.md 공유로 Claude발 스냅샷이 Codex에 보여 부분 완화되나, codex발 스냅샷 자체는 여전히 미보장. 0.9 범위에 포함할지 이월할지.

## 6. 적대적 검증 결과 및 규범 수정 (post-synthesis, 2026-07-15)

합성안을 2-lens(codebase-reality / requirement-fidelity)로 적대 검증한 결과. 아래 수정사항은 본문에 **우선하는 규범**이다 (중복 발견은 병합; major 6·minor 7).

### 6.1 Major — 설계 수정 필수

**M1. `delegate run` double-claim race가 §2.2로는 닫히지 않는다 (양 lens 공통 발견).**
project lock 스팬이 record-dir mkdir(delegate.py:577)에서 끝나지만, owner-lock 스캔(`_iter_delegations`, delegate.py:421-423)은 `exposure.json`이 존재하는 레코드만 본다. exposure.json은 snapshot(`git add -A` temp index)·update-ref·worktree add **이후**(delegate.py:588-591)에야 쓰이므로, 락 해제 후 수 초짜리 창에서 두 호스트가 같은 task를 동시에 claim할 수 있다(did는 timestamp 키라 서로 다름 — mkdir CAS로는 못 막음).
**수정**: 락 안에서 mkdir 직후 최소 claim 파일(task_id 포함)을 기록하고, `_active_delegation_for_task` 스캔이 claim-but-no-exposure 레코드를 lock-holding으로 취급하도록 확장(corrupt-record fail-safe delegate.py:465-469와 같은 형태). 락을 exposure 기록까지 붙들지 않는다(runner 아님에도 snapshot 동안 `task set`이 막히는 건 수용 가능하나, claim 마커가 더 값싸고 스팬 원칙 유지).

**M2. Phase-1 전체 rename이 레거시 worktree의 git back-link를 기계 전역에서 일괄 파괴한다 (스크래치 재현 완료).**
`~/.claude/waystone` → `.pre-0.9` rename 후에는 `git worktree move`가 **절대 성공할 수 없고**(rc=128 — 원경로도 신경로도 not-a-working-tree), `git worktree prune`(delegate.py:977의 `_cleanup`이 apply/discard마다 실행 + 사용자 측 `git gc`)이 admin dir를 지우면 mv+`worktree repair`로도 복구 불가가 된다. Phase 2가 lazy라 미접촉 프로젝트의 worktree가 무기한 파손 상태로 방치되고, needs-review delegation은 verify 경로(delegate.py:927-928)를 잃는다.
**수정**: **Phase-1 rename에서 `worktrees/`를 제외** — 레거시 worktree는 원경로에 그대로 두고, Phase 2에서 프로젝트별로 아직-유효한 원경로로부터 진짜 `git worktree move`를 수행(이때 비로소 이식 #3의 move-first가 성립). 나머지 레거시 루트만 rename.

**M3. `delegate apply`는 record lock만으로는 부족 — project lock도 잡아야 한다 (양 lens 공통 발견).**
apply는 live tree에 `git apply`(delegate.py:991)하는데 §2.2는 record lock만 배정했다. round close의 preflight read(round.py:185)→commit write(round.py:235) 사이나 rollback(round.py:234-255) 스팬에 apply가 끼어들면 patch hunk가 조용히 되돌려진다(record는 'applied'로 남음) — 정확히 owner가 지목한 silent last-writer-wins.
**수정**: apply는 `project → record` 순서로 두 락을 모두 획득(§2.3 순서 규칙과 정합, apply는 빠른 verb라 비용 무시 가능). `delegate run`의 락 경계 논리는 그대로(runner는 락 밖).

**M4. profile 마이그레이션의 execution-키 제거는 verifier 바인딩에만 적용해야 한다.**
implementer 바인딩은 `execution: external-runner`를 하드 요구(delegate.py:228-231)하므로, §1.5 Phase 1-3의 무조건 키 제거는 모든 `delegate run`을 fail-loud로 죽이는 profile을 배송한다 — §0(iii)의 "재작성 강제 없음" 주장과 자기모순.
**수정**: 키 제거·optional-deprecated 규칙 모두 **verifier 바인딩 한정**. implementer는 무접촉.

**M5. verdict 게이트에 "기준 충족" 검사가 빠져 있다.**
§3.3 게이트 1-4는 상태·기준 집합 일치·증거 존재·blocker만 검사 — 전 기준 `met: false`인 verdict로도 apply가 승인된다. owner의 핵심 요구("기준에 대해 판정")가 산문 집행으로 남는 것.
**수정**: **게이트 5 추가** — `decision: apply`는 모든 criterion `met: true` 필수. 탈출구는 `--override-unmet --reason`(overrides[] 기록, blocker 게이트와 동형).

**M6. blocker override의 '인간 전용' 제한은 자율성 요구·헌장과 긴장 관계다.**
헌장 "Main session owns … final acceptance"는 바로 이 어려운 수락 판단을 main agent에 배정하는데, 게이트 4는 verifier의 false-positive blocker(적대 리뷰어에겐 일상) 하나로 잔여 인간 프롬프트를 강제한다. verifier 주장(모델 산출물)을 agent 판단 위에 사실로 승격하는 셈 — delegate-claimed 승격 금지 원칙과도 비대칭.
**수정**: 열린 결정 #4를 확장 — (a) 현행 인간 전용, (b) `decided_by: main-session` override 허용하되 해당 finding을 구체 반박하는 `agent_checks` 인용 필수(overrides[] 기록), (c) 탈출구 삭제. **설계 기본값은 (b)**로 변경하되 owner 결정 대기.

## 7. 방향 수정 (owner 지시, 2026-07-15): profile·improve의 project 단위 재배치

### 7.1 profile — project 단일 (owner 결정: 2-레이어 기각, "쓸데없이 복잡")

**profile = `{root}/.waystone/profile.yml`, 유일한 위치.** 부재 시 기존 fail-loud 템플릿 거부(불변). 머신 default 레이어는 만들지 않는다 — precedence 규칙·레이어 기록·관련 테스트가 통째로 불필요해진다.

project 단일이 정확성을 잃지 않는 근거 2가지:

1. `.waystone/`은 비커밋·체크아웃별 파일이므로 **머신 축이 자동 커버**된다 — 같은 프로젝트라도 머신마다 체크아웃의 profile이 따로 존재. "이 머신엔 codex가 없다" 류는 파일 차원에서 이미 갈라지고, 그래도 남는 불일치는 러너 호출 시점 fail-loud가 정직하게 잡는다.
2. **기존 프로젝트 마찰은 마이그레이션 시딩으로 제거** — 레거시 host profile은 Phase 1에서 이동하지 않고 `.pre-0.9` 보존 루트에 남긴다(§1.5 Phase 1-3 대체). Phase 2(프로젝트별 lazy)가 해당 프로젝트에 profile이 없으면 보존 루트에서 복사해 생성(건별 stderr 보고; verifier `execution` 키 처리는 §6 M4 규칙 적용; 양 레거시 호스트의 profile이 상이하면 해당 프로젝트에서 fail-loud — 기존 충돌 정책 유지). 신규 프로젝트만 템플릿 보고 1회 작성.

exposure 기록은 기존 byte fingerprint 그대로(레이어 필드 불필요).

### 7.2 improve — 기본 모드를 single-project로

- **기본**: `waystone improve trace/audit/...`는 project root 기준으로 **해당 프로젝트의 세션 로그만** 분석(현행 project 필터를 기본값으로 승격), 산출물(`sessions/delegations/reviews/evidence` 투영, facts, coverage)과 **decisions.jsonl**을 `{root}/.waystone/improve/`에 기록. 권고→overlay 승격 루프가 한 프로젝트 안에서 닫힘.
- **옵트인 user-wide 모드**(플래그명 구현 시 결정): cross-project 사용자-습관 렌즈(main-direct-work 비율, 위임 습관 등) 전용, 산출물은 `~/.waystone/improve/`. 프로젝트 렌즈와 습관 렌즈의 분류는 구현 시 렌즈별로 명시(둘 다에 걸치는 렌즈는 각 모드에서 자기 스코프 데이터만).
- 스킬 재포지셔닝: "로그가 충분히 쌓인 프로젝트에서 주기적으로" — round 스킬의 improve 권장 문구도 project 모드 기준으로.
- dashboard(`waystone status`)의 improve 관련 표면이 있으면 레이어 표기.
