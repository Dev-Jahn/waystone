## 0. 결정 기록

**사용자 결정 (2026-07-13):**
- **D1. dirty-state = snapshot commit 자동 생성.** dirty tree여도 위임 가능; git plumbing으로 현재 상태를 히스토리 오염 없는 immutable commit object로 고정해 base로 사용(§4.5 ADR 확정).
- **D2. M1 = 위임 primitive 먼저.** M1 = binding config + `waystone delegate`(snapshot→worktree→env prep→codex runner→artifact contract) + exposure record + 평가 경로. overlay/warn/replay/Tune/evidence 정식화/main contract는 M2+.

**스펙 결정 (본 노트, 각 항 근거 병기):**
- **S1. Snapshot plumbing = temp-index 방식** (`git stash create` 아님), **시딩은 `read-tree HEAD`로 통일**: 스크래치 실검증(2026-07-13) — read-tree-HEAD 시딩과 `.git/index` 복사 시딩이 동일 tree sha 산출(staged+수정+untracked 케이스), 그리고 linked worktree에서는 `.git`이 gitdir pointer **파일**이라 index 복사 방식 자체가 오류. read-tree 시딩은 경로 해석·index 복사가 불필요해 더 단순하고 main repo/worktree 어디서나 동일하게 동작한다. untracked 비무시 파일 **포함**(연구 라운드의 신규 파일 WIP가 일상 — "delegate가 보는 것 = 사용자가 보는 것"), gitignored 제외, worktree/index 무변경, 단일 부모(-p HEAD). stash create는 untracked 제외 + merge commit 구조라 탈락.
- **S2. Result도 같은 primitive로 고정:** runner 종료 시 worktree 최종 상태를 result snapshot commit(-p base snapshot)으로 고정. patch = `git diff --binary --no-renames <base> <result>` — delegate가 worktree 안에서 커밋했든 안 했든 최종 파일 상태 기준으로 결정적. base/result 모두 `refs/waystone/delegations/*` ref로 gc-생존 보장(gc --prune=now 실검증).
- **S3. 자기보고와 하네스 계산의 provenance 분리(#3·#8·#11 정신):** patch·changed_files·base/result sha = **하네스가 git에서 직접 계산(explicit)**. verification·limitations·risks = **delegate-claimed** 로 명시 표기 — delegate의 자기보고를 사실로 승격하지 않는다. 독립 검증은 main/verifier 몫(§12. 평가 경로).
- **S4. 자기보고 수신 채널 = worktree 루트의 `JW_REPORT.yaml`:** prompt가 양식을 지정, runner 종료 후 하네스가 **읽고 제거한 뒤** result snapshot을 계산(패치 오염 방지). 누락/파싱불가 시 contract에 `delegate_report: missing|invalid`로 명기 — 조용히 통과 금지.
- **S5. Acceptance criteria fail-loud(#3):** task 스키마에 선택 필드 **`accept`(진짜 YAML list[str], 요소당 기준 1개) 신설** + `waystone delegate run --accept "..."`(기준당 1회, 반복 가능). 둘 다 없으면 exit 1 — 하네스가 기준을 지어내지 않는다. **comma-split 금지**(자유문 기준이 조용히 쪼개지는 왜곡) — `waystone task add/set`은 accept를 다루지 않고(거부+안내), registry의 accept는 tasks.yaml 직접 편집(쓰기 검증됨)으로만.
- **S6. 단일 mutation owner(#2):** lock 해제(terminal) 상태 = **{applied, discarded}뿐**. failed-env/failed-runner는 worktree가 evidence로 보존되는 동안 lock도 유지 — `discard`로 정리해야 재위임 가능. predicate 정의는 §4가 단일 출처.
- **S7. Env prep = 명시 우선, 표준 감지 후순위:** config `delegation.env_prep`(명령 리스트)가 있으면 그대로, 없으면 lockfile 감지(uv.lock→`uv sync --frozen`, pnpm-lock.yaml→`pnpm install --frozen-lockfile`, package-lock.json→`npm ci`, Cargo.toml→`cargo fetch`, go.mod→`go mod download`; 최초 매치 1개만). 아무것도 없으면 `none-detected`로 **기록하고 진행**(문서 프로젝트 정상 케이스). prep 명령 실패 = 위임 거부(exit 1), worktree는 디버깅용 보존(state `failed-env`).
- **S8. Sandbox = `-s workspace-write` 하드코딩(M1):** codex 기본에서 network 차단 → **ad-hoc dependency 설치가 구조적으로 불가능**(§4.5 "agent가 환경 부재를 자의적으로 해결하지 않는다"와 정렬). env prep은 runner 시작 **전에** 하네스가 (네트워크 가능한) 호스트에서 수행. config knob 없음 — sandbox 축은 다른 mode가 실제 필요한 role과 함께 M2에서(§16).
- **S9. 거주지(§14):** 산출물 전부 plugin-local — `~/.claude/waystone/delegations/<pslug>/<did>/`, worktree는 `~/.claude/waystone/worktrees/<pslug>/<did>/`. `<pslug>` = `common._project_slug(root)` 재사용(abs-path hash — resume/start_here와 동일 규약). 커밋 금지, repo 내 잔여물 없음(git ref `refs/waystone/*`는 로컬 repo 메타지만 push되지 않는 refspec 밖 네임스페이스 — 공유 표면 아님).
- **S10. delegation-id = `<UTC yyyymmddTHHMMSSZ>-<task-slug-sanitized>`** (예: `20260713T214501Z-feat-xyz`). 실행 이벤트 기록이므로 timestamp 허용(0.7의 decisions.jsonl 전례). task-slug sanitize = `/`→`-`. 같은 초에 record dir이 이미 존재하면 결정적 suffix `-2`,`-3`…로 유일화(H4 — 기존 레코드에 덧쓰기 금지).
- **S11. exposure record는 시작 시 1회 기록 후 불변(#4·#12).** 상태 전이는 별도 `status.json`(mutable)이 담당 — immutable record와 lifecycle 상태의 분리.
- **S12. 평가 경로 = worktree 보존 + `show`/`apply`:** worktree는 applied/discarded까지 보존 — verifier(main 포함)가 **같은 base**에서 직접 테스트 실행 가능(#12의 물리적 구현). `waystone delegate apply` = **plain `git apply`**(3-way 아님 — §12의 실검증 근거). base snapshot이 delegate 시점의 dirty tree를 이미 포함하므로 live tree가 그 뒤 안 바뀌었다면 plain apply가 정확히 맞고, drift 시 원자적으로 실패해 정직하게 표면화된다.
- **S13. Profile은 `~/.claude/waystone/profile.yml`,** binding 누락 시 fail-loud 안내(기본 모델 추측 금지). M1 소비 role은 `implementer`뿐 — `--role`에 다른 값이 오면 exit 1 `role <x> not consumed in M1`(조용한 수용 금지). 스키마는 3축(role/execution/backend) 일반형(§8.2) — codex 외 backend **구현**은 하지 않음(스키마만, §3.8).
- **S14. 신규 skill 없음(M1).** conventions에 delegation 절 추가 + round skill에 한 줄 언급만. guided flow는 M2(overlay·improve 연계)에서 재고.

**적대 리뷰 반영 (2026-07-13, 25 findings → 12 confirmed 전건 수용):**
- R1. owner-lock terminal set 자기모순 해소 — terminal = {applied, discarded}, failed-*도 lock 유지, stuck `running`은 discard가 수용(S6·§4).
- R2. apply를 `--3way`에서 **plain `git apply`로 교체** — 3-way는 unstaged WIP 파일에서 index 불일치로 전면 거부됨(리뷰 실증). plain apply의 원자성·dirty 동작은 본 세션 재검증(§12).
- R3. patch 생성에 `--binary` 추가(바이너리 변경 유실 방지).
- R4. diff 계열에 `--no-renames` 명시 고정(사용자 git config 영향 차단), changed_files status {M,A,D,T} 문서화.
- R5. worktree의 `.git`은 pointer 파일 — index 복사 시딩 폐기, read-tree HEAD 시딩으로 통일(S1, 동등성 실검증).
- R6. merge/rebase/cherry-pick 진행 중 snapshot 거부(conflict marker가 base에 구워지는 것 실증됨) — §3 전제조건.
- R7. `runner_sandbox` config knob 삭제, M1 하드코딩(S8).
- R8. accept comma-split 금지 — YAML list + 반복 플래그만(S5).
- R9. apply 실패를 §2 exit 계약에 명시 편입, git raw rc 비누설(apply는 {0,1,2}만).
- R10. blocked task 거부 메시지에서 "deps 미충족" 단정 제거(stale-blocked에 거짓).
- R11. artifact 없는 delegation에 `show --patch/--report` = exit 1 + 사유 명시.
- R12. `--role` 비-implementer 명시 거부(S13).

**하드닝 반영 (2026-07-13/14, M1 구현 후 적대 하드닝: 9 제기 → 7 재현 confirmed → 중복 병합 4 root cause, 전건 수정):**
- **H1. 비-UTF-8 안전성 + `failed-artifact` 상태 신설:** patch는 bytes로 생성·저장(`git diff` stdout 무디코딩, `write_bytes`; empty 판정도 bytes), JW_REPORT.yaml은 bytes로 yaml 파싱(비-UTF-8 = `present: invalid`), delegate의 git 출력 디코딩은 surrogateescape, `show --patch`는 stdout buffer로 raw bytes. post-runner artifact 계산 블록(result snapshot→contract)은 예외 시 **`failed-artifact`**로 전이 — worktree 보존, 기존 failed-*와 동일하게 lock 유지, discard로 해제. `running` 좌초로 owner-lock 영구 점유 금지.
- **H2. `JW_REPORT.yaml` = 예약 파일명:** run 전제조건(§3)에서 live tree에 존재 시 exit 1 거부 — base에 구워져 delegate report로 오독·patch의 D로 phantom deletion되는 경로 차단.
- **H3. 손상 레코드 JSON fail-safe:** 단건 경로(show/apply)는 WorkflowError로 손상 파일 지목(exit 1, traceback 금지); owner-lock 스캔은 손상 status.json/exposure.json을 **lock 보유로 간주**(새 위임 거부 + 파일명·discard 안내); `status` 목록은 해당 행 `[corrupt]` 표시 후 나머지 정상 출력; discard는 손상 레코드 수용(정리 경로 자기봉쇄 금지). status.json 쓰기는 atomic(temp+`os.replace`).
- **H4. 같은 초 delegation-id 충돌:** record dir 존재 시 결정적 suffix `-2`,`-3`… 루프 — 기존 레코드에 상태 전이가 덧붙는 일 금지(S10).

## 1. 모듈 배치

flat `<name>.py` 컨벤션 유지. 신규 1개:

- **`scripts/delegate.py`** — `waystone delegate` CLI 그룹 전체 + snapshot/worktree/env/runner/contract 로직. profile 로딩도 여기(현재 유일 소비자 — common으로 승격은 M2에서 두 번째 소비자가 생길 때).
  - runner 호출은 module-level function `_run_codex(cmd: list[str], ...) -> tuple[int, str]`로 격리 — 테스트에서 monkeypatch(기존 run_tests.py의 fake `_gh` 전례).
  - git 호출은 `common.git_rc` 재사용(신규 plumbing 헬퍼는 delegate 내 private).
- `scripts/validate.py` — task 스키마에 `accept` 선택 필드(**list[str]**) 검증 추가(S5).
- `scripts/tasks.py` — `add`/`set`에서 `accept` 필드는 **거부 + 안내**: "accept is a YAML list of free-text criteria — edit tasks.yaml directly (writes are validated) or pass --accept at delegation time" (R8; comma-split 왜곡 방지).
- `scripts/waystone.py` — dispatcher에 `delegate` 그룹 + docstring Groups 갱신.

## 2. CLI surface

```
waystone delegate run <task-id> [--role implementer] [--accept TEXT]... [--root DIR]
waystone delegate status [<delegation-id>] [--root DIR]        # 인자 없으면 프로젝트의 전체 목록
waystone delegate show <delegation-id> [--patch|--report|--exposure] [--root DIR]
waystone delegate apply <delegation-id> [--root DIR]
waystone delegate discard <delegation-id> [--root DIR]
```

- `--root` 기본값 = cwd에서 `find_project_root`(기존 규약). root 없으면 exit 1.
- exit code 계약: 0 ok / 1 usage·precondition(acceptance 부재, profile/binding 부재, `--role` 비-implementer, owner 충돌, snapshot 전제조건 위반(§3), env prep 실패, runner 비정상 종료, **apply 실패(patch 미적용·drift)**, artifact 없는 delegation에 `show --patch/--report`, 미지 delegation-id) / 2 refused write(plugin-local 디렉터리 생성 불가 등). 메시지 stderr. **모든 subcommand는 {0,1,2}만 반환 — git의 raw rc(128 등)를 절대 누설하지 않는다**(R9).
- `run`의 stdout(1줄씩): base/snapshot sha, worktree 경로, env prep 결과, runner 종료, artifact 경로 — 스크립트 소비 가능하게 `key: value` 형태.
- `status`: delegation-id, task, state(running|needs-review|failed-env|failed-runner|failed-artifact|applied|discarded), base 7-sha, 생성 시각. 손상 레코드 행은 `[corrupt]`로 표시하고 나머지는 정상 출력(H3).
- `show --patch` = changes.patch 원문 stdout(raw bytes — H1); `--report` = contract.yaml; `--exposure` = exposure.json; 무옵션 = 요약. **failed-env/failed-runner/failed-artifact처럼 artifact가 생성되지 않은 delegation에 `--patch`/`--report`는 exit 1**: `delegation <id> is failed-env — no patch/contract was produced` (R11; `--exposure`와 무옵션 요약은 항상 가능 — exposure는 시작 시 기록됨).

## 3. Snapshot 의미론 (D1/S1 — 실검증된 시퀀스)

**전제조건(전부 exit 1 fail-loud):** HEAD 존재(unborn 거부 — 커밋 0개 repo에 위임 불가) / `.gitmodules` 부재(submodule M1 비지원 — 조용한 부분 snapshot 금지) / **live tree에 `JW_REPORT.yaml` 부재(H2)** — 예약 파일명; 존재하면 base에 구워져 delegate report로 오독되고 patch의 D로 phantom deletion된다 / **진행 중 작업 없음(R6)**: `git ls-files -u` 출력이 비어 있고, `--git-path`로 해석한 `MERGE_HEAD`·`CHERRY_PICK_HEAD`·`REVERT_HEAD` 파일과 `rebase-merge`·`rebase-apply` 디렉터리가 없어야 한다 — 아니면 conflict marker·중간 상태가 base에 조용히 구워진다(리뷰 실증).

```
tmp_index = mktemp()
GIT_INDEX_FILE=tmp_index git read-tree HEAD        # 시딩 (S1 — index 복사 아님)
GIT_INDEX_FILE=tmp_index git add -A                # tracked 수정 + staged + untracked(비ignored) 포함
tree = GIT_INDEX_FILE=tmp_index git write-tree
if tree == git rev-parse HEAD^{tree}:              # clean-tree 지름길 (실검증)
    base_sha = HEAD sha; dirty = false             # snapshot commit 생성하지 않음
else:
    base_sha = git commit-tree tree -p HEAD -m "waystone delegation snapshot: <task-id> <did>"; dirty = true
rm tmp_index
git update-ref refs/waystone/delegations/<did> base_sha  # gc 생존 고정 (gc --prune=now 실검증)
```

- 검증 사실(스크래치 실험): tracked 수정·staged·untracked 포함 / .gitignore 제외 / live worktree·index 무변경 / `git worktree add --detach <path> <sha>` 정상 / ref 덕에 `gc --prune=now` 생존 / clean tree 시 write-tree == HEAD^{tree} / **read-tree-HEAD 시딩 == index-복사 시딩 (동일 tree sha)** / **linked worktree 안에서도 동일 시퀀스로 result snapshot 정상 산출**(`.git`이 pointer 파일이므로 복사 방식은 애초에 불가 — `git rev-parse --git-path index`가 `<main>/.git/worktrees/<n>/index`로 해석됨을 확인).
- untracked 포함은 **의미론 선언**이다: "delegate의 base = 사용자가 지금 보는 tracked+untracked(비ignored) 상태". 문서·README에 명기.

## 4. Worktree 공간·생명주기 + owner lock (S6/S9/S12 — lock predicate의 단일 정의처)

```
~/.claude/waystone/worktrees/<pslug>/<did>/     ← git worktree add --detach <path> <base_sha>
```

- 생성: `run` 시. 제거: `apply`/`discard` 시 `git worktree remove --force` + `git update-ref -d refs/waystone/delegations/<did>`(-result ref 포함) + worktree dir 삭제. delegation record dir(§9 산출물)은 **보존**(이력 evidence).
- **Lock predicate(#2, R1):** delegation은 state ∈ **{applied, discarded}일 때만 terminal**. `run`은 같은 root에서 동일 task-id의 **비terminal** delegation(running·needs-review·failed-env·failed-runner·failed-artifact)이 하나라도 있으면 거부: `waystone delegate: task feat/xyz already has active delegation <did> (state <s>) — apply or discard it first`. failed-*가 lock을 유지하는 이유: worktree가 evidence로 보존되는 동안은 같은 task의 두 번째 mutation surface를 만들지 않는다 — 정리(discard)가 곧 lock 해제다. **손상 레코드(status/exposure JSON 파싱 불가)는 lock 보유로 간주**(H3 fail-safe) — 거부 메시지가 손상 파일과 discard 경로를 지목한다.
- **stuck `running`의 해제 경로(R1):** M1의 `run`은 동기 실행이므로, 새 CLI 호출 시점에 관측되는 `running`은 정의상 crash 잔재다. 따라서 `discard`는 running 상태도 그대로 수용한다(별도 플래그 불필요 — 실행 중인 프로세스와 경합할 수 없는 상태이기 때문).
- stale worktree(레코드는 terminal인데 worktree 잔존 등)는 `status`가 표시만(자동 청소 금지 — §3.8 단순성).

    - uv sync --frozen

## 6. Runner — codex exec (실검증된 CLI 표면, codex-cli 0.143.0)

```
codex exec -C <worktree> -m <model> -s workspace-write --color never \
  --output-last-message <record>/last_message.md --json < prompt.txt > <record>/runner.jsonl 2> <record>/runner.stderr
```

- prompt는 stdin으로 전달(인자 길이·quoting 회피). `-C`로 작업 루트 고정(worktree는 git repo이므로 `--skip-git-repo-check` 불요). `-s workspace-write`는 하드코딩(S8/R7).
- model = profile binding `backend: "codex:<model>"`의 `<model>` 부분. `codex:` 접두 필수 — 다른 접두(claude: 등)는 M1에서 exit 1 `backend not implemented in M1`(스키마는 수용, 실행만 미구현 — S13).
- runner rc≠0 → state=failed-runner, exit 1. **단 exposure/log/last_message는 이미 기록됨** — 실패도 evidence(§3.3).
- `--json` JSONL 전체를 runner.jsonl로 보존(local-only, §14). 파싱하지 않는다(M1) — last_message + JW_REPORT가 계약 채널.
- timeout: config 없음(M1) — codex 자체 종료에 맡기고 subprocess timeout 3600s 고정(초과 시 failed-runner). 근거: §3.8 — 조정 가능성은 실사용 마찰이 확인되면.

**Prompt 렌더링(packet → prompt.txt, 템플릿 `templates/delegate-prompt.md` 신설):**

```

Task packet: id/title/notes/anchor/deps(상태 포함)/acceptance criteria(전체 나열)
Base: <base_sha> (worktree가 이 상태다; 네트워크 없음; 의존성 설치 금지 — 준비된 환경 사용)
Scope: 이 worktree 안에서만 수정. 범위 밖 문제는 JW_REPORT의 escalations에 기록.
보고: 종료 전 worktree 루트에 JW_REPORT.yaml 작성 —
  verification: [{cmd, rc, summary}], limitations: [str], risks: [str], escalations: [str]
```

## 7. Task packet (S5)

`packet.yaml` — `run` 시 tasks.yaml에서 조립(레지스트리 raw 복사가 아니라 필요 필드만):

```yaml
schema: waystone-packet-1
task: {id, title, status, milestone, round, deps: [{id, status}], anchor, notes}
acceptance:            # task.accept(YAML list) + --accept 플래그(기준당 1회) 병합 — 순서 유지, 중복 제거
  - "..."
project: {name, root}  # 레지스트리 name(있으면), root abs path
```

- 전제조건: task 존재 && status ∈ {pending, active}. done/dropped 거부. **blocked 거부 메시지(R10):** `task feat/xyz is blocked — if its deps are now satisfied, set it active and retry` (stale-blocked가 존재하므로 "deps 미충족" 단정 금지 — `next_actionable`의 의미론 참조).
- acceptance 공집합 → exit 1: ``waystone delegate: task feat/xyz has no acceptance criteria — add `accept:` (YAML list) to the task or pass --accept`` (#3, 지어내기 금지).

## 8. Artifact contract (`artifact/contract.yaml`, S2/S3/S4)

```yaml
schema: waystone-artifact-1
delegation_id, task_id
base_sha, result_sha            # 하네스 계산 (explicit)
changed_files:                  # git diff --name-status --no-renames base result (explicit; R4)
  - {path, status: M|A|D|T}
patch_file: changes.patch       # git diff --binary --no-renames base result (explicit; R3/R4; 빈 diff면 파일 없이 empty: true)
delegate_report:                # JW_REPORT.yaml 원문 이식 (delegate-claimed — 사실 승격 금지)
  present: true|false|invalid
  verification: [{cmd, rc, summary}]   # provenance: delegate-claimed
  limitations: [...]
  risks: [...]
  escalations: [...]
env: {prep: explicit|detected:<kind>|none-detected, commands: [...], rc: 0}
runner: {backend: "codex:<model>", rc, duration_s, last_message: last_message.md}
```

- result snapshot: runner 종료 후 **worktree cwd에서** §3과 동일한 read-tree-HEAD 시퀀스(-p base_sha), `refs/waystone/delegations/<did>-result` (linked worktree 동작 실검증 — §3). JW_REPORT.yaml은 **읽고 삭제 후** 계산(S4).
- rename은 탐지하지 않는다(R4) — `--no-renames`로 사용자 git config(`diff.renames`) 영향까지 차단, status는 {M,A,D,T}로 고정.
- diff 공집합(무변경) → `empty: true` + state는 needs-review 그대로(무변경도 보고 대상 — "할 일 없었음"은 main이 판단).
- **자기수락 금지의 구현:** contract에는 pass/fail 판정 필드가 **없다**. 판정은 main/verifier가 `show`·worktree에서 수행(#8).

## 12. 평가 경로 (S12 — #8/#12, R2)

1. `run` 종료 → state=needs-review. main이 `waystone delegate show <did>`로 contract/patch/report 검토.
2. 독립 검증(선택): worktree가 base 그대로 + delegate 변경 상태로 보존되어 있으므로, main/verifier가 **worktree cwd에서** acceptance 명령을 직접 실행 — 검증이 같은 base를 참조(#12)함이 물리적으로 보장된다.
3. 수락: `waystone delegate apply <did>` — live root에서 **plain `git apply artifact/changes.patch`** (3-way 아님).
   - **근거(R2):** `--3way`는 unstaged WIP가 있는 파일에서 "does not match index"로 전면 거부됨(리뷰 실증) — dirty tree를 전제로 하는 D1과 충돌. base snapshot이 delegate 시점의 dirty tree를 이미 포함하므로, live tree가 그 뒤 안 바뀌었다면 plain apply가 정확히 들어맞는다.
   - **검증 사실(스크래치, 2026-07-13):** ① base 이후 무변경 live tree에 정상 적용(rc 0) ② 패치 대상 파일 중 하나에 충돌 drift가 있으면 rc 1로 실패하며 **다른 패치 대상 파일도 일절 쓰지 않음(원자성 — hunk 전수 검사 후 기록)** ③ 패치와 무관한 파일의 dirty는 적용에 영향 없음(plain apply는 index를 참조하지 않음).
   - 실패 시: exit 1 + git이 지목한 실패 파일 목록 stderr 전달 + 안내 `live tree has drifted from the delegation base — commit/stash your changes and retry, or resolve the patch manually`. **부분 적용 없음, 자동 3-way fallback 없음(M1 — 단순성).** state는 needs-review 유지.
   - 성공 시 state=applied + worktree/ref 정리.
4. 거부: `waystone delegate discard <did>` — state=discarded + worktree/ref 정리(record dir 보존). running(crash 잔재 — §4)·failed-* 모두 수용.
- empty patch의 apply = no-op 성공(state 전이만).

### 16.1 codex 플러그인 route 검토 (2026-07-14 조사 — M2 입력)

설치본 `codex@openai-codex` v1.0.6 (openai/codex-plugin-cc) 조사 결과와 결정 방향:

- **기전**: 플러그인은 `codex exec`를 쓰지 않음 — workspace별 **warm broker daemon**(unix socket) 뒤의 `codex app-server`(JSON-RPC)를 구동. role-형 entrypoint 3종: `review`(native, read-only), `adversarial-review`(steerable challenge + `schemas/review-output.schema.json` 구조화 출력), `task/rescue`(범용, `--write`=workspace-write). 부가: `transfer`(Claude 세션→Codex thread), background job(status/result/cancel), stop-review-gate(기본 off — stop마다 Codex 리뷰 1회, BLOCK 시 Claude 정지 차단, 고비용).
- **스크립트 호출 가능**: `node <plugin-root>/scripts/codex-companion.mjs task|review|adversarial-review --json --cwd <dir> [--model|-m] [--effort]` — 플러그인 자신의 Stop hook이 쓰는 경로 그대로, Claude 세션 밖에서도 동작(broker 자동 관리, auth는 codex CLI 공유). 제약: workspace당 1 active turn(직렬, busy 시 -32001), worktree 격리 없음(--cwd로 지정한 실 checkout에서 동작), approvalPolicy 항상 never.
- **M1 runner 결정 유지**: delegation(mutation, implementer)은 **raw `codex exec` 유지** — 위임마다 새 worktree라 warm 이득이 없고(broker가 workspace별), 버전 경로(`cache/openai-codex/codex/<ver>/`)에 대한 의존이 생기며, 격리·결정성은 exec 쪽이 단순.
- **M2 반영 방향**: (1) role binding 스키마에 execution transport로 `codex-companion`을 허용 — 특히 **verifier/reviewer role**: `adversarial-review --json --cwd <보존된 worktree>` = 다른 모델 계열의 독립 시각으로 same-base 검증(#12) + 구조화 출력. transport 부재 시 fail-loud(silent fallback 금지), plugin root는 installed_plugins.json에서 동적 해석. (2) model-레벨 routing은 지금도 가능: main이 review/adversarial-verify를 `/codex:review`·`/codex:adversarial-review`·codex-rescue agent로 배정 — M2 binding config가 이를 감사 가능하게 명명. (3) stop-review-gate는 0.9 live-guard 영역과 겹침 — waystone가 기본 활성화하지 않고 기존 기제로 문서화만.
