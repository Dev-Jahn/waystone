## §1 사용자 문제 제기 (외부 리뷰어에게 보낸 질문, verbatim)

waystone은 refactor task M1-B 까지 진행한 상태인데, 하나 개선하고 싶은게 있어.
기존 waystone이 나름 체계적으로 multi-agent orchestration 시 발생할 수 있는 문제들은 잘 억제를 해왔는데, 이 harness를 채택해서 개발을 진행하다 보면 꼭 test가 비대해지고, 개발시 검증조건을 엄격하게 하는데 집착하게 되면서 원래의 목표는 잃어버리고 개발이 진척이 되지않는 문제가 거의 모든 프로젝트에서 생기더라고.
waystone을 구버전 jahns-workflow 때부터 채택해서 쓰던 newton-omniphysics repo에서 대대적인 clean up을 한번 해보기로 했음. 아예 기존에 작업한 내용은 싹 다 날리고, 기존 source들은 참고자료로 archiving하고 brainstorming 단계부터 다시시작했어.

내가 생각했던 agent driven development가 drift 되는 가장 큰 이유는 크게는 아래 3가지야.
1. 사용자부터가 자기가 뭘 개발하고 싶은지를 정확히 모르는 상태에서 agent에게 추상적인 지시만 주고 자율작업을 시킴. 사용자의 머리속에 뭔가 추상적인 그림은 있는데, 그걸 본인도 확정하지 못한상태에서 잘못된 SSOT를 박아놓고 개발을 시작하니 점점 개발이 진행되면서 윤곽이 드러나니 본인이 원했던 것과 뭔가 다르다는 사실을 깨닫게 됨. 사용자의 target project에 대한 domain knowledge가 충분할 수록 이런 문제가 적은데, omniphysics의 경우 이 문제가 극단으로 치달은 대표 예시였음. 내가 그래서 waystone에 ideate skill을 임시방편으로나마 추가해뒀던 건데, 좀 부족했던 걸로 보임. 내가 첨부한 `프로젝트 분석 및 진행 상황.md`를 보면, 다른 대화 세션에서 omniphysics 프로젝트의 realign을 위해 진행한 소크라테스식 대화의 모든 turn이 기록되어있음. 대화를 통한 구체화 자체도 유익했지만, '프로젝트가 왜 산으로 가게 되었나'에 대한 분석과 해결책 제시가 인상적이었음.

2. gpt pro의 리뷰가 너무 강력한 것이 오히려 부작용을 유발. 특유의 강력한 reasoning으로 개발 세션에서는 잡아내지 못하는 온갖 major/minor issue들을 잡아내는데 이게 개발 진행에 실질적으로 문제를 일으킬 확률이 적은 문제이더라도 버그의 실제 존재 여부나 발생했을 때의 심각도는 사실이기때문에 ingest 후 개발 목표 drive 보다 버그 수정에 치중하게됨.
"""
`[Pasted text #1 +244 lines]` 이건 bootstrap 이후 상태에서의 평가인데, 기존 omniphysics를 싹 밀어버렸던 이유가 자꾸 리뷰의 지적사항을 과도하게 고치고 방지하다가 산으로 간거였음. 굳이 별도 문서화나 문제 방지를 위한 harness 오버엔지니어링 같은거 하지말고, 실재하는 문제만 깔끔하게 고쳐둬.
"""
위와 같이 리뷰를 전달하니 버그는 버그대로 고쳐지면서, 오히려 waystone으로 review ingest 할때보다 개발 진행이 쾌적해진 느낌이었음. 리뷰 자체의 문제라기보다는 처리하는 방식의 문제일수도.
리뷰로 인한 버그 폭발 자체 외에도, 하나 또 우려되는건 main agent가 미리 잡아내지 못한 문제들을 리뷰가 계속 정확하게 찾아내고 그게 사실로 밝혀지는 일이 반복되니, review를 수신하는 agent가 점점 기가 죽는다고 해야하나? 리뷰에 대한 의존도도 높아지고, 리뷰에서 지적한 문제는 완전히 틀린 지적을 제외하고는 거의 수용하고 우선 수정을 시도하는 경향성이 보임.

3. orchestrator-worker agent 간 context 단절. main 세션은 프로젝트의 거시적 흐름과 목적, 현재 상황 들을 상대적으로 정확히 파악하고 있지만, delegation이 일어나는 순간 main-worker 사이의 prompt 지점에서 극심한 정보 병목이 생기고 그동안 파악한 모든 context가 사실상 최종 결과물의 품질에 도움을 주지 못함.

다소 설명이 길어졌는데, 결국 원하는 건 하나야.
waystone이 이 모든 내용을 반영해서 위 세가지 문제를 어떤 방식으로든 해결했으면 좋겠다는거.
0.12.0 개선안에 이 문제들에 대한 충분한 고려가 녹아들지 않은 것 같아.

## §2 외부 리뷰 응답 1 — 상위 control plane 설계 (verbatim)

# 판단

동의합니다. **현재 0.12.0 설계는 “주어진 목표를 안전하게 실행하는 방법”은 매우 강하게 다루지만, “그 목표가 아직 가설에 불과한가”, “발견된 결함을 지금 고치는 것이 제품 진척에 유익한가”, “worker가 올바른 판단을 내릴 만큼 의미 맥락을 받았는가”는 거의 다루지 않습니다.**

M1-B는 transactional store, effect reconciliation, lease/fencing, independent verification, observability 등 실행 신뢰성 계층을 성공적으로 구축했습니다. 반면 그 과정에서 계약 테스트가 838개에서 1,088개로 증가했고, worker prompt는 I-10 준수를 위해 더 얇아져 현재는 사실상 `title + scope + acceptance + report schema`만 전달합니다. 테스트 증가는 M1-B의 성격상 그 자체로 잘못은 아니지만, **검증 기계가 프로젝트의 산출물보다 빠르게 성장하는 현상을 제어하는 반대 방향의 메커니즘은 아직 없습니다.**

첨부한 omniphysics 대화가 도출한 핵심도 동일합니다. 문제는 검증을 약화하는 것이 아니라, **탐색 단계부터 승격 단계의 검증 규율을 적용하면서 구현·학습·제품 진척이 거버넌스 루프로 대체된 것**입니다.

현재 세 문제는 다음의 하나의 feedback loop로 연결됩니다.

```text
아직 불확실한 사용자 의도
→ 너무 이른 hard SSOT/acceptance 고정
→ 강한 verifier/reviewer가 모든 결함을 발견
→ REAL finding이 곧바로 task와 blocker가 됨
→ 테스트·gate·방어 기계가 계속 증가
→ worker에는 국소 acceptance만 전달
→ worker가 목적보다 criteria를 최적화
→ 실행 결과가 사용자의 실제 그림에서 멀어짐
→ 더 많은 review와 hardening
```

따라서 새 기능 세 개를 따로 추가하기보다, Waystone에 빠진 **상위 control plane**을 하나 추가해야 합니다.

> **Waystone은 execution state뿐 아니라 intent의 불확실성, 작업의 lifecycle stage, review finding의 처분, worker에게 전달할 의미 맥락을 함께 다뤄야 합니다.**

# 현재 0.12 설계에서 정확히 빠진 것

## 1. 프로젝트 의도가 이미 올바르다고 가정함

현재 `RunSpec`의 frozen job input은 다음만 갖습니다.

```text
task_id
title
acceptance_criteria
scope
dependencies
input_digest
```

task freeze도 `title`, `accept`, `scope`, `deps`만 읽습니다.

즉 엔진은 다음 질문에는 답할 수 있습니다.

> 이 task가 실행 도중 변했는가?
> worker가 scope를 벗어났는가?
> acceptance가 비어 있는가?

그러나 다음 질문에는 답하지 않습니다.

> 이 acceptance가 아직 검증되지 않은 제품 가설에서 파생됐는가?
> 이 task를 완료하면 사용자가 원하는 결과에 실제로 가까워지는가?
> 현재 SSOT가 너무 이른 확정을 포함하고 있는가?

ADR-0010은 coordinator가 작성한 acceptance의 달성 가능성·scope·evidence adapter를 검증하는 좋은 계약입니다. 하지만 그것은 **criterion 내부의 readiness**를 검사하는 것이지, 프로젝트 방향 자체의 타당성을 검사하지는 않습니다.

## 2. Review finding의 진위와 실행 우선순위가 결합되어 있음

현재 review skill은 finding을 실제 코드에 대조해 `REAL / REJECTED / NEEDS-RULING`으로 검증합니다. 여기까지는 맞습니다. 문제는 그 다음 단계입니다.

> **각 REAL finding을 task registry에 등록한다.**
> blockers는 다음 round 전에 반드시 해결해야 한다.

이 구조에서는 다음 두 문장이 사실상 동일해집니다.

```text
이 결함은 실제로 존재한다.
→ 그러므로 지금 수정해야 한다.
```

하지만 실제로는 서로 다른 판단입니다.

```text
truth: 이 결함은 존재한다.
impact: 발생하면 심각하다.
exposure: 실제 사용 경로에서는 거의 발생하지 않는다.
relevance: 현재 prototype 목표와 무관하다.
cost: 일반화된 방어까지 하면 architecture가 크게 팽창한다.
disposition: 지금 고치지 않고 accepted risk로 둘 수 있다.
```

현재 ADR-0010은 모든 run에 adversarial review를 강제하지 않고 trust surface에만 review를 요구하도록 제한합니다. 이는 좋은 기반입니다.  그러나 **review를 받기로 한 이후 finding을 어떻게 비례적으로 처리할지**는 정의하지 않았습니다.

## 3. “prompt 최소화”가 “의미 맥락 최소화”로 구현됨

I-10의 본래 문구는 타당합니다.

> 모델에는 목표, 경계, 판단 요청만 전달하고 bookkeeping protocol은 전달하지 않는다.

하지만 현재 구현은 이를 지나치게 문자적으로 적용했습니다. worker는 다음을 받지 않습니다.

* 왜 이 task가 지금 필요한가
* 어떤 상위 outcome을 전진시키는가
* 기존 구현이 왜 이런 형태인가
* 이미 실패했던 접근은 무엇인가
* 어떤 설계 선택은 고정이고 어떤 선택은 worker 재량인가
* acceptance의 배경이 되는 사용자 의도
* task가 architecture 전체에서 차지하는 위치
* 현재 프로젝트가 탐색 중인지, 정식 승격 중인지

현재 prompt는 protocol noise는 적지만, **decision-relevant context도 함께 제거된 상태**입니다. 이는 단순 구현 task에는 적합하지만, architecture·research·제품 judgment가 필요한 delegation에서는 정보 병목을 구조화합니다.

# 권고하는 통합 설계

## 핵심 구조

```text
Project Frame
  commitments / hypotheses / open questions
            ↓
Lifecycle Stage
  explore / evaluate / promote
            ↓
Semantic Work Brief
  why / current state / decisions / freedom / evidence
            ↓
Worker execution + harness evidence
            ↓
Review Finding Record
  truth ≠ priority ≠ disposition
            ↓
Coordinator adjudication
            ↓
선택된 finding만 task 또는 promotion blocker가 됨
            ↓
Outcome / capability delta
```

이것은 새로운 workflow OS를 하나 더 얹자는 제안이 아닙니다. 기존 state machine 위에 다음 세 가지 구분을 추가하는 것입니다.

1. **확정된 의도와 아직 열린 가설**
2. **탐색과 공식 평가와 mainline 승격**
3. **결함의 존재와 수정 우선순위**

# 1. 잘못된 SSOT 조기 고정을 막는 방법

## `SSOT`를 “완성된 진실”이 아니라 `Project Frame`으로 취급

SSOT나 project brief에는 최소한 다음 세 종류의 진술이 구분되어야 합니다.

### Commitments

사용자가 현재 명시적으로 선택했으며, agent가 임의로 바꾸면 안 되는 방향입니다.

```text
primary user
desired outcome
prototype boundary
hard constraints
explicit non-goals
human approval boundaries
```

### Working hypotheses

프로젝트가 현재 시험 중이지만 아직 사실이나 요구사항으로 승격되지 않은 판단입니다.

```text
이 solver family가 적합할 것이다.
이 UX가 주요 사용 경로일 것이다.
이 feature set이 첫 prototype에 충분할 것이다.
이 architecture가 장기 확장에 유리할 것이다.
```

### Open questions

답이 없다는 사실 자체가 현재 상태인 항목입니다.

```text
첫 convincing prototype은 무엇인가?
어떤 정확도 수준이 실제로 필요한가?
어떤 subsystem이 mainline-critical인가?
어느 결과가 나오면 현재 방향을 폐기해야 하는가?
```

핵심 규칙은 다음입니다.

> **Hypothesis와 open question은 hard acceptance criterion으로 자동 변환하지 않는다.**

`commitment`만 owner-authored constraint가 됩니다. `hypothesis`는 exploration의 방향을 제공할 수 있지만, 반증되었을 때 task 실패가 아니라 project learning으로 처리합니다.

## `ideate`를 bootstrap skill에서 framing/realignment skill로 확장

현재 ideate는 이미 산파술적 질문을 하도록 설계되어 있습니다. 하지만 기본적으로 새 프로젝트 시작 시 3–6회 정도의 대화로 north-star SSOT를 작성하는 bootstrapper입니다.

omniphysics 사례가 보여준 것은 훨씬 더 깊은 **project realignment**입니다. 따라서 같은 public surface를 유지하되 동작을 두 가지로 나누는 편이 좋습니다.

```text
uninitialized project  → framing
existing project       → realignment
```

별도의 `/realign` 명령까지 만들 필요는 없습니다. `/waystone:ideate`가 기존 프로젝트를 감지하면 다음을 먼저 분석합니다.

* 최근 작업 중 feature/output과 review/fix/docs의 비중
* 마지막으로 실제 실행 가능한 outcome이 달라진 commit
* active milestone과 project commitment의 관계
* 반복적으로 reopen되거나 PARK되는 연구 축
* test·gate 증가와 user-visible capability 변화
* 현재 roadmap의 임계경로가 사용자 목표에서 실제로 파생되는지

그 후 질문은 현재처럼 한 turn에 하나씩 하되, **고정된 3–6회 목표는 제거**합니다. 종료 조건은 다음입니다.

> 남은 질문이 architecture detail일 뿐, 답에 따라 제품 방향·prototype·non-goal이 달라지지는 않는다.

출력은 바로 “완성된 SSOT”를 만드는 것이 아니라 다음 상태를 명시해야 합니다.

```text
frame_status: provisional | committed
unresolved_directional_questions: [...]
revision_triggers: [...]
```

사용자가 `committed`를 승인한 이후에만 project-wide execution constraint로 사용합니다.

## 자동 realignment trigger는 hard gate가 아니라 advisory

다음 현상이 반복되면 status가 realignment를 제안할 수 있습니다.

* 연속된 여러 run이 review remediation뿐이고 declared outcome 변화가 없음
* task completion은 증가하지만 executable capability가 변하지 않음
* 같은 project assumption 때문에 반복적으로 task가 reopen됨
* 현 milestone의 성공이 open research question 하나에 무기한 종속됨
* user feedback가 기존 project commitment와 반복적으로 충돌함

숫자나 비율을 universal hard rule로 고정하면 다시 Goodhart가 발생합니다. 따라서 이는:

```text
project-direction-may-be-stale
```

라는 advisory reason만 제공하고, 사용자와의 realignment 대화를 자동 시작할지 묻는 정도가 적절합니다.

# 2. Review가 프로젝트를 지배하지 않게 하는 방법

## Reviewer를 “high-recall defect sensor”로 제한

GPT Pro reviewer는 가능한 많은 실제 failure mechanism을 찾는 데 최적화할 수 있습니다. 그러나 다음 판단의 권위자는 아닙니다.

* 지금 무엇을 고칠 것인가
* 어느 정도 일반화해서 막을 것인가
* 새 abstraction이나 harness를 도입할 것인가
* 현재 milestone을 중단할 것인가
* 제품 목표보다 이 결함을 우선할 것인가

따라서 review output은 **finding truth evidence**이고, remediation plan은 아닙니다.

## Finding model을 분리

현재의 `REAL + severity + task` 대신 최소한 다음을 분리해야 합니다.

| 축                     | 예시                                                                 |
| --------------------- | ------------------------------------------------------------------ |
| **Validity**          | confirmed / rejected / unresolved                                  |
| **Impact**            | blocker / major / minor — 발생했을 때의 결과                               |
| **Exposure**          | common / edge / adversarial / unknown                              |
| **Relevance**         | current-objective / promotion-bound / future / out-of-scope        |
| **Disposition**       | fix-now / fix-before-promotion / backlog / accept-risk / no-action |
| **Remediation scope** | local / bounded / architectural                                    |

가장 중요한 규칙은 다음입니다.

> **`confirmed major`는 자동으로 `fix-now`를 의미하지 않는다.**

예를 들면 다음 판정이 정식으로 허용되어야 합니다.

```yaml
validity: confirmed
impact: major
exposure: edge
relevance: future
disposition: accept-risk
reason: >
  현재 supported envelope 밖이며, 구조적 방어는 이번 prototype의
  objective와 무관한 새 state machine을 요구한다.
```

이것은 finding을 부정하는 것이 아닙니다. **사실과 제품 결정을 분리하는 것**입니다.

## Review skill의 핵심 변경

현재의 다음 규칙은 제거해야 합니다.

> 각 REAL finding을 task로 등록한다.

대신:

1. 모든 finding은 immutable finding record로 보존
2. coordinator가 objective, lifecycle stage, exposure, cost를 고려해 disposition 결정
3. `fix-now`와 `fix-before-promotion`만 task 또는 current run blocker로 materialize
4. `backlog`, `accept-risk`, `no-action`은 finding record로만 남음
5. disposition이 바뀔 수 있지만 원 finding은 덮어쓰지 않음

task registry는 **발견된 모든 문제의 집합**이 아니라, 실제로 실행하기로 결정한 work의 집합이어야 합니다.

## Lifecycle stage별 assurance allocation

첨부 대화에서 도출한 `Exploration → Evaluation → Promotion`을 Waystone의 보편적인 assurance model로 채택하는 것이 적절합니다.

### Explore

목적은 답을 찾거나 candidate를 만드는 것입니다.

* 구현과 반복 측정 허용
* acceptance는 “올바른 최종 결과”보다 “무엇을 배울 것인가” 중심
* external macro-review 기본 비활성
* worker self-check는 claim으로 기록
* 임시 probe·temporary test 허용
* 실패한 candidate는 가볍게 폐기
* permanent ADR 및 regression suite 요구 없음
* 완료 결과는 `candidate`, `measurement`, `refuted hypothesis` 중 하나

### Evaluate

동결된 candidate가 약속한 성질을 만족하는지 판단합니다.

* candidate SHA/config 동결
* evaluation criteria와 test set 동결
* independent verifier 필수
* 이미 본 evaluation 결과로 같은 candidate를 수정하지 않음
* 수정하면 새 candidate revision
* review는 architecture/domain risk가 있을 때만

### Promote

mainline의 durable behavior로 승격합니다.

* 최소한의 영구 regression contract
* relevant trust surface의 adversarial review
* migration·compatibility·public behavior 검토
* release 또는 integration result에 결속
* 지원 범위와 accepted risks 기록

현재 ADR-0010의 risk-gated review와 exact digest binding은 `promote` 단계에서 그대로 강점이 됩니다. 버릴 필요가 없습니다. 문제는 그 수준의 assurance가 exploration까지 전염되지 않도록 하는 것입니다.

## Test lifecycle도 함께 분리

테스트를 하나의 영구 집합으로만 취급하면 exploration이 곧 test accretion이 됩니다.

### Probe

* exploration 중 가설을 확인하는 일회성 또는 임시 check
* artifact 또는 candidate branch에만 존재 가능
* mainline permanent suite에 들어갈 의무 없음

### Evaluation check

* 동결된 candidate를 판정
* evaluation generation에 결속
* 다음 candidate에서는 교체될 수 있음
* 반드시 mainline permanent test일 필요 없음

### Regression contract

* promoted behavior만 보호
* 사용자-visible behavior, trust invariant, 반복 가능한 실제 failure mechanism에 한정
* finding 하나당 여러 case를 늘리기보다 property/invariant test를 우선
* 계약이 폐기되거나 범위가 축소되면 테스트도 삭제·통합 가능

영구 test를 추가하려면 다음 중 하나가 있어야 합니다.

```text
promoted public behavior
stable trust invariant
plausibly recurring production failure
```

단순히 “reviewer가 실제 edge case를 찾았다”는 이유만으로 permanent regression matrix를 추가하지 않습니다.

## Reviewer에 대한 main agent의 과도한 복종 방지

이를 prompt로 “기가 죽지 마라”라고 할 수는 없습니다. 권한 구조를 바꿔야 합니다.

```text
Reviewer:
  failure mechanism과 evidence를 제시

Adjudicator/coordinator:
  진위만 독립 검증

Objective-aware decision:
  지금 고칠지, 언제 고칠지, 어느 규모로 고칠지 결정
```

reviewer가 제안한 remediation architecture는 기본적으로 비권위입니다. 첫 수정안은 다음 순서를 따라야 합니다.

1. 실제 failure mechanism의 최소 root cause 확인
2. current objective에 영향을 주는 최소 범위 확정
3. local/bounded fix 우선
4. architecture 일반화가 필요하면 별도 owner decision
5. 동일 부류가 반복된 evidence가 없으면 새 harness subsystem을 만들지 않음

또한 한 candidate에 대해 review→fix cycle 수를 유한하게 제한해야 합니다. 예를 들어 2회 이후에도 finding이 계속 확장되면 자동으로 추가 hardening을 하지 않고 다음 중 하나를 owner에게 제시합니다.

```text
accept current risk
reduce supported scope
simplify architecture
start a separately approved research track
```

이 중 “새 대형 research track”을 coordinator가 자동 선택해서는 안 됩니다.

# 3. Orchestrator–worker context 병목 해소

## I-10을 다음처럼 개정해야 함

현재:

> 모델에는 목표, 경계, 판단 요청만 전달하고 bookkeeping protocol은 전달하지 않는다.

권장:

> **모델에는 판단에 필요한 최소 충분한 semantic context를 전달하고, 상태 전이·artifact naming·retry·bookkeeping protocol은 전달하지 않는다. Context 최소화는 protocol 최소화를 뜻하며, 목적·근거·과거 실패·결정 공간의 제거를 뜻하지 않는다.**

즉:

```text
minimize protocol
≠
minimize meaning
```

## `Semantic Work Brief` 추가

`tasks.yaml`을 다시 비대하게 만들 필요는 없습니다. dispatch 시 coordinator가 RunSpec에 결속하는 content-addressed artifact 하나면 충분합니다.

```yaml
schema: waystone-work-brief-1

objective:
  ref: "<project commitment / milestone / owner request>"
  why_now: "..."

current_state:
  summary: "..."
  last_known_result: "..."
  known_failures:
    - "..."

decisions:
  fixed:
    - "..."
  worker_may_choose:
    - "..."
  requires_escalation:
    - "..."

constraints:
  - "..."
non_goals:
  - "..."

evidence_expected:
  - "..."

references:
  - path: "..."
    anchor: "..."
    digest: "..."

open_questions:
  - "..."
```

worker에게 보여줄 prompt는 여전히 자연스럽고 짧을 수 있습니다.

```markdown
## Why this matters
...

## Current context
...

## Goal
...

## Freedom and constraints
...

## Acceptance
...

## Relevant sources
...

## Report
...
```

현재 prompt의 bookkeeping surface는 늘리지 않으면서, main이 가진 거시적 의미를 worker에게 전달할 수 있습니다.

## Context provenance를 구분

모든 context가 동등한 권위는 아닙니다.

```text
owner-source          — 사용자 또는 Git-tracked commitment
harness-observation   — Git/state에서 재도출한 사실
coordinator-summary   — main의 해석과 압축
```

worker prompt에서도 이 차이를 잃지 않아야 합니다.

예:

> **Coordinator context — interpretation, not owner authority**
> 기존 구조가 X를 택한 이유는 Y로 보인다. 실제 결정 근거는 아래 source refs를 우선한다.

이렇게 하면 main의 tacit context를 전달하면서도, coordinator의 요약이 새 SSOT가 되는 것을 막을 수 있습니다.

## `context_request`를 실행 중 1급 action으로 허용

모든 내용을 처음부터 prompt에 넣으면 다시 context bloat가 됩니다. 대신 worker가 decision-critical 정보가 부족할 때 실행을 멈출 수 있어야 합니다.

```yaml
action: context_request
question: "A와 B 중 어느 compatibility contract가 authoritative한가?"
blocked_decision: "public API 유지 여부"
why_required: "둘에 따라 변경 파일과 migration 방식이 달라진다"
```

engine은 이를 일반 실패로 처리하지 않고:

```text
waiting_context
```

로 전이합니다. coordinator가 brief revision을 발행하면 새 digest와 새 attempt로 재개합니다.

이는 worker가 불충분한 맥락에서 넓은 변경을 한 뒤 reviewer가 잡는 것보다 훨씬 저렴합니다.

## Delegation 판단에 context transfer cost 포함

다음과 같은 작업은 무조건 worker에 보내는 것이 오히려 손해일 수 있습니다.

* main session이 긴 대화를 통해 획득한 tacit intent가 핵심
* 수정 자체는 작지만 trade-off가 제품 방향에 직접 연결됨
* relevant context를 brief로 압축하기 어려움
* worker 실행보다 handoff 작성 비용이 더 큼

이 경우에는:

```text
in-session
```

이 더 올바른 routing일 수 있습니다.

반대로 scope가 크고 context가 source references로 전달 가능한 경우에는 worker delegation이 적합합니다. 따라서 routing compiler는 task 크기뿐 아니라 **context transfer loss**를 판단 축으로 가져야 합니다.

모호할 때만 사용자에게 묻습니다.

# 현재 M1-B 이후 적용 순서

M1-B를 되돌리거나 clean slate로 다시 만들 필요는 없습니다. **M1-B의 trust kernel은 그대로 유지해야 합니다.** 문제는 그 위의 control plane입니다.

## M1-C 전에 반드시 넣을 것

### 1. I-10 개정과 Semantic Work Brief

M1-C가 기존 delegate front door를 새 엔진으로 전환하기 전에 필요합니다.

그렇지 않으면 현재의 context-starved prompt가 Waystone의 canonical delegation model로 굳어집니다.

최소 구현:

* optional semantic brief artifact
* RunSpec digest 결속
* worker prompt의 `why/current context/freedom` section
* context provenance
* early `context_request`

## Review skill은 다음 macro-review 전에 바로 수정

full review feature migration을 기다릴 필요가 없습니다.

* `REAL → task` 자동 연결 제거
* finding disposition 추가
* current objective와 lifecycle stage를 반영
* `fix-now`와 `fix-before-promotion`만 task 생성
* architecture 확대는 owner decision 필요

이는 storage refactor가 아니라 현재 workflow incentive 수정입니다.

## M2에 편입할 것

M2의 multi-job scheduler를 현 계획대로 먼저 구현하면 잘못된 incentives를 더 빠르게 자동화하게 됩니다. 현재 M2는 bounded closure, integration worktree, evidence rebinding을 매우 상세히 다루지만, run이 **무슨 종류의 work인지**와 outcome progress는 정의하지 않습니다.

M2에는 다음이 선행되어야 합니다.

* lifecycle stage
* stage별 assurance policy
* review finding disposition
* candidate/review cycle budget
* objective reference
* outcome delta
* context-request resume

## M3에 넣을 것

* ideate realignment
* provisional/committed project frame
* status의 objective-progress view
* review-remediation spiral advisory
* SessionStart capsule에는 현재 objective, lifecycle stage, last outcome만 표시

현재 M3는 prompt와 사용자 surface를 얇게 만드는 데 집중합니다.  이때 semantic context와 progress semantics까지 함께 정리하는 것이 맞습니다.

# 구현 task를 최소화한다면

별도 task를 수십 개 만들지 말고 세 묶음이면 충분합니다.

## A. Intent & Context

```text
Project frame의 commitments/hypotheses/open questions
RunSpec semantic brief
I-10 amendment
context_request/resume
```

## B. Assurance & Review

```text
explore/evaluate/promote
finding truth/priority/disposition 분리
review cycle budget
temporary/evaluation/regression test lifecycle
```

## C. Progress & Realignment

```text
outcome delta
status projection
review-only drift advisory
ideate realignment
```

각 묶음에 수십 개의 matrix test를 추가해서는 목적을 다시 위반합니다.

필요한 핵심 end-to-end test는 대략 다음 정도입니다.

1. architecture task의 worker가 목적·현재 상태·결정 공간을 받지만 state-machine protocol은 받지 않는다.
2. worker가 context 부족을 보고하면 변경 없이 `waiting_context`가 되고 brief revision 후 재개한다.
3. `confirmed major` finding이 `accept-risk` 또는 `backlog`로 처분될 수 있고 exploration을 자동 차단하지 않는다.
4. 같은 finding이 promotion contract를 위반하면 `fix-before-promotion`으로 실제 차단한다.
5. explore run은 candidate와 임시 probe를 결과로 완료할 수 있고, permanent test·ADR·macro review를 강제하지 않는다.
6. promote run은 최소 regression contract와 필요한 reviewer evidence 없이는 완료되지 않는다.
7. status가 review/fix task 수 증가를 progress로 보고하지 않고 마지막 objective delta를 보여준다.

그 이상은 각 모듈의 내부 unit test가 아니라 실제 결함이 나타났을 때 추가하면 됩니다.

# 최종 권고

현재 0.12.0 계획을 폐기할 필요는 없습니다. 다만 정의를 다음처럼 확장해야 합니다.

현재 정의:

> 세션과 worker를 넘어 지속되는 evidence-gated development run engine

권장 정의:

> **세션과 worker를 넘어 프로젝트 의도와 실행을 이어가되, 불확실한 가설은 탐색 가능하게 유지하고, 검증·리뷰 비용은 승격 위험에 비례시켜, worker에게 목적에 충분한 맥락을 전달하는 development run engine**

가장 중요한 세 문장은 다음입니다.

> **SSOT에는 확정된 commitment만 넣고, hypothesis를 requirement로 위장하지 않는다.**

> **Review finding의 존재와 심각도는 remediation priority가 아니다.**

> **Harness prompt를 줄인다는 것은 workflow protocol을 줄인다는 뜻이지, worker가 올바르게 판단하는 데 필요한 의미 맥락을 제거한다는 뜻이 아니다.**

Waystone은 지금까지 agent가 거짓으로 “완료했다”고 말하는 문제를 상당히 잘 막았습니다. 다음 단계에서는 한 가지를 더 막아야 합니다.

> **모든 불확실성과 모든 reviewer 관측을 영구적인 개발 의무로 바꾸면서 프로젝트의 실제 목적을 잃는 것.**

SSOT.md 라는 파일명이 다소 agent에게 오해를 줄 수 있을것 같은데 이름을 바꿀까?

## §4 외부 리뷰 응답 2 — PROJECT_BRIEF.md 개명 (verbatim)

# 결론

**바꾸는 것이 맞습니다.** 추천 이름은 **`PROJECT_BRIEF.md`**입니다.

`SSOT.md`라는 이름은 다음과 같은 잘못된 추론을 유도하기 쉽습니다.

* 문서의 모든 문장이 확정된 요구사항이다.
* 문서에 없는 선택은 허용되지 않는다.
* 현실과 문서가 충돌하면 구현보다 현실을 문서에 맞춰야 한다.
* 열린 질문이나 임시 가설도 반드시 충족해야 할 acceptance criterion이다.
* 문서의 내부 정합성을 높이는 것이 프로젝트 진척보다 우선한다.

특히 agent는 `Single Source of Truth`라는 표현을 강하게 해석하는 경향이 있습니다. 프로젝트의 방향, 가설, 연구적 불확실성을 담는 문서에 이 이름을 붙이면 **provisional intent가 immutable specification으로 승격**되기 쉽습니다. Omniphysics에서 나타난 현상과 정확히 맞닿아 있습니다.

현재 `ideate` skill의 실제 내용도 이미 SSOT라기보다 project brief에 가깝습니다. 문서는 “north-star altitude”에 머물러야 하고 spec이 아니며, architecture와 세부 API를 배제하고, open question을 자산으로 남기라고 규정합니다. 즉 **내용의 의미와 파일명이 서로 충돌하고 있습니다.**

# 권장 명칭

## Public filename

```text
PROJECT_BRIEF.md
```

## Public concept

```text
Project Brief
```

## 내부 도메인 객체가 필요하다면

```text
ProjectFrame
```

내부적으로는 commitments, hypotheses, open questions를 구조화한 `ProjectFrame`이라고 불러도 되지만, 사용자에게 노출되는 용어는 표준적인 **Project Brief**가 낫습니다.

`PROJECT_FRAME.md`는 Waystone 고유 jargon처럼 보일 수 있고, `PROJECT_CHARTER.md`는 지나치게 확정적·거버넌스 중심이며, `NORTH_STAR.md` 역시 하나의 고정된 방향이 이미 확립됐다는 인상을 줍니다.

| 후보                            | 판단                                                   |
| ----------------------------- | ---------------------------------------------------- |
| `PROJECT_BRIEF.md`            | **권장.** 표준적이고, 방향성 문서이되 완전한 spec은 아니라는 의미가 명확        |
| `PROJECT.md`                  | 단순하지만 역할이 모호함                                        |
| `INTENT.md`                   | 의미는 좋지만 업계 표준성이 낮음                                   |
| `PROJECT_CHARTER.md`          | 너무 공식적이고 고정적으로 들림                                    |
| `NORTH_STAR.md`               | 방향은 표현하지만 불확실성과 가설을 담기에는 지나치게 단일 지향적                 |
| `VISION.md`                   | constraints, non-goals, prototype scope까지 담는 문서로는 좁음 |
| `SPEC.md` / `REQUIREMENTS.md` | 현재 문제를 더 악화시킴                                        |

# 이름만 바꾸면 충분하지 않음

가장 중요한 것은 문서 안의 진술을 **권위 수준별로 분리하는 것**입니다.

권장 구조는 다음과 같습니다.

```markdown
# <Project>

## Purpose
왜 존재하는 프로젝트인지.

## Commitments
사용자가 현재 확정한 방향. Agent가 임의로 변경할 수 없음.

## Prototype scope
현재 프로토타입에서 실제로 달성할 범위.

## Long-term direction
프로토타입 이후 확장 방향. 현재 구현 의무는 아님.

## Non-goals
현재 명시적으로 하지 않을 것.

## Working hypotheses
시험 중인 판단. 요구사항이 아니며 반증될 수 있음.

## Open questions
아직 결정하지 않은 사항. Agent가 임의로 채우지 않음.

## Revision triggers
어떤 증거·상황에서 brief를 다시 검토해야 하는지.
```

권위는 다음처럼 달라야 합니다.

| 구획                    | 의미             | 자동 acceptance 변환 |
| --------------------- | -------------- | ---------------- |
| `Commitments`         | owner가 확정한 방향  | 가능               |
| `Prototype scope`     | 현재 지원·완료 범위    | 가능               |
| `Non-goals`           | 현재 범위 제한       | 가능               |
| `Long-term direction` | 미래 확장 방향       | 금지               |
| `Working hypotheses`  | 반증 가능한 가설      | 금지               |
| `Open questions`      | 명시적인 미결정       | 금지               |
| `Revision triggers`   | realignment 조건 | 직접 변환 금지         |

핵심 계약은 다음이어야 합니다.

> **`PROJECT_BRIEF.md`는 owner commitments와 명시적 non-goals에 대해서만 authoritative하다. Working hypotheses와 open questions는 요구사항이나 acceptance criteria가 아니다.**

# 문서 상태도 명시하는 것이 좋음

```yaml
---
schema: waystone-project-brief-1
status: provisional
---
```

지원 상태는 두세 개면 충분합니다.

```text
provisional
committed
superseded
```

### `provisional`

* ideation 또는 realignment 진행 중
* 방향성 있는 exploration은 가능
* project-wide promotion·장기 architecture lock-in은 제한
* hypothesis를 hard requirement로 사용 금지

### `committed`

* 사용자가 현재 방향을 확인
* commitments와 non-goals가 실행 권위가 됨
* 그래도 working hypotheses는 계속 비권위

### `superseded`

* 새 brief가 존재
* 역사적 provenance로만 판독

`ideate`가 대화를 끝냈다고 바로 `committed`로 만들기보다, 기본 산출물은 `provisional`로 두고 사용자의 명시적 채택으로 전환하는 편이 안전합니다.

# “하나의 SSOT” 대신 “fact별 단일 authority”

현재 Waystone의 refactor 방향과도 이쪽이 더 일관적입니다.

이미 runtime 설계는 task intent, run state, Git tree, review evidence, published closeout 등에 각각 별도의 authority를 배정합니다. 즉 새 구조의 원칙은 사실상 **하나의 전역 SSOT가 아니라, fact마다 authority가 정확히 하나인 구조**입니다.

예를 들면:

| Fact                        | Authority                     |
| --------------------------- | ----------------------------- |
| 프로젝트 commitment·non-goal    | `PROJECT_BRIEF.md`            |
| working hypothesis          | `PROJECT_BRIEF.md`, 단 비구속 상태  |
| 확정된 architecture decision   | accepted ADR                  |
| evaluation metric·threshold | versioned evaluation spec     |
| task intent·dependency      | `tasks.yaml`                  |
| 현재 run state                | SQLite runtime store          |
| cross-machine closeout      | Git-tracked closeout manifest |
| reviewer가 실제로 검토한 결과        | digest-bound review artifact  |

따라서 `SSOT`라는 개념을 완전히 폐기할 필요는 없습니다. 다음처럼 **좁은 fact domain의 속성**으로만 사용하면 됩니다.

> `tasks.yaml` is the single authority for task intent.
> The runtime store is the single authority for active run state.

반면 프로젝트 전체를 설명하는 한 문서를 “모든 것의 진실”로 부르지는 않는 편이 맞습니다.

# Waystone에서의 구체적인 변경안

## 1. 신규 프로젝트

```yaml
project_brief: PROJECT_BRIEF.md
```

또는 더 간단히:

```yaml
brief: PROJECT_BRIEF.md
```

개인적으로는 public config도 `brief`가 가장 낫습니다.

현재 파일 경로는 `.waystone.yml`에서 설정되므로, filename 변경 자체는 구조적으로 어렵지 않습니다.

## 2. 기존 프로젝트

자동 rename하지 않는 것이 좋습니다.

```text
brief: PROJECT_BRIEF.md     # 신규 canonical key
ssot: SSOT.md               # legacy alias
```

규칙:

* 둘 중 하나만 허용
* 둘 다 있으면 typed refusal
* 기존 `ssot:` 프로젝트는 그대로 동작
* 신규 init과 ideate는 `brief:`만 생성
* 1.0까지 legacy alias 판독
* 파일 이동이나 내용 변환은 사용자 명시 요청 없이는 수행하지 않음

## 3. Skill 문구

현재 표현:

```text
the whole workflow is SSOT-anchored
this is where the anchor is forged
write a ready-to-adopt SSOT.md
```

권장 표현:

```text
clarify the project's current commitments, hypotheses, and open questions
write a provisional PROJECT_BRIEF.md
commit only the direction the user has actually chosen
preserve uncertainty instead of converting it into requirements
```

## 4. 내부 tooling

외부 surface를 먼저 바꾸고 내부 파일명은 단계적으로 정리해도 됩니다.

```text
ssot.py        → 당장 유지 가능
waystone ssot  → compatibility alias
waystone brief → 신규 canonical command
```

이번 변경의 목적은 package rename 정리가 아니라 agent semantics 수정이므로, 내부 모듈까지 한 번에 바꾸다가 범위를 확대할 필요는 없습니다.

# Omniphysics clean slate에는 바로 적용하는 것이 좋음

앞서 만든 문서 세트도 다음처럼 바꾸는 편이 낫습니다.

```text
SSOT.md
→ PROJECT_BRIEF.md
```

그리고 `CLAUDE.md`와 `AGENTS.md`에서는 다음과 같이 표현해야 합니다.

기존:

> SSOT를 위반하거나 변경하지 않는다.

권장:

> `PROJECT_BRIEF.md`의 commitments와 non-goals를 임의로 변경하지 않는다.
> Working hypotheses는 시험할 수 있으며, 반증은 실패가 아니라 project learning이다.
> Open questions는 worker가 추측해 닫지 않는다.

이 차이가 중요합니다. 첫 문장은 agent를 문서 준수 최적화로 끌고 가지만, 두 번째 문장은 **확정된 방향은 보존하면서 탐색은 허용**합니다.

# 최종 권고 문구

Waystone의 문서와 코드에 다음 문장을 기준 계약으로 넣는 것을 권합니다.

> **`PROJECT_BRIEF.md` is not a complete specification or a global source of truth. It is the owner-authored authority for current project commitments and non-goals. Working hypotheses remain falsifiable, and open questions remain unresolved until an owner decision promotes them.**

한국어로는:

> **`PROJECT_BRIEF.md`는 완전한 명세나 프로젝트 전체의 단일 진실 원천이 아니다. 현재 사용자가 확정한 commitment와 non-goal의 권위 문서다. Working hypothesis는 반증 가능하게 유지하며, open question은 사용자 결정 전까지 미결정 상태로 남긴다.**

따라서 제 권고는 명확합니다.

> **파일명과 public concept를 `SSOT.md`에서 `PROJECT_BRIEF.md`로 바꾸고, “전역 SSOT”라는 개념은 fact별 단일 authority 모델로 대체하는 것이 좋습니다.**

## §5 최종 사용자 지시 (verbatim)

이번 세션은 위 내용들을 참고해서 plugin을 자유롭게 뜯어고쳐봐. 기존 계획도 따를 필요없고, 기존 어떤 test도 통과할 필요도 없어. migration도 고려할 필요없어. 다 구현해 놓고, 나중에 임시 migration script 하나 추가하면돼.
waystone을 내 의도를 완전하게 반영하는 `세션과 worker를 넘어 프로젝트 의도와 실행을 이어가되, 불확실한 가설은 탐색 가능하게 유지하고, 검증·리뷰 비용은 승격 위험에 비례시켜, worker에게 목적에 충분한 맥락을 전달하는 development run engine`으로 만들어줘.
