## 0. 요약

0.12.0은 기능 추가 없이 waystone을 **하나의 durable run engine + 얇은 UX**로 재구성한다.
proposal의 골격(run engine, 두 종류의 state, bounded closure, integration worktree, 역할·용어
축소, 얇은 skill, prompt capsule)을 채택하되 §2의 수정을 가한다. 전략 한 문장은 proposal §18:

> 기능을 더하는 대신, 한 번의 bounded run이 계획·병렬 실행·독립 검증·통합·handoff를 끝까지
> 소유하도록 만들고, 사용자와 모델에는 그 run의 목적과 현재 필요한 결정만 보여준다.

**성공의 정의 (r3 채택 + r4 보강).** 새 엔진의 성공은 상태 파일 수가 줄었다는 것으로
증명하지 않는다. 다섯 가지로 증명한다:

1. 같은 effect가 중복 실행되지 않는다 (§3-4).
2. 어떤 evidence도 자신이 검토하지 않은 result를 승인하지 않는다 (E-07).
3. 사용자가 선택한 delivery policy 밖의 변경이 live tree에 도달하지 않는다 (§6 M2-4).
4. fact당 authority가 하나다 (§5-2).
5. **진행 중인 run에 대해 사용자는 liveness·progress·current의 확정 가능한 값, unknown인
   항목의 구체적 사유, 마지막 확정 시점과 안전한 next action을 로그 파일·프로세스·worktree를
   뒤지지 않고 확인할 수 있다** (§3-8 — r5 정밀화: "언제나 생사를 안다"가 아니라 **"확정 가능한
   것과 불가능한 것을 구분해 안다"**. carrier에 관측 채널이 없으면 정직한 답은 `unknown`이다).

### 왜 지금 (실측 근거 — r2에서 진단 이원화, r3 유지)

- **모듈·테스트가 한계 규모다.** `scripts/` 19,791줄(improve 4,667 / delegate 3,819 /
  overlay 2,676 / review 2,514), 테스트 단일 파일 20,479줄에 828건, CLI 앞문 18 group.
  기능 하나에 skill 텍스트·CLI 검사·git 상태·artifact 관계 네 층을 재방어한다.
- **0.11의 major 결함은 두 부류였다.** ① **부분 상태/중복 발행 부류**(dedup 부재의 sidecar
  재발행, cache-body 이원 권위, partial write) — 단일 transaction store가 **구조적으로
  제거**하는 부류. ② **authority·ordering 의미 부류**(v1/v2 ordering, marker vs sidecar 권위
  선택, 손상-후-fallback, runtime config identity 정의 부족 — round-5의 011~013이 전부 이
  부류) — store로 사라지지 않으며 **명시적 authority·ordering·digest·corruption policy 설계**가
  계속 필요한 부류. 따라서 M1의 성공 기준은 "sidecar가 사라졌다"가 아니라 **"fact당 authority가
  하나가 되었다"**다 (§5-2 matrix).
- **자율 round의 스펙이 이미 검증돼 있다.** 5라운드의 수동 fleet 운영(병렬 lane → 실측 →
  적대 검증 → verdict → 순차 병합 → 게이트 → close)이 곧 run engine의 스펙이며 실패 모드까지
  실측돼 있다 (§6 M2).
- **진행 중 실행이 관측 불가하다 (r4 — 2026-07-19 실사고).** round-6 dogfooding 중 네트워크
  전환으로 러너 stderr에 오류가 쌓이자, **조정자(main session)조차 두 delegation이 살아 있는지
  죽었는지 판정하지 못했다.** 근거로 삼을 수 있었던 것은 "요약 로그가 비어 있다"뿐이었는데 그건
  *아직 아무것도 쓰지 않았다*는 뜻이었다 — 실제로는 하나는 정상 완료, 하나는 실행 중이었다.
  오판으로 두 레코드를 discard하려 했고 record lock만이 그것을 막았다. 관측 수단이
  `pgrep`·로그 tail·worktree `git status`인 시스템은 **불변 지향점 ③(사용자는 내부를 알 필요가
  없다)을 이미 위반**하고 있다. 사람 사용자에게는 이 수단조차 없다.

## 2. Proposal·초판과 의도적으로 다른 결정

### 2-1. 리뷰 증거는 git-tracked 파일로 남긴다 — 단, "변경 금지"가 아니라 "결함 폐쇄 후 의미 동결"

머신 경계를 넘는 증거(리뷰 요청/회신/binding sidecar)는 git-tracked 유지, 0.11의 digest 기계
보존, DB는 cache/index만(권위는 재파생). 초판의 "features/external_review 변경 금지"는
과도했다 — 미해결 trust finding(011~013)을 **먼저 폐쇄**하고(round-5: 코드 착지 완료, 재리뷰
회신 대기) 그 시점의 observable contract를 동결한 뒤, 0.12에서는 의미 변경 없이 이관한다.
그래야 characterization이 알려진 결함을 "보존할 동작"으로 고정하지 않는다.

### 2-2. Data migration은 거의 0 — 단, schema migration 기계는 처음부터 둔다

0.11이 완료 판정 권위를 git-tracked 원본의 읽기 시점 재파생으로 옮겨 놓았으므로 새 runtime
DB는 빈 상태에서 review/pending 상태를 재구성할 수 있다 — **legacy runtime record의 bulk
migration은 하지 않는다** (read-only archive + `inspect` 판독). 그러나 다음은 migration이며
처음부터 둔다: DB schema version + transactional migration sequence, upgrade 전 backup(§5-5),
실패 시 원본 보존, idempotent 재실행, 구버전 plugin의 신버전 DB open 거부, dry-run, rollback은
forward-fix version으로. `consents.jsonl` import·probe marker 이관·profile v1 adapter는
idempotent one-time importer로.

## 3. 목표 아키텍처 (구현 기준)

### 3-1. 두 종류의 state + runtime store 모델 (full event sourcing 철회)

```text
A. Git-tracked intent & cross-machine evidence   ← 사람이 diff, git 동기화
   .waystone.yml, tasks.yaml, SSOT/brief, ADR
   PROGRESS.md  ← 현행 의미 유지(git-tracked work log) — DB projection 아님
   ROADMAP.md   ← tasks.yaml의 순수 projection (유일한 생성 뷰 권위 관계)
   docs/reviews/* (0.11 기계 그대로), published policy, run closeout manifest(§5-4)

B. Local transactional runtime  →  .waystone/state.db (SQLite, WAL) + artifacts/
   [authority]  runs/jobs/attempts/actions 의 current-state row
                — version column + compare-and-swap 갱신
   [audit]      transitions — append-only (prev_state, next_state, entity_version,
                reason enum, evidence digest); 재작성 금지
   [immutable]  artifacts/sha256-* — content-addressed bytes
   [cache]      git/review fact의 digest + 관측 시점 (권위는 항상 원본 재파생)
```

- **canonical event journal + 전면 projection rebuild는 하지 않는다** (event schema 영구
  관리·deterministic replay·upcasting·replay 중 외부 효과 억제라는 신규 복잡성이 "Converge &
  Simplify"에 역행). 한 transaction = `entity version 확인 → transition INSERT →
  current-state UPDATE → artifact reference 연결 → COMMIT`.
- `doctor --rebuild`는 **derived cache/projection 재생성으로 한정** — authoritative
  current-state의 replay 복원 기능이 아니다. transitions는 진단·감사용이며 완전 replay를
  보장하지 않는다(정직한 한계).
- stdlib `sqlite3`, WAL + busy_timeout. schema version 관리 §2-2, 물리 운영 계약 §5-5.

### 3-2. Package layout (초판 유지 + 이동 순서)

```text
waystone/
  cli/  core/  project/  runs/  jobs/  adapters/  features/
```

의존 방향: `features → runs API → core`, `adapters → core protocol`, core는 상방 import 금지.
M1-A에서 improve/overlay/review를 전부 옮기지 않는다 — run kernel이 필요로 하는
경계(common/tasks/git/delegate 상당)만 먼저 추출하고, features 모듈 이동은 M2 이후 점진
(M4 전 완료). hook은 public read API만 사용.

### 3-3. 실행 표면 (사용자 표면과 transport protocol 분리 + submit 검증 계약)

```text
# 사용자/skill 표면
waystone run start <selector> [--mode conservative|balanced|autonomous]
waystone run resume [<run-id>]
waystone run status [<run-id>] [--json]   # 1회 스냅샷 (§3-8)
waystone run watch  [<run-id>]            # r4: 갱신되는 진행 뷰 (사람용, --json 없음)
waystone run deliver <run-id>          # r3: delivery_policy=manual일 때의 명시적 승인 행위
waystone run cancel <run-id> --reason <enum>

# carrier/내부 transport (사용자 문서에 노출하지 않음)
waystone run actions next <run-id> --json
waystone run actions submit <action-id> --file <result>
```

action은 최소 `action_id, action_kind, executor_kind, input_digest, entity_version,
lease_epoch, result_schema, ownership/expiry`를 가진다. `submit` 검증: 현재 action인가 /
input digest 일치 / fencing epoch 현재 / action별 result schema 충족 / artifact digest가 실제
bytes와 일치 / **Git-파생 사실(changed files·result SHA·test 결과)은 carrier 보고를 신뢰하지
않고 엔진이 재도출** (I-03의 API 계약화). 상태 전이는 항상 엔진이 결정하고, CLI 실패는 typed
(exit enum + `{"ok":false,"code":...,"recoverable":...,"next_actions":[...]}`).

**run id 생성 (r3 확정):** canonical id = `<UTC compact timestamp>-<slug>-<6자 random>`.
timestamp 선행으로 정렬 가능성을 유지하고 random suffix로 머신 간 생성 충돌을 막는다(날짜+slug
만으로는 두 머신이 같은 id를 만들 수 있고, git closeout manifest는 사후 검출만 가능하다).
slug는 사람이 읽는 label이며 identity의 일부가 아니다.

**개정 (2026-07-20):** 위 timestamp-slug-random 문법은 ADR-0005의 RFC 9562 UUIDv7 canonical
lowercase hyphenated `run_id` 문법으로 대체되었다. 위 원문은 결정 이력으로 보존하며 신규 `run_id`
생성에는 적용하지 않는다.

### 3-4. External effect commit protocol (M0-B ADR로 확정)

DB transaction은 Git ref/worktree/process/artifact write/push/marker 게시를 원자화하지
못한다. 따라서 store schema보다 먼저 다음 계약을 ADR로 확정한다
(`docs/adr/ADR-xxxx-runtime-effect-commit-protocol.md`):

```text
planned   → immutable input digest + idempotency key를 DB에 기록
claimed   → owner token + fencing epoch 획득
effect    → 모든 외부 산출물(git ref, worktree 경로, artifact 메타)에 run/job/action id 각인
observed  → 엔진이 Git/filesystem/process 결과를 재도출
completed → observed digest와 함께 DB transaction commit
```

**정확성의 근거 (r3 — 핵심 규칙):**

> **Lease expiry는 liveness를 위한 것이고, correctness는 fencing epoch + expected-state
> CAS가 보장한다.** 벽시계가 흔들리거나 heartbeat가 지연되어도 lease를 잃은 stale worker는
> submit·apply·push를 성립시키지 못한다.

**Effect 종류별 계약 (r3 신설 — 추상 lifecycle 하나로는 구현 단계에서 임의 예외가 생긴다):**

| Effect | Idempotency 기준 | 관측(observation) | **관측 채널 불가 시** | Retry |
|---|---|---|---|---|
| Git ref 생성·갱신 | expected old OID CAS | 실제 ref OID | 로컬이라 사실상 항상 가능; 불가 시 `unknown-effect` | 동일 action 재조정 |
| Worktree 생성 | action별 고정 경로·ref | `git worktree list` + HEAD | 동일 | 검증 후 재사용/정리, 동일 action |
| Artifact write | digest 경로 + atomic rename | bytes 재해시 | 읽기 불가 = `unknown-effect` (없음으로 간주 금지) | 동일 action |
| Runner 실행 | **action당 at-most-once** | completion marker + process identity | **`unknown-effect`로 대기 — 재실행 금지** | 새 attempt + 새 action id로만 |
| Patch integration | expected parent/tree | integration commit/tree | 동일 | 상태에 따라 reconcile |
| Push | expected remote OID | live remote ref 조회 | **네트워크 불가 = `unknown-effect` 대기** (미푸시 가정 금지) | CAS 재시도 |
| GitHub marker | remote dedupe key | remote event 재조회 | 동일 | 동일 key로 재조정 |

**r3가 리뷰 표에 추가한 것 — "관측 채널 불가" 열.** 관측 자체가 실패할 수 있고, 그때
"관측되지 않음"을 "일어나지 않음"으로 접으면 0.11이 겪은 실패 부류(probe absent vs unreadable
혼동)를 새 엔진에서 재생산한다. 관측 불가는 **honest-unknown 상태로 차단**하며 기본값으로
붕괴하지 않는다.

recovery는 "running이면 재실행"이 아니라 **action record + git ref/worktree + artifact
digest + process/result marker의 reconcile**로 "효과 전 사망"과 "효과 후 commit 전 사망"을
구분한다 (decision table을 ADR에 포함). 이 프로토콜 없이는 M1·M2의 "kill→resume 중복 없음"
exit gate를 증명할 수 없다.

**"중복 실행 0"의 정의 (M2 exit 기준 — r3 확정):**

> 같은 `action_id`의 effect는 두 번 실행되지 않는다. 재시도는 **새 attempt + 새 action id**로만
> 가능하다. 불확실한 runner effect는 재실행하지 않고 `unknown-effect`로
> fail-toward-verification한다.

### 3-5. Lock·lease·fencing 책임 분리

- SQLite transaction/unique constraint — run·job·action **claim**
- lease + fencing token — **오래 실행되는 action ownership** (heartbeat/expiry)
- OS advisory file lock — live tree mutation·final delivery·release 등 **단일 머신 짧은
  critical section** (process 사망 시 자동 해제가 SQLite lease보다 단순)
- record-level version(CAS) — stale submit/apply 거부

"파일 lock 0"은 목표가 아니다. `.waystone/lock`은 개념적으로 위 4분류로 대체된다.

### 3-6. Executor 경계 — engine / carrier / user (r3 신설)

모든 mechanical action을 host가 받아 수행하면 skill 텍스트가 얇아져도 **orchestration
복잡성이 transport protocol로 이동할 뿐**이고, 모델은 여전히 workflow orchestrator다. 이는
프로젝트 불변 지향점 ②(harness-derived prompt 최소화)에 정면으로 어긋난다. 따라서 action은
`executor_kind`를 가진다.

| executor_kind | 소유 범위 |
|---|---|
| **engine** | DB transition · git snapshot/ref/worktree · external runner 실행 · artifact hashing · deterministic check · integration · remote verification |
| **carrier** | host-native clean/forked subagent · Claude Workflow처럼 waystone이 직접 프로세스를 만들 수 없는 실행 · 제한적 coordinator judgment |
| **user** | acceptance criterion 판정 불가 · 정책 충돌 · delivery 승인 · 명시적 waiver |

`run resume` / `actions next`는 **engine-owned action을 엔진 내부에서 소진하고, 실제로 host나
user가 수행해야 하는 action만 반환한다.**

**r3 보강 — 비차단 계약.** engine-owned action에는 수 분이 걸리는 것(runner 실행)이 있으므로
`actions next`가 그 완료까지 동기 블로킹하면 안 된다. 반환 계약은 3분기다:

```text
{"action": {...}}                        # carrier/user가 수행할 action
{"action": null, "engine": "busy",       # 엔진이 진행 중 — 폴링 지점과 함께
 "poll_after_s": N, "run_state": "..."}
{"action": null, "engine": "idle",       # 더 진행 불가 — 이유는 typed
 "run_state": "waiting_user|completed|blocked", "reason": "<enum>"}
```

### 3-7. 요청·응답 시 host가 절대 결정하지 않는 것

상태 전이 · retry 여부 · effect 재실행 · evidence 승격. 전부 엔진 결정이며 skill 문서에
프로토콜을 적지 않는다(§6 M3의 skill 금칙어 lint가 강제).

### 3-8. 실행 관측 계약 (r4 신설 · r5 정밀화 — M0-B ADR + M1-B 구현)

진행 중인 run에 대해 사용자가 묻는 질문은 셋이고, **서로 다른 증거원을 가진다. 하나로 뭉치면
반드시 거짓말이 된다.**

| 질문 | 증거원 | 없을 때의 정직한 답 |
|---|---|---|
| **살아 있는가** (liveness) | 해당 action의 관측 계약이 주는 **긍정적** 신호 (supervisor의 process identity 대조 + heartbeat 신선도) | 사유를 동반한 `unknown` |
| **어디까지 갔는가** (progress) | **frozen closure**를 분모로 한 job/task 상태 집계 | 마지막 확정 지점까지만 |
| **지금 무엇을 하는가** (current) | 현재 claimed action의 `action_kind` + claimed 시각 | `unknown-current` |

**E-08의 운영 규칙 — 침묵도 종료의 증거가 아니고, 종료 관측은 종료의 증거다 (r5 정밀화).**
출력 부재·로그 미기록·stderr 오류 누적·heartbeat 부재는 전부 종료 신호가 아니다. 그러나 이를
문자적으로 과잉 적용해 **종료를 긍정적으로 관측했는데도 `exited`를 말하지 못하는 반대 오류**에
빠져서는 안 된다. 구분은 다음과 같다:

```text
침묵 / heartbeat 부재                            ≠ 종료 증거
supervisor wait status / action에 결속된 atomic
completion evidence / 동일 process identity의
종료를 긍정적으로 관측                            = 종료 증거
```

**`stalled`는 FSM 상태가 아니라 derived health다 (r5 — 중요).** heartbeat 신선도는 시간이
흐르기만 해도 바뀌는 **읽기 시점 속성**이다. 이를 authoritative run state로 저장하면 ⑴ 조회
행위가 상태를 바꿔야 하고 ⑵ 시간 기반 상태와 transactional 상태가 섞이고 ⑶ heartbeat 복귀 시
역전 전이가 필요하고 ⑷ audit log가 health polling으로 오염된다. 따라서 분리한다:

```json
{"run_state": "running",            // authoritative, DB current-state row
 "health": "unknown",               // derived, read-time projection (비권위)
 "health_reason": "heartbeat-stale-process-observation-unavailable"}
```

`failed`·`waiting_user`·`canceled`·`completed` 같은 authoritative 전이는 `run resume`이
§3-4의 reconcile을 마친 뒤에만 일어난다.

**liveness는 run 하나로 뭉치지 않는다 (r5).** multi-job run에서 대상은 여럿이다(엔진 supervisor,
job별 runner, carrier, 대기 job). liveness는 **action/job 단위로 계산하고 run은 집계**한다.
한 lane만 unknown인데 run 전체를 `stalled`로 표시하면 거짓이다:

```text
run: running (degraded, attention_required)
jobs: 2 completed · 1 running · 1 unknown · 1 waiting
```

`stalled`는 **다음 progress를 만들 수 있는 action이 하나도 없고 그 원인이 미해소
unknown-effect일 때만** 쓴다. 독립 lane이 계속 진행 가능하면 run은 `running`이다.

**진행률의 분모는 frozen closure다 (r5 정정).** action 수는 retry·verification·reconciliation에
따라 동적으로 늘어나므로 "완료 action / 전체 action"은 감소하거나 100%의 의미가 달라진다. 주
progress는 안정된 분모를 쓴다 — `accepted|terminal jobs / frozen closure tasks`. 표시는 백분율
대신 상태 count가 정직하다. action은 분모가 아니라 **현재 phase 설명**에 쓴다.

```text
Tasks: 3/5 completed          Jobs: 1 running · 1 waiting
Wave: 2/3                     Current: verifying task-x · 4m 12s
Last confirmed transition: 38s ago
```

**진행률은 엔진 사실에서만 파생한다.** 러너의 free-text 출력·stderr를 파싱해 진행률이나 생사를
추정하지 않는다(complexity budget "free-text stderr 분류 0"). worker의 보고는 언제나 claim이고,
**supervisor의 process 관측만 harness fact**다.

**`status`·`watch`는 완전한 read-only projection이다 (r5 신설).** 조회는 lease를 갱신하지 않고,
action을 claim하지 않고, reconcile하지 않고, state transition을 쓰지 않고, cache migration이나
artifact repair를 실행하지 않는다. 읽기 실패는 `status-unavailable` 또는 해당 field의 `unknown`
으로 표시한다. **단순 조회가 run을 "살아 있는 것처럼" 보이게 해서는 안 된다.**

heartbeat는 append-only `transitions`에 매 tick 기록하지 않는다(긴 run이 audit log와 WAL을
tick으로 채운다):

```text
leases/action_runtime  = 최신 heartbeat를 덮어쓰는 mutable telemetry
transitions            = planned·claimed·process-started·effect-observed·
                         completed·cancel-requested 등 의미 있는 변화만 append
```

**표면 설계 원칙 (불변 지향점 ③).** 사용자는 worktree 경로·pid·로그 파일·delegation 레코드
디렉터리를 알 필요가 없다. 내부 식별자(worktree 경로, action id, lease epoch, process identity)는
기본 출력에서 감추고 `--json`/`--verbose`에서만 노출한다. 진행 뷰는 **모델을 거치지 않는다** —
스크립트가 DB에서 직접 렌더링하므로 harness-derived prompt를 늘리지 않는다(불변 지향점 ②).

### 3-9. 취소·정지·정리의 안전 계약 (r5 신설 — 필수)

**2026-07-19 실사고의 진짜 위험은 오판 자체가 아니라 그 오판으로 살아 있는 실행을 discard하려
한 것**이었다. 그때 막아준 것은 설계가 아니라 우연이다 — 0.11의 `discard_delegation()`은
`running`을 포함한 모든 nonterminal 상태를 정리할 수 있고, 안전성은 장시간 붙잡고 있는
`record.lock`에 사실상 의존했다. r4에서 그 lock은 DB claim·lease·fencing·짧은 OS lock으로
분해된다. **우연히 제공되던 "실행 중 cleanup 차단"을 명시적 계약으로 옮기지 않으면 안전성이
리팩터와 함께 사라진다.**

> **`running`·`alive`·`unknown-effect`·`cancel-pending` 상태인 action의 worktree·ref·artifact는
> 삭제하지 않는다.** `run cancel`은 **취소 의도만 기록**하며, supervisor가 해당 process/effect의
> 정지를 **긍정적으로 관측**하고 reconcile을 완료한 뒤에만 terminal cancellation과 cleanup이
> 가능하다. **lease 만료나 heartbeat 부재만으로 destructive cleanup을 허용하지 않는다.**

```text
running → cancel-requested → stopping → canceled
running → cancel-requested → cancel-pending(reason=unknown-effect)
```

- signal은 **verified process identity가 있을 때만** 보낸다. 불명확하면 `cancel-pending(unknown-effect)`로 남긴다.
- stale worker의 뒤늦은 submit은 fencing epoch + CAS가 거부한다(§3-4).
- cleanup은 `observed-quiescent` 이후 **별도 idempotent action**으로 수행한다.
- compatibility `delegate discard`는 실행 중·불명 상태를 직접 정리하지 않고 새 cancel/reconcile 경로를 쓴다.
- crash 잔여물 정리는 `run resume`(또는 expert recovery 명령)이 reconcile한 뒤에만.

이 계약은 E-08의 안전 귀결이며 별도 E 번호를 만들지 않는다. 다만 **traceability matrix에는
독립 행을 둔다**(§2-5).

### 3-10. Process supervision·identity (r5 신설 — M0-B ADR `docs/adr-run-observability-contract`)

PID만 저장하면 PID 재사용으로 오판한다. 최소 process identity:

```text
host boot identity · pid · process start token/time · action_id ·
supervisor owner/fencing epoch · resolved executable 또는 invocation digest
```

**핵심: worker(모델·러너)가 authoritative heartbeat나 completion marker를 직접 쓰지 않는다.**

```text
engine-owned supervisor
  ├─ heartbeat 갱신          ├─ child process identity 보존
  ├─ exit status 회수        ├─ stdout/stderr artifact digest 확정
  └─ atomic completion marker 작성
```

completion marker 최소 필드: `action_id · fencing_epoch · process_identity · started_at ·
finished_at · returncode|signal · stdout_artifact_digest · stderr_artifact_digest`.

heartbeat 신선도 판정은 wall clock이 아니라 **monotonic clock + host boot identity**로 한다
(재부팅·시계 보정 후 옛 heartbeat가 새것으로 보이는 것을 차단). 사용자 표시 시각만 wall time.

## 4. 보존 불변조건 = I-01~I-12 + E-01~E-09

| ID | 불변조건 (r3 확정 문구) | 새 구조에서의 집 |
|---|---|---|
| E-01 | 리뷰 요청 generation은 rendered digest로 결속되고 회신은 그 digest를 에코한다 | features/external_review — round-5 폐쇄 후 의미 동결 이관 |
| E-02 | 완료 판정은 mutable cache가 아니라 verbatim body의 읽기 시점 재파생에서 온다 | DB는 cache만; 재파생 경로가 권위 |
| E-03 | runner probe proof는 checkout·machine·principal·runtime(config 내용 포함, round-5) 축 exact-match에서만 재사용, not-observed는 상태 동등 | probe 테이블 + adapters/codex |
| E-04 | run 기존성·closeout의 **cross-machine 판정 권위는 git-tracked closeout manifest 단독**이다. DB는 그 digest와 관측 시점만 기록하며 충돌 시 git이 우선한다 (§5-4) | runs + project/projections |
| E-05 | attempt **record**는 append-only(재작성·삭제 금지). attempt의 현재 상태는 current-state row의 CAS 갱신으로 바뀌고 모든 전이는 transitions audit에 남는다. decision은 criterion별 evidence에 결속 | store 계약 (§3-1) |
| E-06 | **logical 손상**(개별 artifact·record row)은 해당 run으로 격리되어 타 run 투영을 중단시키지 않는다. **SQLite file corruption 자체는 git evidence나 artifact bytes를 수정하지 않지만 artifact reference 판독을 막을 수 있으므로**, 복구 후 content digest 재검증을 요구한다. filesystem·disk corruption은 별도 위협으로 분리 기술한다 | store + doctor (§5-5) |
| **E-07** (r3 신설) | **어떤 verifier artifact도 자신이 검토하지 않은 result digest를 승인하지 않는다.** 판정은 항상 `(base, patch bytes) → result digest` 쌍에 결속되며, base가 바뀌면 그 판정은 자동으로 이월되지 않는다 (§6 M2-3) | jobs/verification + integration |
| **E-09** (r4 신설 · ADR-0009 개정) | **Durable identity·ownership·attribution·evidence ordering은 대상이 변하지 않아도 독립적으로 변할 수 있는 incidental ambient 값에 결속하지 않는다.** Durable identity는 대상이 변하지 않는 동안 유지되는 intrinsic identifier(예: canonical UUID, content digest, Git object ID)에서만 온다. boot ID·PID·process start token·monotonic time 같은 ambient 값은 계약에 observation scope와 lifetime이 명시되고 그 안에서 재관측될 때 locator 또는 freshness evidence로만 사용할 수 있다. scope 밖·불일치·재관측 불가에서는 `unknown`으로 강등하며 durable identity로 승격하지 않는다. hostname·cwd·mtime/ctime/inode·directory stat/열거 순서는 진단·표시·탐색을 넘는 identity·ownership·attribution 근거가 될 수 없다. filename delimiter 분해도 legacy adapter 밖에서는 owner identity의 추론 근거가 될 수 없다. Git-tracked filename/path는 검증된 intrinsic identity에 대한 주소와 artifact kind 식별에는 사용할 수 있으나, 이름을 분해해 owner identity를 추론할 수 없다. | core/attribution + adapters |
| **E-08** (r4 신설 · r5 정밀화) | **실행 상태 정직성.** 출력 침묵·로그 부재·stderr 오류·heartbeat 부재만으로 실행 종료를 주장하지 않는다. `alive`는 해당 action의 관측 계약이 주는 **긍정적 liveness evidence**로만, `exited`는 **supervisor wait status 또는 action에 결속된 atomic completion evidence**로만 주장한다. 어느 쪽도 성립하지 않으면 **사유를 동반한 `unknown`**이다. **`unknown` 또는 running effect는 destructive resolution의 근거가 될 수 없다**(§3-9). 진행률은 frozen closure와 authoritative transition 사실에서만 파생하며, worker free-text는 liveness·progress·acceptance 어느 것의 증거도 아니다 | runs/observability + core lease + supervisor |

**E-09 개정 note (ADR-0009):** r4 원문은 “**신뢰·귀속 판정은 파일시스템 메타데이터에 결속하지
않는다** — mtime/ctime/inode/디렉터리 stat/열거 순서는 git으로 전달되지 않고 재파생 불가능하므로
권위의 근거가 될 수 없다. 판정 근거는 파일 내용·파일명·git-tracked 사실에서만 나온다 (근거:
round-5 JW-GPT-013 디렉터리 stat→내용 digest, round-6 015 attempt-3의 mtime 단조성 — 같은 실수의
2회 재발)”였다. accepted ADR-0009가 이 r4 문구를 supersede했으며, 위 E-09 행이 현행 확정 문구다.

fail-direction 원칙(모든 신뢰 표면은 fail-toward-verification)은 core 계약으로 성문화 (I-09
구체화). §2-5의 traceability matrix가 이 표의 각 행을 테스트 4층에 결속한다.

| `~/.waystone/cache/worktrees/*` | 순수 cache — 재생성 가능, 처분 대상 아님. delegation 레코드 소비(apply/discard) 시 정리 |
| `~/.waystone/waystone.pre-0.9/*` (legacy seed) | read-only legacy, `chore/migration-sunset` 범위 |

**원칙 보강**: 권위는 ⑴ git에 있거나 ⑵ 재파생 가능하거나 ⑶ **machine-local임이 명시되고 그
한계가 문서화**되어야 한다. ⑶은 §5-1 초판에 없던 층이며, 감사가 적출한 역사적 local-only 기록
(delegation·exposure·consent)과 이 registry가 여기 속한다 — `docs/runtime-state-audit.md`의
"수용된 잔여" 절이 그 문서화다.

원칙(r6 개정): 권위 있는 것은 git에 있거나, 재파생 가능하거나, **machine-local임이 명시되고 그 한계가 문서화**되어야 한다. 세 조건 어디에도 속하지 않는 항목은 M0 finding.

### 5-2. Fact authority matrix (M0-B에서 확정)

**한 줄 원칙: DB는 Git fact를 authoritative하게 복제하지 않는다 — digest와 관측 시점만
기록한다.**

| Fact | Authority | Cache/Projection | 머신 간 전달 |
|---|---|---|---|
| task intent·deps·acceptance | `tasks.yaml` | DB snapshot | Git |
| canonical project mapping | §5-1 machine-tier의 `~/.waystone/projects.json` (machine-local 권위; ADR-0011) | — | 전달 안 됨(로컬) |
| profile routing intent (M3 split 전) | §5-1의 `.waystone/profile.yml` (local 전이 권위; 이후 분리는 `fix/profile-intent-vs-capability-split` 소유) | — | 전달 안 됨(로컬) |
| run runtime state | DB current-state row | status/start-here | 전달 안 됨(로컬) |
| run base/result tree | Git object/SHA | DB reference | Git |
| patch·log·verifier result | content-addressed artifact | DB metadata | §5-4 한계 명시 |
| review request/reply/binding | git-tracked 파일(+GitHub marker) | DB index | Git/GitHub |
| **published run closeout** | **git-tracked closeout manifest** (E-04) | DB digest+관측시점 | Git |
| **action 실행 상태** (r5) | DB current-state row | — | 전달 안 됨(로컬) |
| **heartbeat** (r5) | local mutable telemetry row (leases/action_runtime) | — | 전달 안 됨 |
| **process liveness** (r5) | supervisor/OS의 현재 관측 | — | 전달 안 됨 |
| **run health** (r5) | 위 셋의 read-time projection — **비권위** | status/watch 출력 | 전달 안 됨 |
| local consent | DB (원본 jsonl 보존) | — | 로컬 |
| published policy | git 파일 | local index | Git |

run id uniqueness: 생성 단계 충돌은 §3-3의 random suffix가, 프로젝트 전체 사후 검출은 closeout
manifest(git)가, 로컬 진행 중 run은 DB unique가 담당 — 충돌 시 git 우선.

**개정 (2026-07-20):** 위 §3-3의 random suffix 방식은 ADR-0005의 UUIDv7 generator/CSPRNG
계약으로 대체되었다. 위 원문은 결정 이력으로 보존하며 신규 `run_id`에는 적용하지 않는다.

### 5-3. Cross-machine 경계 (ruling #6)

0.12 계약(권고): session/process crash 자동 resume ○ · 같은 머신 다른 host resume ○ ·
**다른 머신의 nonterminal run resume ×** (task·review·published closeout은 git으로 공유).
`run export`가 portable bundle을 뜻하면 artifacts+checksums+redaction 계약 필요 — 아니면
`inspect --json`으로 명명(권고: 0.12는 inspect만, portable export는 비범위).

### 5-4. Run closeout manifest 최소 계약 (r3 신설 — M0-B에서 확정)

E-04가 cross-machine 권위를 이 파일에 두므로, **이 파일이 다시 새로운 sidecar graph가 되지
않도록** 처음부터 강하게 제한한다.

**제약:** run당 정확히 하나 · publication 이후 immutable · runtime lifecycle 전체를 복제하지
않음 · local DB row의 dump가 아님 · **파일 자신의 publication commit SHA를 내용에 넣지 않음**
(0.11 리뷰 packet이 겪은 self-reference 문제의 재발) · result SHA는 **원격에서 도달 가능한
commit** · 이후 external review는 기존 `docs/reviews/*` protocol로 별도 기록.

```yaml
schema: waystone-run-closeout-1
run_id: ...
base_snapshot_sha: ...
code_result_sha: ...            # integration worktree가 만든 코드 결과
closure_digest: sha256:...
task_ids: [...]
decision_digest: sha256:...
verification:
  integrated_result_digest: sha256:...
  summary: ...
delivery:
  mode: manual|apply|commit|push
  result_ref: ...
review:
  request_digest: ...           # 없으면 null
```

**두 SHA의 구분:** `code_result_sha`(코드 결과) ≠ publication commit(tasks·PROGRESS·이
manifest·review request를 담아 git에 게시된 commit). publication commit은 **"이 파일을 포함하는
git commit"으로 파생**하며 파일 내용이 스스로를 명명하지 않는다.

**M0-B에서 결정할 것:** local verifier artifact가 삭제되면 manifest에는 digest만 남는다. 이를
(a) 최소 criterion verdict + check summary를 manifest에 포함해 완화할지, (b)
`deep audit requires local artifact store`라고 정직하게 명시할지 — **권고: (b)**. (a)는
manifest를 DB dump로 팽창시키는 첫걸음이다.

### 5-5. SQLite 물리 운영 계약 (r3 신설 — M0-B에서 확정)

**Filesystem.** WAL은 NFS·공유 mount·동기화 디렉터리에서 안전한 기본값이 아니다. 계약:
init/doctor에서 filesystem을 판정하고 — **권고안** — 기본은 project-local `.waystone/state.db`,
미지원 filesystem이면 **typed refusal + machine-local 재배치 제안**(`$WAYSTONE_HOME` 아래,
canonical project root 경로로 키잉, 프로젝트에는 포인터만). silent fallback 및 무단 journal
mode 강등 금지 — 다른 mode로의 전환은 명시적 승인 시에만. (전면 machine-local 배치 안은
ruling #4의 하위 결정으로 사용자 판단에 올린다.)

**Backup.** WAL 활성 상태의 raw file copy를 migration backup으로 쓰지 않는다 —
`sqlite3.Connection.backup` · 명시적 checkpoint 후 일관 backup · `VACUUM INTO` 중 하나만.

**Artifact GC.** nonterminal run의 artifact 삭제 불가 · published closeout이 참조하는 artifact
기본 보존 · failed/discarded attempt는 명시적 retention 경과 후에만 · GC는 DB row 삭제가 아니라
**mark-and-sweep** · 삭제 전 digest/reference 검증 · `doctor`는 **dangling reference**(참조는
있는데 bytes 없음)와 **orphan artifact**(bytes는 있는데 참조 없음)를 구분해 보고한다.

### M2 — Multi-job run: bounded closure + integration worktree

초판 계약(fleet 실측 규칙: acceptance 없는 dispatch 거부, tempdir 검증 격리, env 정규화, rc
명시 캡처) + 계약 5건:

**M2-1. Job base 규칙.** wave N 병렬 job = 동일 integration HEAD에서 시작; accepted 결과는
결정적 순서로 integration commit; wave N+1 downstream은 **통합 후의 새 HEAD**에서 시작(최초
run base 아님).

**M2-2. Delivery 상태·정책 (r3 재설계 — "사용자 질문 0"과의 모순 해소).** r2는 기본 정지를
`ready-for-delivery`로 두면서 routine run 질문 0을 동시에 요구했다 — 수동 승인을 대화형
질문으로 받으면 둘은 동시에 성립하지 않는다. 해소: **정책을 run 시작 시 고정한다.**

```text
delivery_policy: manual | apply | commit | push
review_policy:   off | publish-after-push
```

- **기본값은 `commit` + `review_policy: off` (ruling #5 확정)**. `init`은 이를 묻지 않는다 —
  합리적 기본값 + 프로젝트 설정 override, 대화형 질문 0.
- `manual` → run은 `waiting_user(reason=delivery-approval)`에 머문다. 모델은 **질문하지 않고
  상태를 보고만** 하며, 사용자가 `waystone run deliver <run-id>`를 실행한다.
- `apply|commit|push` → 설정된 정책대로 자동 진행.
- **commit이 기본이라는 사실의 안전 귀결 (r5)**: 사용자 승인 없이 커밋이 발생하므로 M2-4의
  live-tree drift 3분기와 **delivery 직전 dirty-tree snapshot 재검증이 선택이 아니라 필수**다.
  push와 review publication은 기본 비활성 — **저장소 밖으로 나가는 행위는 명시적 상향 시에만**.
- review publication은 **push 또는 도달 가능한 commit이 있을 때만** 가능.
- `ready-for-delivery`는 terminal이 아니라 **resumable waiting state**.
- `completed`는 선택된 delivery/review policy가 충족된 뒤에만.

상태열: `… → integrating → final-verifying → ready-for-delivery → delivering → delivered →
[review-published] → completed`.

**M2-3. Integration 이후 evidence 재결속 (r3 신설 — E-07의 구현).** 같은 wave의 job A·B가
모두 `H0`에서 시작해 각각 검증됐어도, A를 먼저 통합하면 B는 실제로 `H1 = H0+A` 위에 적용된다.
B의 verifier는 `H0+B`를 봤지 `H1+B`를 보지 않았다 — declared scope가 겹치지 않아도 semantic
interaction은 가능하다. 계약:

1. 정확한 patch를 새 integration HEAD에 적용한 뒤 **task-specific deterministic check를 재실행**한다.
2. reconciliation이 patch bytes나 result tree를 바꾸면 **새 attempt**로 취급한다.
3. 수정된 patch에는 이전 verify·decision을 재사용하지 않는다.
4. 새 attempt는 independent verification과 criterion verdict를 다시 거친다.
5. 최종 integration tree에는 **closure 전체를 대상으로 한 run-level independent verification**을
   수행한다.

*r3 보강 — 이월(carry-forward)의 정확한 의미.* patch bytes가 불변이고 1의 재실행이 green이면
per-task verdict는 **"(H0, patch B)에 대한 판정"으로서 유효하게 남는다**. 다만 그것이 통합
result digest의 승인으로 **승격되지는 않는다**(E-07). 이월된 verdict는 항상 재검증 evidence
digest를 동반해 기록되며, 통합 결과에 대한 승인은 5의 run-level 판정만이 발급한다. 이 구분
덕분에 매 통합마다 모든 per-task verifier를 재실행하지 않으면서도 E-07이 성립한다.

**M2-4. Live-tree drift 3분기.** base 동일→적용 / 변화가 delivery patch와 완전 무관→정책 결정 /
겹침·판정불가→typed `delivery-drift` waiting. silent stash·3-way 금지(I-05).

*Dirty-tree snapshot 계약 (r3 명시).* run base snapshot은 0.11 delegation과 동일하게 **HEAD +
staged + unstaged tracked + non-ignored untracked**를 포함한다. 최종 delivery patch는 이
synthetic snapshot과 integration result의 차이이며, **delivery 직전에 live tree가 그 snapshot과
여전히 동일한지 재검증**한다. 그렇지 않으면 사용자가 시작 시 보던 상태와 worker가 작업한 상태가
어긋나거나, 기존 uncommitted 변경이 delivery patch에 중복 포함된다.

**M2-5. Goal selector 결정론화.** goal은 coordinator가 explicit task id 집합을 제안 → 엔진이
freeze + closure digest 기록 (+최대 task 수·scope·budget 제한). 자연어 scope 확장 재유입 금지.

**직렬화 규칙 (r3 정정).** 실행 전에 알 수 있는 것은 **declared scope overlap**뿐이다:
선언 scope 겹침 → 사전 직렬화 / scope 미선언 → 보수적으로 직렬화 / 선언은 분리됐는데 실제
changed file이 겹침 → **integration 시 scope violation + reconciliation**. "실제 same-file 편집의
사전 자동 감지"는 불가능하므로 주장하지 않는다.

**Exit (회귀 fixture 포함):** 독립 3 task 병렬 + 2단계 dep chain 자동 진행 / kill→resume 중복
없음(§3-4 정의) / live tree는 delivery 전 불변 / **declared scope가 동일한 두 job의 자동
직렬화**(실사고 재현 fixture) / accepted patch 중복 적용 불가 / downstream base가 통합 HEAD임을
검증하는 fixture / **재통합 후 evidence 재결속 fixture**(A 통합 후 B의 verdict가 자동 승격되지
않음) / **부분 unknown fixture (r5)**: 한 job unknown + 다른 독립 job alive → run은 dead/stalled가
아니라 `running(degraded)`, 안전한 독립 job은 계속 진행, unknown job의 dependent·integration만 차단.

**추가된 범위(2).** ①  진행 중 run의 상태·진행 상황을 사용자가 직관적으로 확인하는 설계를 1급
범위로 편입: §3-8 실행 관측 계약(liveness / progress / current 3분리와 각각의 honest-unknown
규칙) · **E-08** 신설 · `run watch` 표면 · M0-B ADR 산출물 · M1-B(엔진 신호)·M3(사람용 뷰)
이관 · complexity budget 3행 · 리스크 1행 · task 2건(`feat/run-liveness-signal`,
`feat/run-progress-view`). ② **E-09**(파일시스템 메타데이터를 신뢰 근거로 쓰지 않음) — round-6에서 같은 실수가 2회 재발해 개인 규율이 아니라 불변조건으로 승격.

**설계 근거는 실사고다.** 2026-07-19 round-6에서 조정자가 러너의 생사를 판정하지 못해 살아 있는
delegation을 discard하려 했다(§0 왜 지금). 원인은 "출력이 없다"를 "죽었다"로 접은 것 —
§3-4의 "관측 채널 불가" 열과 정확히 같은 오류 유형이다. 따라서 관측성은 UX 편의가 아니라
**fail-direction 규율의 한 사례**로 편입했고, 그래서 budget 항목이 아니라 불변조건(E-08)이 됐다.

### 3차 리뷰(**설계 승인**; 필수 2 + 정밀화 8) → r5

**수용(9).** ① 취소·quiescence·cleanup 안전 계약 신설(§3-9) — *리뷰 최대 기여*: record.lock이
**우연히** 제공하던 "실행 중 cleanup 차단"이 lock 분해와 함께 사라진다는 지적. 실사고에서 나를
막아준 게 설계가 아니라 우연이었다는 진단이 정확하다 ② E-08 양방향 정밀화(침묵≠종료 증거,
그러나 종료 관측=종료 증거 — 과잉 적용 시 `exited`를 말 못 하는 반대 오류) ③ **`stalled`를 FSM
state가 아닌 derived health로** — heartbeat 신선도는 읽기 시점 속성이라 상태로 저장하면 조회가
상태를 바꾸고 audit이 오염된다 ④ liveness를 action/job 단위로 계산하고 run은 집계
(`running(degraded)`) ⑤ progress 분모를 action 수 → **frozen closure**로 (action은 retry로
증가하므로 분모 부적격) ⑥ `status`/`watch`의 엄격 read-only 계약 + heartbeat는 transitions가
아닌 mutable telemetry row로(WAL·audit 오염 방지) ⑦ process identity 최소 집합 + supervisor
소유(worker가 자기 heartbeat/completion을 쓰지 않음) + monotonic clock·boot identity ⑧ fault-injection
fixture 5건 + M2 부분-unknown fixture 1건 ⑨ 성공 정의 5의 과대 약속 교체("언제나 생사를 안다"
→ "확정 가능한 것과 불가능한 것을 구분해 안다").

**조정(1) — M0-A 순서.** 리뷰는 `014·015 폐쇄 → 재리뷰 → 0.11.2 → tag → freeze`를 권고했다.
**리뷰어가 알 수 없었던 두 사실 때문에 조정한다**: ⑴ round-6에서 각 4회전을 시도해 발산을
확인했고(014 a4는 정상 입력 위양성 차단을 유입) 사용자 결정으로 M1 이월했다 ⑵ 두 결함은 전부
**PR-mode 표면**에 있고 이 저장소의 dogfooding은 **packet mode**라 개발 하네스 경로를 지나지
않으며, merge gate는 두 경우 모두 계속 차단한다. 따라서 0.11.2를 내지 않고 **0.11.1을 baseline으로
tag**하되, ⑴영향 범위 문서화 ⑵M1-B 수용 기준 편입을 조건으로 건다(§6 M0-A). 리뷰의 우려
