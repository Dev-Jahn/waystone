## 3. Trace schema (고정 전 provenance 필수 — 불변식 #11)

**Provenance 규약 (schema 전체에 일관 적용):**
- 레코드에서 직접 읽은 값 = **explicit** (별도 표기 없음 — 기본값).
- 규칙으로 유도한 의미 label = **inferred**: 반드시 `{"value": ..., "provenance": "inferred", "rule": "<rule-id>"}` 형태 또는 필드 그룹에 `"rule": "<rule-id>"` 동반. rule-id는 버전 포함(예: `verify-cmd-v1`).
- 판정 불가 = **unknown**: `{"provenance": "unknown"}` — 추측으로 채우지 않는다.
- 파일 상단 상수 `PARSER_VERSION = "waystone-trace-1"` — 모든 산출물에 기록(§3.8 evidence integrity).

**`sessions.jsonl`** — transcript 파일당 1행 (main/subagent/workflow-subagent 각각):
```
project            slug 그대로 (explicit — §4.2: 자동 병합 금지, cwd는 별도 필드로 관측만)
session_id, kind(main|subagent|workflow_subagent), agent_id?, workflow_id?
file               절대경로 (evidence pointer 기준점)
agent_meta         {agentType, description, spawnDepth} | null   (meta.json, explicit)
cwd, git_branch    최빈값 (explicit)
started_at, ended_at, cli_versions
models             {normalized_model: message_group_count} (explicit)
turns              user_instruction 수 (inferred, rule=turn-index-v1 — cli_control/compact/meta 제외 로직은 fable 이식분)
usage              message-group 단위 dedupe 후 합산 {input, output, cache_read, cache_creation} (explicit)
tools              category별 tool_use 수 {shell, file_read, file_write, agent_spawn, ...} (categorization rule=tool-category-v1)
errors             {api, tool, parse} 수 (explicit)
delegations        agent_spawn tool_use 수 (explicit)
verification       {runs, failed, rule: "verify-cmd-v2", provenance: "inferred", examples: [{line, head}]}
                   — Bash command를 test/lint/typecheck 패턴으로 분류(v2: build는 별도 `build` 필드로 분리; test/`tests/`
                   경로는 러너로 시작할 때만 verification — `cat tests/x`류 오분류 제거). 매칭 안 되면 세지 않음(unknown ≠ 0으로
                   오인 금지: 비매칭 shell 수를 unclassified_shell로 별도 보고)
retry_loops        {count, rule: "same-cmd-refail-v1", provenance: "inferred", examples: [{line, head}]}
                   — 동일 정규화 command가 is_error 결과 후 재실행되는 연쇄(≥3)
context_heavy      {tool_results_over_100kb, max_result_bytes} (explicit)
```
- **내용 비저장 원칙:** prompt/파일 내용/명령 전문은 저장하지 않는다. `head`류는 최대 120자 truncate(민감정보 표면 최소화 — 산출물이 local-only여도 동일).
- **evidence pointer** = `file` + 라인 번호(1-based). examples는 최대 5개.

**`delegations.jsonl`** — agent_spawn tool_use당 1행:
```
project, session_id, file, line
tool               Agent|Task|Workflow|SendMessage (explicit)
subagent_type, model_requested (input에서, explicit; 없으면 null)
resolved_model, agent_id, status, is_async   (toolUseResult에서, explicit; 결과 없으면 provenance: unknown)
linked_transcript  subagents/agent-<agent_id>.jsonl 실존 시 경로, 아니면 null (explicit 링크)
```

**`parse_coverage.json`**:
```
parser_version, generated_from(sources 목록), files_by_kind, files_skipped(kind unknown_*),
event_type_counts, unknown_raw_types, record_parse_errors, replayed_records_skipped,
partial_tail_lines(활성 세션 미완 라인 수), row_totals
```
- §3.8: unknown/에러는 **조용히 버리지 않고 여기 집계**. 단 방어적 adapter/versioning 계층은 만들지 않는다.

## 4. Discovery 규칙

- source 루트 바로 아래 디렉터리 = project slug (leading `-` 주의: 경로 연산은 pathlib만, 쉘 인자로 넘기지 말 것).
- slug당 재귀 walk → `detect_kind`: 위 레이아웃 매핑. `memory/`, `workflows/scripts/*.js`, `tool-results/*`는 **manifest에 kind로 기록만** 하고 파싱하지 않음(M1).
- 스카우트가 발견한 spurious project(다른 세션의 workflow cwd로 생긴 `.jsonl` 없는 dir)는 자연히 files_by_kind에 transcript 0으로 드러남 — 특별 처리 없음.
- 활성 세션의 **미완 마지막 라인**: JSON decode 실패 시 그 라인이 파일 마지막이면 `partial_tail_lines`로 집계(parse_error와 구분), 아니면 parse_error.

## 9. `waystone improve audit` — deterministic audit facts (task #5)

**입력:** out dir의 `sessions.jsonl` + `delegations.jsonl` + `reviews.jsonl` + `parse_coverage.json`만 읽는다(원 로그 재파싱 금지 — 계층 분리). 누락 파일은 해당 lens를 `skipped_lenses`로 보고.

```

```
0.7 활성 lens (설계 §4.3, guard·waiver lens 제외):
1. `main_direct_work` — main transcript의 file_write·shell 상위 세션(절대량+비율), delegation 0인 heavy 세션
2. `verification_debt` (v2) — file_write>0 && verification.runs==0 && build.runs==0 세션 수/비율(passing build도 검증이므로 debt 제외); **build_only_sessions**(build만 있고 test/lint/typecheck 없음) + **unclassified_shell 총량 병기**(분류 한계의 정직한 표기)
3. `retry_loops` — session rows 집계 + 상위 예시
4. `context_heavy` — 100KB+ tool_result 유입 세션, max_result_bytes 상위
5. `delegation_pattern` — model_requested/resolved_model/subagent_type/status 분포, async 비율, workflow 사용량
6. `error_landscape` — api/tool/parse 오류율 per project
7. `review_association` — reviews.jsonl의 round·severity 집계를 project 수준에서 병치. **round↔세션 mapping은 0.7에서 provenance: unknown** — timestamp 휴리스틱으로 추측하지 않는다(불변식 #11)
8. `coverage_caveats` — parse_coverage의 unknown/partial/skip을 facts에 그대로 승계(리포트 신뢰도 표기용)

**CLI:** `waystone improve reviews [--out DIR]`, `waystone improve audit [--in DIR]`(기본 in=out 기본값). exit code 계약 동일.

# 수직절단 5단계 설계 (task #6)

## 11. `/waystone:improve` skill + `waystone improve decide`

**skill (`skills/improve/SKILL.md`, ~100–120줄, 기존 skill 컨벤션):**
1. **수집(결정적):** `waystone improve trace` → `waystone improve reviews` → `waystone improve audit` 순서로 실행(기본 source; 사용자가 대상 지정 시 --source/--project 전달). 이전 `decisions.jsonl`이 있으면 읽는다.
2. **해석(모델, §3.10 2단계):** facts.json의 lens별 사실 + coverage_caveats + evidence pointer를 근거로 recommendation 도출. **facts에 없는 주장 금지.** 각 recommendation에 반드시: 근거 lens·수치, evidence pointer(파일+라인), 그리고 **근거 강도 표기** — inferred rule 기반이면 "패턴상 추정(rule-id)", explicit이면 단정 가능(불변식 #11의 사용자-facing 구현). coverage_caveats가 유의미하면 리포트 신뢰도 한계를 명시.
3. **제시·기록:** recommendation마다 AskUserQuestion으로 승인/거부(범용 wizard 금지 — 부록 A). 결정은 `waystone improve decide`로 기록. **이 버전에서 승인은 "기록"일 뿐 자동 적용 없음** — 적용은 사용자가 직접(리포트가 무엇을 하면 되는지는 알려줌).
4. **재발 억제:** rec_id는 model이 `<lens>/<kebab-gist>` 형태로 mint(같은 취지 재발 시 같은 id가 되도록 안내). 이전에 거부된 rec_id는 새 evidence가 실질적으로 다를 때만 재제시.
- 성숙도 정직성: round/review evidence가 얇으면(Bootstrap/Calibrate) 리포트 서두에 그 수준을 명시하고 soft recommendation으로만.

**`waystone improve decide <rec-id> accept|reject [--title T] [--note N] [--out DIR]`:**
