## 총평

이전 설계안보다 훨씬 낫습니다. 특히 다음 변화가 문서를 단순한 기능 목록이 아니라 **제품의 운영 헌장**에 가깝게 만들었습니다.

* 전체 비전을 `Observe & Advise → Delegate & Verify → Adapt & Enforce`의 3개 버전으로 분해한 점
* 특정 모델이 아니라 역할·capability·evidence를 중심 설계 단위로 삼은 점
* shadow replay가 검증할 수 있는 것과 없는 것을 명시적으로 제한한 점
* review finding을 핵심 supervision으로 격상한 점
* main-as-orchestrator를 기본으로 두고 scale-up topology를 별도로 연 점
* base preset과 adaptive overlay를 구분한 점
* 개인 도구라는 위상을 robustness·통계적 엄밀성·개발 우선순위의 트레이드오프에 연결한 점

전반적으로 사용자의 의도가 잘 반영되었고, 지금 상태에서도 구현 방향을 통제하는 상위 설계 문서로 충분히 기능합니다.  기존 CLAUDE.md의 품질 우선, 단순성, 명시적 검증, main의 task 관리, 적절한 Agent/Fork/Workflow 선택 원칙도 단순 복사하지 않고 하네스 철학으로 적절히 추상화했습니다. 

다만 구현에 들어가기 전에 아래 여섯 가지는 문서 수준에서 정리하는 편이 좋습니다. 대부분 기능 추가가 아니라 **개념 축의 혼선이나 향후 잘못된 최적화를 방지하는 수정**입니다.

# 반드시 정리하는 것이 좋은 부분

## 1. `External Delegate`는 role이 아니라 execution backend에 가까움

§8.2에서는 다음을 같은 수준의 worker role로 열거합니다.

* Implementer
* Clerk
* Verifier
* Reviewer
* External Delegate

그런데 Codex 같은 external delegate는 실제로는 역할이 아니라 **역할을 실행하는 backend 또는 transport**입니다.

예를 들어 Codex는 상황에 따라 다음 역할을 모두 수행할 수 있습니다.

```text
Codex-backed implementer
Codex-backed reviewer
Codex-backed test generator
Codex-backed verifier
```

반대로 Claude subagent도 implementer 또는 reviewer가 될 수 있습니다. 현재 구분을 그대로 유지하면 `role-model 정적 binding`과 `model-agnostic routing`에서 축이 섞일 가능성이 있습니다.

다음의 세 축으로 분리하는 편이 더 정확합니다.

### 책임 축

```text
main
orchestrator
implementer
clerk
verifier
reviewer
```

### 실행 방식 축

```text
main session
clean subagent
forked subagent
deterministic workflow
external runner
```

### 모델·도구 binding 축

```text
claude:<model>
codex:<model>
gemini:<model>
기타 backend
```

따라서 §8.2의 `External Delegate`는 worker role 목록에서 빼고, 다음처럼 표현하는 것이 좋습니다.

> Worker role은 Claude subagent, fork, workflow 또는 external runner를 통해 실행될 수 있다. External delegation은 별도 역할이 아니라, 역할을 수행하는 실행 backend다.

이 수정은 문서의 model-agnostic 원칙을 더 선명하게 만듭니다.

## 2. 파생 데이터의 local-only 원칙과 project policy의 배포 범위가 충돌함

§14에서는 trace, audit, policy overlay, evidence 기록을 모두 plugin-local에 두며 공유 repository에 절대 포함하지 않는다고 규정합니다. 프라이버시 관점에서는 타당합니다.

그러나 다음은 서로 다른 성격의 데이터입니다.

### 공유되면 안 되는 행동 데이터

```text
사용자 습관 profile
main 과잉 개입 위치
finding density
raw trace
recommendation history
waiver 패턴
개인별 model/budget 선호
```

### 명시적으로 승인되면 프로젝트 계약이 될 수 있는 정책

```text
이 프로젝트의 acceptance command
delegation 환경 준비 규칙
특정 경로의 scope guard
고위험 변경의 review requirement
project-local agent 정의
project-level hook 설정
```

현재 문구대로라면 후자까지 모두 local-only여서, 같은 프로젝트를 다른 머신에서 사용할 때나 repository를 재클론했을 때 project contract가 재현되지 않습니다. 반대로 후자를 자동으로 repository에 넣으면 행동 기반 개인화 정보가 유출될 수 있습니다.

두 종류를 분리하는 것이 좋습니다.

> **관측·추론으로 생성된 행동 데이터와 미승인 overlay는 항상 local-only다. 사용자가 검토하고 명시적으로 승인한 비민감 project policy는 별도의 materialization 단계를 통해 project configuration으로 승격할 수 있다.**

개념적 흐름은 다음과 같습니다.

```text
local recommendation
    ↓ 사용자 승인
sanitized project policy candidate
    ↓ 적용 범위 선택
local project policy 또는 committed project policy
```

여기서도 자동 커밋은 하지 않고, interactive consent를 요구하면 됩니다.

이 구분이 없으면 0.9의 `Project overlay`, project-level hooks, project-local agents와 §14의 local-only 규칙 사이에 계속 긴장이 남습니다.

## 3. Review supervision을 사용하려면 `effective policy exposure`가 반드시 기록되어야 함

문서는 review finding을 route, guard state, verification evidence와 연결한다고 설명합니다. 방향은 정확합니다. 그러나 prospective evidence로 정책 효과를 판단하려면 각 round가 **어떤 정책 아래 실행되었는지**를 알아야 합니다.

예를 들어 다음 두 round를 비교하려면:

```text
Round A:
  verification guard = observe
  route = main direct implementation
  major finding = verification 부족

Round B:
  verification guard = warn
  route = delegated implementer
  major finding = 없음
```

단순히 현재 policy file을 읽는 것으로는 부족합니다. 정책은 round 이후 바뀔 수 있기 때문입니다.

따라서 설계 불변식으로 다음을 추가하는 것이 좋습니다.

> 각 round와 delegation은 실행 당시의 effective policy, route decision, model/backend binding, guard stage를 식별할 수 있는 immutable policy exposure record를 가진다.

반드시 전체 policy를 복사할 필요는 없지만, 최소한 다음 의미는 보존돼야 합니다.

```text
policy fingerprint
policy version
적용된 overlay 집합
활성 guard와 stage
route와 execution backend
명시적 override 또는 waiver
```

이것은 단순 provenance가 아니라 adaptive loop의 성립 조건입니다. 이것이 없으면 나중에 finding이 줄었더라도 어느 정책 변화와 연결되는지 알 수 없습니다.

또 하나, review finding과 route의 관찰적 상관관계를 바로 인과로 해석해서는 안 됩니다. 어려운 task가 더 강한 모델이나 복잡한 topology로 배정되는 선택 편향이 있기 때문입니다.

문서에 다음 경계를 명시하는 것이 좋습니다.

> 과거 로그와 review finding의 연관은 policy candidate를 생성하는 데 사용한다. 특정 route 또는 guard가 품질을 개선했다고 주장하는 근거는 prospective 적용 이후의 evidence, 직접적인 contract violation 예방, 또는 명시적으로 통제된 비교에서만 얻는다.

`fabulous-fable`의 무거운 통계 체계를 가져올 필요는 없지만, 이 **관찰적 후보 생성과 prospective 검증의 구분**은 유지해야 합니다.

## 4. §3.8의 “Robustness보다 impact”가 지나치게 절대적임

이 철학 자체는 이해되고, 개인용 도구의 위상에도 맞습니다. 문제는 현재 표현이 다음과 같이 매우 강하다는 점입니다.

> 포맷 versioning, adapter 계층, 방어적 스키마 감지 같은 사전 엔지니어링을 하지 않는다.

그리고:

> 기능적으로, 정성적으로 더 impactful한 작업이 동작의 견고함을 높이는 작업보다 항상 우선한다.

여기서 **speculative compatibility infrastructure를 만들지 않는 것**과 **분석 유효성을 확인하지 않는 것**은 구분해야 합니다.

Trace parser가 포맷 변경으로 일부 record를 조용히 버리거나 잘못 분류하면, 이후 audit과 policy recommendation 전체가 오염됩니다. 이는 robustness 문제가 아니라 evidence integrity 문제입니다.

따라서 최소한 다음은 필요합니다.

```text
알 수 없는 record를 조용히 분류하지 않음
parse coverage와 unparsed record 비율을 보고
필수 필드가 사라지면 trace를 fail-loud 또는 degraded 상태로 표시
입력 source와 parser version을 산출물에 기록
```

별도 adapter framework나 장기 호환성 계층을 선제 구축할 필요는 없습니다. 하지만 “깨졌는데 정상적으로 분석된 것처럼 보이는 상태”는 반드시 막아야 합니다.

권장 문구는 다음과 같습니다.

> 불확실한 미래 포맷을 위한 범용 compatibility architecture는 만들지 않는다. 다만 trace의 분석 유효성은 타협하지 않는다. 알 수 없는 포맷과 낮은 parse coverage는 명시적으로 드러나야 하며, 조용한 데이터 손실이나 잘못된 정규화는 허용하지 않는다. 포맷이 실제로 바뀌면 그때 필요한 최소한의 hotfix를 한다.

또한 “impact가 robustness보다 항상 우선”보다는 다음이 더 정확합니다.

> 실패가 가시적이고 복구 가능하며 evidence를 오염시키지 않는 범위에서는 broad robustness보다 사용자-visible impact를 우선한다.

이렇게 해도 사용자가 원한 anti-overengineering 철학은 그대로 유지됩니다.

## 5. 0.8의 `personal overlay`는 이름과 진화 경로가 다소 모호함

문서는 0.8에서 사용자 습관과 프로젝트 특성을 하나의 `personal overlay`에 넣고, 0.9에서 user/project로 분리합니다. 조기 분리를 피하려는 의도는 타당합니다.

그러나 다음이 불명확합니다.

* 0.8 overlay는 어느 프로젝트에 귀속되는가?
* 사용자 공통 선호와 project-specific rule이 섞인 상태에서 다른 프로젝트가 이를 상속하는가?
* 0.9에서 어떤 evidence를 기준으로 user/project를 분리하는가?
* 분리 전의 project-specific 항목이 잘못 전역 적용될 위험은 없는가?

0.8에서는 `personal overlay`보다는 **project-local adaptive overlay** 또는 단순히 **adaptive overlay**라고 부르는 편이 낫습니다.

각 delta에는 이미 provenance를 넣기로 했으므로, 내부적으로는 다음 의미를 보존할 수 있습니다.

```text
candidate_scope:
  user_candidate
  project_candidate
  unresolved

observed_in:
  project A
  project B

effective_scope:
  current project only
```

즉, 0.8에서는 모든 adaptive delta를 현재 프로젝트 범위에서만 적용하고, 0.9에서 여러 프로젝트에 반복된 evidence가 있는 항목만 user overlay로 승격합니다.

권장 원칙은 다음과 같습니다.

> 0.8의 adaptive overlay는 적용 범위를 현재 프로젝트로 제한한다. 다만 각 delta가 사용자 공통 성향 후보인지 프로젝트 특성 후보인지 provenance를 기록한다. 0.9에서 cross-project evidence가 축적된 항목만 user overlay로 승격한다.

이렇게 하면 초기 구조를 단순하게 유지하면서도 0.9 migration이 명확해집니다.

## 6. Recommend, warn, enforce의 evidence 기준을 분리해야 함

§6.5에서는 readiness threshold를 관대하게 두고, 반복 패턴이 몇 번 보이면 recommendation을 내리고 적용한 뒤 틀리면 되돌리는 접근을 취합니다. 개인 도구라는 점에서 적절합니다.

하지만 이 원칙이 `enforce` 승격까지 동일하게 적용되는 것으로 읽히면 위험합니다.

각 단계의 비용이 다르므로 evidence 기준도 달라야 합니다.

### Recommend

낮은 threshold가 적절합니다.

```text
반복 관측
합리적인 mechanism
사용자가 판단 가능한 설명
되돌리기 쉬움
```

### Warn

조금 더 높은 threshold가 필요합니다.

```text
여러 번 반복
명시적인 문제 가능성
낮은 warning frequency
사용자가 실제로 actionable하다고 판단
```

### Enforce

강한 조건이 필요합니다.

```text
계약 위반의 의미가 명확
false positive 가능성이 낮음
shadow replay에서 nuisance rate가 낮음
prospective observe/warn evidence가 있음
override 경로가 명확
차단 비용보다 예상 결함 비용이 큼
```

통계적 유의성까지 요구할 필요는 없지만, recommendation을 만드는 기준과 action을 차단하는 기준은 분리해야 합니다.

문서에 다음 문장을 추가하면 충분합니다.

> Readiness gate는 recommendation 생성에 관대하게 적용한다. 그러나 warn과 enforce는 서로 다른 evidence threshold를 갖는다. 특히 enforce는 단순 반복 빈도가 아니라 계약의 명확성, 낮은 해석 모호성, retrospective friction 점검, prospective observe/warn evidence를 요구한다.

# 중요하지만 상대적으로 작은 수정 사항

## 7. 0.7의 “review finding ingest”는 기존 review ingest와 이름이 겹침

현재 `waystone`에는 이미 review reply를 ingest하고 triage하는 기능이 있습니다. 따라서 0.7의 새 기능은 review ingest 자체라기보다 다음에 가깝습니다.

```text
기존 review artifact의 분석용 정규화
finding taxonomy projection
round/task/route와의 analytic join
```

`Review finding ingest` 대신 다음 표현이 명확합니다.

> **Review evidence projection:** 기존 review 및 triage artifact를 audit이 소비할 수 있는 구조화된 evidence view로 투영한다.

그렇지 않으면 0.7에서 기존 기능을 다시 구현하는 것으로 오해될 수 있습니다.

## 8. 0.7의 “리포트 재현성”은 deterministic core와 model synthesis를 분리해야 함

§5.1에서는 동일 리포트를 다시 돌렸을 때의 재현성을 성공 기준으로 둡니다. 그러나 §3.10에서는 audit 해석과 policy 제안을 강한 모델이 수행합니다. 모델 synthesis는 bit-identical 재현성을 보장하기 어렵습니다.

따라서 재현성을 두 종류로 나누는 편이 좋습니다.

### 반드시 결정적인 것

```text
trace event tables
집계 metric
발동 후보 수
review finding join
evidence pointer
parse coverage
```

### 안정성을 평가할 것

```text
recommendation 우선순위
해석 문구
policy candidate grouping
```

권장 표현:

> 동일 입력에 대해 trace와 audit의 구조적 사실·수치는 결정적으로 재현되어야 한다. 모델이 생성하는 설명과 recommendation은 byte-level 동일성을 요구하지 않지만, 핵심 evidence와 우선순위가 불합리하게 변동하지 않아야 한다.

## 9. Shadow replay의 `false positive` 표현은 조금 더 엄밀하게

문서가 shadow replay의 반사실적 한계를 매우 정확히 설명한 것은 강점입니다. 다만 과거 로그만으로 실제 false positive를 자동 판정할 수 있는 것은 아닙니다.

정책이 발동했지만 실제로 불필요했는지는 다음 중 하나가 있어야 판단 가능합니다.

```text
사용자의 retrospective labeling
후속 waiver/override
명확한 contract exception
수동 표본 검토
```

따라서 단순 로그 기반 산출물에는 `false-positive rate`보다 다음 용어가 안전합니다.

```text
estimated nuisance rate
non-actionable trigger candidate
retrospective friction proxy
manual-review rejection rate
```

## 10. Policy lifecycle과 invalidation이 빠져 있음

문서는 policy가 완화·제거될 수 있다고 말하지만, 정책 상태 자체의 lifecycle은 정의하지 않습니다.

Adaptive policy는 다음 이유로 stale해질 수 있습니다.

* main/worker model binding 변경
* 프로젝트 stack 변경
* test command 또는 directory 구조 변경
* Claude Code 기능 변화
* 이전 finding의 근본 원인 제거
* 새 base preset이 동일 문제를 해결

추상 설계 수준에서 다음 정도의 lifecycle을 두면 충분합니다.

```text
proposed
accepted
observing
warning
enforced
suspended
retired
```

그리고 다음 원칙을 추가할 수 있습니다.

> Policy delta는 영구 진리가 아니다. 생성 근거와 적용 환경이 바뀌면 재검토되며, stale evidence 또는 반복 waiver는 suspension·retirement의 근거가 된다.

이것은 정책이 계속 누적되는 문제를 막는 문서의 철학과도 잘 맞습니다.

---

## 11. 같은 logical project의 path alias 처리 의도가 빠져 있음

초기 요구에서 동일 프로젝트가 머신이나 cwd 변경으로 여러 project directory로 분기되는 사례가 있었습니다. Observation layer가 raw path를 곧바로 project identity로 사용하면 evidence가 분산됩니다.

추상 설계에 다음 원칙을 추가하는 것이 좋습니다.

> Project identity는 raw cwd 문자열과 동일시하지 않는다. 여러 머신, path encoding, cwd 변경으로 나뉜 source가 같은 logical project일 수 있으며, 명시적 alias 또는 canonical identity를 통해 결합할 수 있어야 한다. 반대로 근거 없이 자동 병합해서도 안 된다.

구체적 식별 알고리즘까지 설계할 필요는 없습니다.

## 12. 0.8의 `warn` 범위를 명확히 해야 함

0.8은 project-level hook을 아직 설치하지 않고 script/workflow 수준의 검사를 사용한다고 명시합니다. 따라서 이 버전의 warning은 다음 경계에서만 가능할 것입니다.

```text
delegation 시작·종료
round close
review ingest
명시적 guard/check command
```

Main session의 임의 Bash/Edit 시점에 실시간 warning을 주는 것은 0.9 hook 도입 전에는 어렵습니다.

따라서 다음을 명시하는 편이 좋습니다.

> 0.8의 warn은 하네스가 소유한 실행 경계에서 제공되는 boundary warning이다. 임의의 Claude Code tool call을 가로채는 live warning은 0.9의 project-level hook 도입 이후 제공한다.

## 13. §10의 `fail-safe` 용어는 바꾸는 편이 좋음

동일 scope 충돌 시 덜 강제적인 단계로 내리는 것은 이 설계의 목적에는 적절합니다. 그러나 일반적으로 `fail-safe`는 보수적으로 차단하는 의미로도 자주 쓰입니다.

여기서는 다음처럼 표현하는 것이 명확합니다.

> 동일 scope에서 상충하는 policy delta가 존재하면 기본적으로 **least-restrictive resolution**을 사용한다. 충돌은 자동으로 숨기지 않고 friction evidence로 기록한다.

## 15. 0.9의 범위가 상당히 큼

0.9에는 다음이 동시에 들어갑니다.

```text
0.9 core:

0.9 scale-up profile:
  deterministic fan-out workflow

```

# 추가하면 좋은 설계 불변식

현재 문서는 철학이 풍부한 대신, 구현자가 반드시 보존해야 하는 최소 불변식이 여러 절에 분산되어 있습니다. 별도의 짧은 절로 다음을 모으면 문서의 실행력이 크게 좋아집니다.

## 제안: “운영 불변식”

1. **Final acceptance에는 항상 단일 소유자가 있다.** 기본 소유자는 main이다.
2. **하나의 task/worktree에는 동시에 하나의 mutation owner만 존재한다.**
3. **모든 완료·통과 주장은 실제 evidence에 연결된다.**
4. **모든 round와 delegation은 실행 당시의 effective policy exposure를 기록한다.**
5. **관찰적 상관관계만으로 policy 효과나 모델 우열을 주장하지 않는다.**
6. **Recommendation은 자동으로 enforce로 승격되지 않는다.**
7. **알 수 없는 trace format은 조용히 무시되지 않는다.**
8. **Worker는 scope·SSOT·acceptance authority를 스스로 확장하지 않는다.**
9. **Adaptive policy는 생성·승격될 수 있는 만큼 완화·폐기될 수도 있다.**
10. **행동 evidence와 미승인 개인화 산출물은 local-only다.**

이 절이 있으면 이후 구현 PR을 검토할 때도 “이 구현이 설계에 맞는가”를 빠르게 판단할 수 있습니다.

# 유지해야 할 특히 좋은 부분

다음은 수정하지 않고 유지하는 편이 좋습니다.

## Shadow replay 의미론

과거 trajectory에 대한 replay로 품질 향상을 입증할 수 없고, 발동률과 마찰만 평가할 수 있다고 명시한 부분은 매우 좋습니다. 이 구분은 이후 분석이 과도한 인과 주장을 하는 것을 막습니다.

## Main-as-orchestrator 기본값

Main이 task graph, 사용자 clarification, routing, 최종 acceptance를 소유하고, workflow와 orchestrator subagent는 scale-up 경로로만 두는 구조가 적절합니다.

## “추론은 모델에, 반복 절차는 하네스에”

이 문장은 전체 설계의 가장 좋은 핵심 원칙 중 하나입니다. Worktree, environment preparation, external runner, artifact collection은 script가 하고, decomposition·trade-off·acceptance는 모델이 담당한다는 경계가 분명합니다.

## Review 중심 supervision

단순 agent status가 아니라 verified review finding과 remediation을 quality signal로 쓰는 방향은 `waystone`의 기존 강점을 그대로 확장합니다.

## Base preset과 adaptive overlay의 분리

Adaptive system이더라도 안정된 base harness가 observation 실패와 무관하게 작동해야 한다는 원칙이 중요합니다. 이는 개인화가 core workflow를 불안정하게 만드는 것을 막습니다.

## 개인 도구라는 명시적 위상

Broad product compatibility보다 실제 사용자의 workflow impact를 우선한다는 결정은 설계 범위를 통제하는 데 유효합니다. 다만 앞서 언급한 것처럼 analytical integrity까지 포기하는 의미로 확장되지만 않으면 됩니다.

# 최종 판단

현재 문서는 **방향을 다시 잡아야 할 수준의 문제가 없습니다.** 제품 의도, version arc, adaptive loop, evidence philosophy가 대체로 일관됩니다.

우선순위를 매기면 다음 네 가지 수정이 가장 중요합니다.

1. **role과 execution backend를 분리**
2. **local behavior data와 승인된 project policy의 거주지를 분리**
3. **round별 effective policy exposure를 불변식으로 추가**
4. **robustness 비목표와 trace evidence integrity를 분리**

그다음으로 `personal overlay`의 0.8 의미, recommendation/warn/enforce 기준, policy lifecycle을 다듬으면 구현 전 상위 설계안으로 상당히 완성도가 높아집니다.
