## 0. 결정 기록

**범위 결정:**
- **M0. M2 = §5.2 잔여 전부, 이 노트 한 장.** project-local adaptive overlay 저장·적용(observe/warn) / /improve 확장(materialize·Tune) / shadow replay / boundary warn / evidence layer 정식화 / main operating contract 주입 / exposure 확장(round-level + overlays 채움) / verifier codex-companion transport + sandbox 축 정리 / guided skill / smoke 마찰 수정. **어느 것도 0.8.0 밖으로 밀지 않는다.** 구현은 §17의 클러스터 순서(C1–C7)로 커밋.
- **재확인 cut line:** enforce 승격 없음 / user·project overlay 분리 없음 / orchestrator subagent 없음 / CLAUDE.md 이관 없음 / hook 기반 live warn 없음 / cross-project overlay 상속 없음 / **committed project policy materialization 없음**(§14의 승격 경로는 0.9 consent 확대와 함께 — M2 overlay는 전부 plugin-local).

**스펙 결정 (각 항 근거 병기):**
- **S1. Delta 저장 = per-delta JSON 파일** (`overlay/<pslug>/deltas/<delta-id 파일명화>.json`, atomic 쓰기). M1 delegation record의 검증된 패턴(불변 core + mutable 상태 + transitions) 재사용. 이벤트 로그 성격 — 시각 허용(0.7 decisions.jsonl 전례).
- **S2. delta-id = rec에서 유래 시 rec_id 그대로**(`<lens>/<kebab-gist>` — 0.7 재잔소리 억제와 정렬), 수동 생성 시 동일 문법 강제. 파일명화 = `/`→`--`.
- **S3. Delta status enum = {proposed, observing, warning, suspended, retired}.** 설계 §10 lifecycle과의 대응: "accepted"는 proposed→observing 전이 **이벤트**(transitions에 기록), "recommend" 단계는 improve 리포트 층위(delta로 저장되기 전)다. observing = check 활성·기록만(stderr 없음), warning = 기록+stderr. 근거: stage와 status 이중 축은 동일 정보의 중복 — 단일 enum이 단순하고 lifecycle 그림과 손실 없이 대응.
- **S4. Rule = 하네스 구현 rule-id vocabulary + params, 표현식 언어 없음(§3.8).** 경계에서 기계 평가 가능한 것만 rule로: v1 = 2개(§4). trace-lake 통계형 패턴(main 직접 작업 비중 등)은 rule이 아니라 audit lens 소관 — 경계 체크 대상이 아님을 명시. 미지 rule-id 평가 시: 경고를 log에 evaluation-error로 기록 + stderr notice, **host 동작은 절대 차단하지 않음**.
- **S5. Warn은 항상 비차단(§3.5, #6):** 어느 경계에서든 warn 발화·평가 실패가 host 명령의 exit code를 바꾸지 않는다(round close에 warn이 있어도 exit 0). enforce는 0.9.
- **S6. Observe 단계 = 동일 평가·기록, stderr만 억제.** §6.5의 warn 승격 evidence(발화 이력)가 여기서 쌓인다. 모든 발화·비발화 아님 — **발화만 기록**(비발화 기록은 로그 폭증, 기회 수는 replay가 계산).
- **S7. Replay 산출 = fire_rate + 사례 포인터, estimated_nuisance_rate는 라벨 없인 null.** §4.4의 정의상 nuisance 판정은 retrospective labeling이 필요 — 자동 판정치를 지어내지 않는다(#5·#11). 출력 어휘에 품질 이득 주장 금지("would have fired", "estimated nuisance rate (unlabeled: null)"). replay corpus는 rule이 선언(§4): delegation-records 또는 reviews. **계산은 결정적**(동일 입력→byte 동일; replayed_at은 delta 파일에만).
- **S8. Warn 승격 게이트(#6): `promote`는 replay 결과가 delta에 존재해야 허용**(없으면 exit 1 + `run waystone overlay replay first`). demote/suspend/retire는 무조건 허용(#9 — 완화·폐기는 항상 열려 있다).
- **S9. 동일 scope 충돌 = least-restrictive(§10):** 같은 rule을 참조하는 활성 delta가 복수면 유효 단계 = 덜 강제적인 쪽(observing < warning), 충돌 사실을 warnings.jsonl에 `conflict` 이벤트로 기록(다음 improve cycle의 friction 입력).
- **S10. Evidence projection의 join 키 = task-id.** 0.7 reviews 파서(triage 테이블의 task-id 열)와 delegation record(exposure.task_id)가 이미 공유하는 유일한 명시 키. round↔session 연결은 여전히 provenance "unknown"(#11 — 0.7과 동일, 지어내지 않음). 산출물 결정적(무시각).
- **S11. Round exposure 기록 실패 = stderr 경고 + close는 성공 유지.** #4는 기록을 요구하지만, 기록 실패로 이미 완료된 closeout을 롤백하는 것은 과잉(§3.8) — 실패가 가시적이면 충분. delegation exposure(run 시)는 M1대로 기록 실패 시 위임 자체가 실패.
- **S12. Main operating contract = session_context.py 확장, 별도 hook 신설 없음.** 기존 SessionStart 주입 경로에 블록 추가(§10). constitution 본문은 `references/main-contract.md` 정적 파일(≤8줄) — 부록 A 경계 준수: 전역 CLAUDE.md는 읽지도 쓰지도 않는다.
- **S13. Contract의 live evidence는 파생물 소비만:** needs-review delegation 수(레코드 dir 스캔 — 저렴)는 직접, review finding 요약은 evidence.jsonl이 **존재할 때만** 그 요약을 읽음(세션 시작마다 reviews 재파싱 금지 — 비용·결정론). 없으면 생략.
- **S14. Verifier transport = codex-companion, 단일 호출 (R6에서 정정 — parser 실측):** 최초 스펙은 `--help` usage 문자열 기반의 2단계(--wait 후 result --json)였으나, 리뷰가 parser까지 확인했고 본 세션에서 재검증한 결과 **usage 문자열이 비완전**했다: `handleReviewCommand`(codex-companion.mjs:712-719)는 `valueOptions: [base, scope, model, cwd]` + `booleanOptions: [json, background, wait]` + alias `m→model`, 전역 alias `C→cwd`를 수용하고 model은 `executeReviewRun`까지 전달된다. 따라서 **단일 호출**: `adversarial-review --json --wait --scope working-tree -C <worktree> -m <model> <focus>` — 구조화 payload가 stdout으로 온다. **model이 소비되므로 verifier binding도 `backend: "codex:<model>"` 필수**(M1 원칙 일관: binding 부재 시 fail-loud, 모델 추측 금지) — exposure/verify 산출물에 실제 모델 기록. plugin root는 `~/.claude/plugins/installed_plugins.json`의 `plugins["codex@openai-codex"][0].installPath`(실측 shape)로 동적 해석 — 부재/미설치 시 exit 1 fail-loud, silent fallback 금지.
- **S15. Verify는 evidence이지 판정이 아니다(#8):** `waystone delegate verify` 후에도 state는 needs-review 유지. 수락/폐기는 여전히 apply/discard(main 소유). verify 산출물은 `artifact/verify-<n>.json`, provenance = `independent-verifier`(delegate-claimed와 구분되는 제3의 레이블).
- **S16. Companion broker 정리 = 정확 키잉 + shutdown RPC 우선, glob 금지 (R5에서 강화):** state/* glob + 무차별 SIGTERM은 같은 stateRoot에 키된 사용자의 라이브 세션·동시 delegation의 broker를 죽일 수 있다(오폭). companion의 state dir 키 = **`<basename(workspaceRoot)>-<sha256(canonical workspaceRoot).hex[:16]>`**(state.mjs:38-40 실물 확인; workspaceRoot = 해당 cwd의 git root — 우리 worktree의 realpath). 정리 시퀀스: (1) 이 worktree의 키로 state dir **하나만** 도출해 그 broker.json(`{endpoint, pid, pidFile, ...}`)을 읽고, (2) **기록된 endpoint로 `broker/shutdown` JSON-RPC 우선**(broker-lifecycle.mjs:43-48 `sendBrokerShutdown` 실물 확인 — unix socket에 `{"id":1,"method":"broker/shutdown","params":{}}` 1행; 자기식별적이라 pid 재사용 위험 없음), (3) RPC 실패 시에만 broker.json의 cwd가 이 worktree와 일치함을 확인한 뒤 pid SIGTERM. 모두 best-effort — 실패 시 stderr notice(잔존 daemon 명시, 비차단). 위임마다 새 worktree라 방치 시 daemon이 누적되기 때문.
- **S17. Sandbox 축 = role별 고정값, config knob 여전히 없음(R7 정신 유지):** implementer = workspace-write(M1 하드코딩), verifier = read-only(companion adversarial-review가 전송 수준에서 강제). "축 도입" = exposure·profile 문서화이지 사용자 knob이 아니다 — 다른 mode를 요구하는 role이 실재하기 전까지.
- **S18. UV_CACHE_DIR = worktree-local 상시 설정 + snapshot은 명령 한정 pathspec-exclude (R10b에서 방식 교체):** env prep과 runner 실행 env에 `UV_CACHE_DIR=<worktree>/.waystone-uv-cache`를 항상 설정(비-uv 프로젝트엔 무해). snapshot 제외는 repo 메타데이터를 일절 변조하지 않는 pathspec으로: result snapshot의 add를 `git add -A -- . ':(exclude).waystone-uv-cache'`로 실행. **스크래치 실증(2026-07-14):** GIT_INDEX_FILE 하에서 pathspec-exclude가 `.waystone-uv-cache`를 tree에서 배제하고 다른 untracked는 포함(C1·C2 PASS). 최초 안이던 `info/exclude` append는 공유 메타데이터 영구 변조라 기각(R10b), 리뷰 제안 `-c core.excludesFile=<throwaway>`도 동작 확인(C3 PASS)했으나 그 명령 동안 **사용자의 전역 excludesFile을 대체**해 전역 ignore가 무효화되는 부작용이 있어 pathspec 채택 — 부작용 0, 명령 한정(리뷰 의도인 "무변조·명령 한정"을 더 깨끗하게 충족).
- **S19. Tune 프레이밍 게이트(관대, §6.5):** improve 리포트에서 프로젝트가 `rounds_with_feedback ≥ 5 && findings_total ≥ 20`(facts.json review_association 수치)이면 "Tune 단계 — overlay 제안 가능"으로 프레임. 이 게이트는 **recommendation 층위**(관대) — warn 승격은 S8(replay 필수)이 별도로 지킨다.
- **S20. improve→overlay 매핑표는 명시적 유한 표:** rec의 lens가 rule로 대응 가능한 것만 materialization 제안(§7의 표). 대응 없는 rec는 0.7대로 기록만 — "모든 수용 권고의 자동 delta화"는 하지 않는다(§4.4 — 소수 고가치 delta만).
- **S21. Verify 전 worktree 정규화 (R1 — #12의 실제 보장):** "worktree = HEAD base + working-tree 변경" 전제는 delegate가 커밋하지 않았을 때만 성립 — M1 S2는 커밋 여부 무관을 명시적으로 설계했고 run은 HEAD를 리셋하지 않는다. verify는 리뷰 전에 worktree를 결정적으로 정규화한다: `git checkout --force --detach <base_sha>` → `git clean -fd`(untracked 비ignored 제거; **ignored(.venv 등)는 보존** — 스크래치 실증 A3) → empty가 아니면 `git apply artifact/changes.patch`. **스크래치 실증(2026-07-14):** delegate가 커밋한 경우(A1–A4 PASS — HEAD=base 복귀·untracked 정리·binary 포함 result 내용 복원)와 커밋 안 한 경우(B1 PASS — 멱등) 모두 정규화 후 working tree == result 내용. 실패(어느 단계든 rc≠0) = exit 1 fail-loud, verify 미실행, state 불변. 정규화는 내용 동치 변환이므로 이후의 same-base 수동 검증(M1 §12)도 그대로 유효하다(delegate의 로컬 커밋 객체는 -result ref가 이미 내용을 고정하므로 잃는 것 없음).
- **S22. `overlay add`의 provenance는 명시 플래그 채널 (R2):** §3 스키마의 채움 경로를 CLI로 정의 — `--summary`(필수), `--pointers`(반복, 선택), `--expected-effect`/`--risk`(선택, 기본 `""`), `--candidate-scope`(선택, 기본 `unresolved`, enum 검증), `--observed-in`(반복, 선택, 기본 `[현재 pslug]`), `--from-rec <rec-id>`(선택 — **decisions.jsonl의 rec_id 참조 기록용으로 한정, 파일 해석·자동 채움 아님**; evidence.source를 improve-rec으로, evidence.rec_id를 그 값으로 설정할 뿐). 필수(--rule, --summary) 누락 = exit 1. improve 스킬이 이미 읽은 facts.json 수치·포인터로 플래그를 채워 호출한다(§7).

## 2. CLI surface + exit 계약

```
waystone overlay add <delta-id> --rule <rule-id> --summary S [--pointers P]... [--expected-effect E]
               [--risk R] [--candidate-scope CS] [--observed-in SLUG]... [--from-rec REC]
               [--title T] [--root DIR]
waystone overlay list [--root DIR]
waystone overlay show <delta-id> [--root DIR]
waystone overlay promote <delta-id> [--root DIR]     # observing→warning; replay 없으면 exit 1 (S8)
waystone overlay demote <delta-id> [--root DIR]      # warning→observing
waystone overlay suspend <delta-id> [--note N] / retire <delta-id> [--note N]
waystone overlay replay <delta-id> [--root DIR]
waystone check [--root DIR]                          # 명시 경계 — 활성 delta 전부 현재 상태에 평가
waystone improve evidence [--out DIR] [--project SLUG]...   # projection (레지스트리 프로젝트 대상)
waystone delegate verify <delegation-id> [--root DIR]
```

- exit 계약은 M1 §2 그대로 {0,1,2}. 추가 exit 1 트리거: 미지 delta-id/rule-id, delta-id 문법 위반, `add` 필수 플래그(--rule/--summary) 누락·--candidate-scope enum 위반(S22), replay 없는 promote, 미설치 codex plugin에 verify, needs-review 아닌 delegation에 verify, verifier binding의 미구현 execution/entry(R7), **verify 정규화 실패(S21)**, terminal delta에 promote/demote. **warn 발화·평가 실패는 어떤 host 명령의 exit도 바꾸지 않는다(S5).** `waystone check` 자체도 발화가 있어도 exit 0(평가 실행 자체가 성공이면).
- 거주지: `~/.claude/waystone/overlay/<pslug>/deltas/*.json` + `overlay/<pslug>/warnings.jsonl`, `exposure/<pslug>/round-<round-id>.json`, evidence는 improve out dir(`~/.claude/waystone/improve/evidence.jsonl`). `<pslug>` = `_project_slug` 재사용(M1 S9). 전부 plugin-local(#10), 커밋 금지.

## 3. Delta 저장소·스키마 (S1–S3, S9)

`overlay/<pslug>/deltas/<delta-id 파일명화>.json`, atomic(temp+`os.replace`):

```json
{"schema": "waystone-delta-1",
 "id": "verification_debt/writer-agents-skip-verification",
 "title": "...", "rule": "delegation-verification-evidence-v1", "params": {},
 "scope": {"pslug": "...", "root": "..."},
 "candidate_scope": "project_candidate | user_candidate | unresolved",
 "observed_in": ["<관측 프로젝트 slug>", "..."],
 "evidence": {"source": "improve-rec | manual", "rec_id": "...|null",
              "summary": "관측 수치 요약(자유문)", "pointers": ["file:line", "..."]},
 "expected_effect": "...", "risk": "...",
 "status": "proposed | observing | warning | suspended | retired",
 "replay": null,
 "created_at": "ISO-8601",
 "transitions": [{"to": "observing", "at": "ISO-8601", "note": "accepted via /improve"}]}
```

- `add`는 proposed로 생성 후 즉시 observing 전이까지 수행(수락 행위가 add 그 자체 — improve의 AskUserQuestion 통과 후 호출되므로). 수동 add도 동일(사용자가 명령한 것 = 수락).
- **필드 채움 = S22의 명시 플래그 채널**: `--summary`→evidence.summary(필수), `--pointers`(반복)→evidence.pointers, `--expected-effect`/`--risk`(기본 `""`), `--candidate-scope`(기본 `unresolved`), `--observed-in`(반복, 기본 `[현재 pslug]`), `--from-rec`→evidence.{source: improve-rec, rec_id}(참조 기록만 — decisions.jsonl을 해석하거나 자동 채움하지 않음), 미지정 시 evidence.source=manual·rec_id=null. improve 스킬이 facts에서 읽은 값으로 플래그를 채워 호출(§7) — 예: 관측이 단일 프로젝트면 `--candidate-scope project_candidate`.
- 상태 전이 규칙: proposed→observing(add), observing↔warning(promote/demote), 어느 상태→suspended/retired(단, retired는 최종 — 이후 전이 exit 1). transitions에 전부 기록.
- 손상 delta JSON은 M1 H3 패턴: 단건 경로 fail-loud(파일 지목), 스캔 경로는 해당 delta `[corrupt]` 표시 + 평가 제외 + stderr notice.

## 4. Rule vocabulary v1 (S4 — 경계 평가 가능한 것만)

각 rule: (id, 평가 경계, 입력, 발화 조건, replay corpus). **v1은 2개** — bw2 수용 권고와 §7(review 중심 supervision)이 실제로 동기 부여한 것만. 확장은 후속 minor에서.

1. **`delegation-verification-evidence-v1`**
   - 경계: `delegate-run`(needs-review 도달 직후), `delegate-apply`(적용 직전), `check`.
   - 입력: 해당 delegation의 contract.yaml.
   - 발화: `delegate_report.present != true` **또는** `delegate_report.verification`이 빈 리스트/부재. 메시지: `delegation <did> carries no delegate-side verification evidence — verify independently before apply (delegate-claimed 부재는 사실 부재가 아니라 보고 부재)`.
   - replay corpus: `delegations` — 이 프로젝트의 기존 delegation record 중 **contract.yaml이 존재하는 것만**(R8 — failed-env/-runner/-artifact는 contract가 없어 live 경계에서도 평가 대상이 아니므로 모집단에서 제외해 발동률 축을 일치시킴). 기회 = contract 보유 record 수, 발화 = 조건 참인 record 수. 손상·파싱 불가 record = evaluation-error — **기회·발화 양쪽에서 제외**하고 replay 출력에 `evaluation_errors: N`으로 별도 보고.
2. **`round-close-open-findings-v1`** (params: `severities`, 기본 `["blocker", "major"]`)
   - **두 status 축의 구분(R3 — 명문화):** reviews 데이터에는 서로 다른 두 status 축이 있다. ① source=`triage` 행의 `status` = **verdict** `{REAL, REJECTED, NEEDS-RULING, None}`(`_parse_triage`의 `_VERDICT_RE` 실물 — "해결 여부"가 아니라 "finding이 진짜인가"), ② finding-유래 **task의 registry status** `{pending, active, blocked, done, dropped}`. "미해결"은 ①로는 평가 불가능하다 — 축을 혼동한 문자 그대로의 구현은 고치고 닫는 정상 경로마다 발화하는 순수 마찰 guard가 된다(리뷰 실증 논거).
   - 경계: `round-close`, `check`.
   - 입력: reviews_dir의 finding-유래 task 연결(0.7 파서 재사용: `_finding_tasks_by_round`의 task 행 + `_parse_triage` 행의 task_id) × task registry의 현재 상태.
   - 발화: **severity ∈ severities인 finding-유래 task가 round close 시점에 registry에서 {done, dropped} 밖에 남아 있을 때**(= severe finding의 후속 task가 아직 열려 있는데 라운드를 닫음). verdict 축은 별도 필터로만 사용: verdict=REJECTED인 triage 행은 대상에서 제외(기각된 finding), REAL/NEEDS-RULING/None은 포함. **연결 task가 없는 triage 행은 resolution provenance "unknown" — 발화시키지 않고**(#11 — 추측 금지) replay·check 출력에 `unlinked_findings: N`으로 보고만. 메시지에 task·round·severity 나열.
   - replay corpus: `reviews` — 각 과거 round에 대해 "그 round의 close 시점에 이 rule이 있었다면"의 근사로 **현재 registry 상태 기준** 발화 수를 계산(round 단위 기회). 근사임을 출력에 명기(과거 시점의 task 상태는 재구성하지 않는다 — 지어내기 금지).
- rule 공통: 평가 예외(파일 손상·파싱 불가)는 evaluation-error로 log + stderr notice, 발화로 치지 않음(지어내기 금지).

## 5. Shadow replay (S7 — §4.4 의미론 고정)

`waystone overlay replay <delta-id>`:

1. delta의 rule이 선언한 corpus를 스캔(delegations → `~/.claude/waystone/delegations/<pslug>/*`; reviews → 레지스트리 root의 reviews_dir).
2. 결정적으로 계산: `{"corpus": "delegations|reviews", "corpus_size": N, "opportunities": N, "fires": N, "fire_rate": 0.xxxx, "examples": [최대 5개 포인터], "estimated_nuisance_rate": null, "nuisance_provenance": "unlabeled"}` — stdout에 동일 내용(무시각, byte-결정적). delta 파일의 `replay` 키에 `replayed_at`(ISO)과 함께 저장(이벤트 로그 — 시각 허용은 파일 안만).
3. **출력 어휘 금지어**: "prevented", "improved", "benefit" — replay는 발동률·마찰 추정뿐(§4.4 위반 = 설계 위반). 리포트 문구는 "would have fired N/M times (fire rate x). Nuisance rate requires labeling — inspect examples."
- corpus가 비어 있으면(기회 0) fire_rate = null + `empty-corpus` 명시(0으로 가장 금지, #11).

## 6. Boundary warn 엔진 (S5/S6/S9)

`overlay.evaluate_boundary(root, boundary, context) -> list[dict]` — 활성(observing·warning) delta 중 해당 경계를 선언한 rule만 평가:

- **boundary별 context payload (R4 — 평가 대상의 단일 정의처):**

  | boundary | context | rule 1 평가 대상 | rule 2 평가 대상 |
  |---|---|---|---|
  | `delegate-run` | `{delegation_id}` | 그 delegation의 contract | — |
  | `delegate-apply` | `{delegation_id}` | 그 delegation의 contract | — |
  | `round-close` | `{round_id, closing_task_ids}` | — | close 대상 + 전체 finding-유래 open task |
  | `review-ingest` | `{round_id}` | — | 그 round의 finding-유래 open task |
  | `check` | `{}` | **프로젝트의 모든 needs-review delegation** 각각의 contract | **finding-유래 severe open task 전수 스캔** |

  context에 대상이 명시되면 그것만, `check`는 표의 전수 의미론으로 평가한다(대상 부재 시 해당 rule은 그 경계에서 no-op — 발화도 error도 아님).
- 발화 시: warnings.jsonl append + (status가 warning일 때만) stderr `waystone warn [<delta-id>]: <메시지>`. observing이면 log만(S6).
- **호출부 계약**: 호출은 try로 감싸 어떤 예외도 host 흐름에 전파 금지(stderr 1줄 notice) — warn 엔진의 버그가 위임·round close를 죽이면 안 된다(S5). host exit 불변.
- warnings.jsonl 행: `{"at": ISO, "boundary": "...", "delta_id": "...", "rule": "...", "delta_status": "observing|warning", "event": "fire | evaluation-error | conflict", "message": "...", "context": {"delegation_id"|"round_id"|"task_ids": ...}}` — §15.4 지표(경고 빈도·같은 경고 반복 노출)의 원천.
- 충돌 해소(S9): 같은 rule의 활성 delta 복수 → 평가는 1회, 유효 status = 덜 강제(observing 우선), `conflict` 이벤트 1행 기록.

## 8. Evidence projection (`waystone improve evidence`, S10/S13)

- 입력: 레지스트리(projects.json)의 각 로컬 프로젝트 — reviews_dir triage(0.7 파서 재사용) + `~/.claude/waystone/delegations/<pslug>/*`의 exposure/contract/status.

## 11. Verifier transport (`waystone delegate verify <did>`, S14–S16)

- 전제: state == needs-review(그 외 exit 1), verifier binding 존재(§12), codex plugin 설치(installed_plugins.json에 `codex@openai-codex` — 부재 시 exit 1: `codex plugin not installed — install it or bind verifier to another transport`).
- 실행(모두 `_run_companion(worktree, args, record_dir) -> tuple[int, str]` 경유 — 모킹 대상):
  1. **worktree 정규화(S21/R1 — #12 보장):** `git checkout --force --detach <base_sha>` → `git clean -fd` → (empty가 아니면) `git apply artifact/changes.patch`. 어느 단계든 실패 = exit 1, verify 미실행, state 불변. 이로써 delegate의 커밋 여부와 무관하게 "HEAD=base + working tree=result 내용"이 보장된다(스크래치 실증 — §0 S21).
  2. focus text 조립: task packet의 acceptance criteria + changed_files 요약 + "이 worktree의 working-tree 변경(HEAD=위임 base 대비)이 위 기준을 실제로 충족하는지 적대적으로 반박하라" (≤1KB).
  3. **단일 호출(S14/R6):** `node <installPath>/scripts/codex-companion.mjs adversarial-review --json --wait --scope working-tree -C <worktree> -m <model> <focus>` (timeout 1800s; model = verifier binding의 `codex:<model>`). 정규화된 worktree에서 `--scope working-tree`가 정확히 base→result 변경을 리뷰한다. 구조화 payload는 stdout — `artifact/verify-<n>.json`에 저장: `{"schema": "waystone-verify-1", "at": ISO, "transport": "codex-companion:adversarial-review", "backend": "codex:<model>", "provenance": "independent-verifier", "payload": <stdout JSON 원문>}` (n = 기존 verify 파일 수+1 — 반복 verify 허용, 각각 보존).
  4. **broker 정리(S16/R5):** 이 worktree의 정확한 state-dir 키(`<basename>-<sha256(realpath)[:16]>`)로 broker.json **하나만** 대상 — 기록된 endpoint로 `broker/shutdown` RPC 우선, 실패 시 broker.json의 cwd 일치 확인 후 pid SIGTERM. 결과를 stdout 1줄로(잔존 시 명시).
- state 불변(needs-review 유지, S15). companion rc≠0/timeout → exit 1, 산출물 없음, state 불변(verify 실패는 delegation 실패가 아님 — 메시지로 구분).

## 20. codex exec 직접 위임 실전 교훈의 plugin 반영 (2026-07-14, M2 개발 중 실측)

M2 C3–C7 구현을 codex exec(gpt-5.6-sol, xhigh)로 직접 위임하며 얻은 교훈과 plugin 설계의 접합:

- **실측 사실**: codex workspace-write sandbox는 `.git` 쓰기를 차단(`index.lock` 생성 불가 → 커밋 불가). 해제 키는 `sandbox_workspace_write.writable_roots=["<repo>/.git"]`(`allow_git_writes`는 미존재). **waystone delegate의 worktree에서는 이 사실이 곧 "delegate의 임의 커밋은 구조적으로 불가"를 의미**한다 — worktree의 `.git`은 pointer 파일이고 실제 gitdir·objects는 main repo 쪽(workspace 밖)이므로. 이는 의도와 정합(delegate는 파일로 제안, snapshot은 하네스가 host에서) — writable_roots를 delegate runner에 추가하지 **않는다**. S21 정규화는 비-sandbox backend 대비 방어로 유지.
- **작업 유실 교훈**: "막히면 되돌리고 중단" 계약은 GREEN까지 간 작업을 유실시켰다. waystone delegate는 snapshot 기반 artifact라 이 실패 모드가 없음(부분 작업도 result snapshot·patch에 보존). 단 `templates/delegate-prompt.md`의 중단 정책 문구를 "작업을 되돌리지 말고 보존 + JW_REPORT.yaml limitations에 보고"로 보강할 것 — **후속 소항목 (a)**.
- **후속 소항목 (b)**: profile binding에 선택 필드 `effort`(none|minimal|low|medium|high|xhigh) — runner가 `-c model_reasoning_effort="<v>"`로 전달, 미지정 시 codex config 기본. (a)·(b) 모두 C7 이후 fix 라운드에서 반영(스펙 충돌 없음 — 실행 중인 C3–C7 범위 밖).
- **구분 명시**: UV_CACHE_DIR pre-warm은 waystone delegate에는 불필요(env prep이 호스트에서 네트워크 가능 상태로 선행 — S18은 worktree-local 캐시로 이미 해결). pre-warm은 "우리가 codex exec를 개발 위임에 직접 쓸 때"(sandbox 안에서 uv 게이트 실행)만 해당.
