# Waystone 0.13 agent migration guide

이 문서는 0.11.x 프로젝트의 의미를 0.13 `PROJECT_BRIEF.md`로 옮기는 agent용 절차다. 먼저
waystone dev checkout에서 다음 일회성 script를 실행해야 한다.

```bash
uv run scripts/migrate_legacy_to_013.py <project-root>
```

- script가 한 일: `.waystone.yml`의 legacy 키 제거와 `brief:` 추가, 유효한 provisional
  `PROJECT_BRIEF.md` skeleton 생성, 구 생성물 디렉터리 삭제, machine registry의 누락된
  `project_id` 부여
- script가 남긴 일: 판단이 필요한 구 `SSOT.md` 내용 매핑, owner 확인과 adoption, 프로젝트 등록,
  참조 잔재 정리

## 내용 매핑

구 SSOT를 읽고 각 판단을 다음 canonical section에 배치한다.

- 문제, 비전, 대상에 관한 설명 맥락 → `Purpose` narrative
- 원칙과 확정된 방향 → `Commitments`
- 현재 프로토타입에서 지원할 범위 → `Prototype scope`
- 프로토타입 이후의 장기 방향 → `Long-term direction`
- 명시적으로 하지 않을 것 → `Non-goals`
- 아직 검증되지 않은 방향성 베팅과 가설 → `Working hypotheses`
- 답이 정해지지 않은 질문 → `Open questions`
- 기존 판단을 재검토해야 하는 조건 → `Revision triggers`

원문의 표현을 기계적으로 복사하기보다 의미 단위와 확정도를 보존한다. 확정인지 가설인지 애매하면
임의로 승격하지 말고 `Working hypotheses` 또는 `Open questions`에 둔다.

## Fact marker 규칙

- grammar는 `(commitment|prototype|long-term|non-goal|hypothesis|question|trigger)/[a-z0-9-]+`다.
- marker는 문서 전체에서 unique여야 하며 section과 prefix가 일치해야 한다.
- fact 하나에는 판단 단위 하나만 둔다.
- marker 없는 narrative는 `Purpose`에만 둔다.
- marker 모양 토큰을 일반 본문에 쓰면 parser가 거부하므로 서술 중 예시로도 쓰지 않는다.

예를 들어 구 SSOT의 “첫 버전은 로컬 Git 저장소만 지원한다.”는 다음처럼 변환한다.

```markdown
## Prototype scope
- [prototype/local-git-only] 첫 버전은 로컬 Git 저장소만 지원한다.
```

## 검증, adoption, 등록

내용 매핑 후 waystone dev checkout에서 parser 검증을 실행한다.

```bash
uv run scripts/waystone.py brief check <root>
```

owner가 brief 내용을 확인한 근거를 별도 파일로 만든 뒤 adoption하고 프로젝트를 등록한다.
`tasks.yaml`에는 비어 있지 않은 `project:` 값이 있어야 한다.

```bash
uv run scripts/waystone.py brief adopt <root> --evidence <file>
uv run scripts/waystone.py project register <root>
```

## 잔재 정리

매핑과 검증이 끝나면 구 SSOT 파일을 `git rm`하고 README, `CLAUDE.md` 등에 남은 SSOT 참조를
`PROJECT_BRIEF.md` 기준으로 갱신한다. 과거 review packet, ADR, PROGRESS는 이력이므로 수정하지 않는다.

이 migration은 `tasks.yaml`, PROGRESS, ADR의 구조를 바꾸지 않는다.
