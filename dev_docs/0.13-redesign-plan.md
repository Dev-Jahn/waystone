- Status: **implementation authority**
- 기준 commit: `f0294ec7019d8e6deb82f85f77f264aafd6e02a8`
- 상위 권위: `dev_docs/0.13-redesign-mandate.md`
- 실증 근거: `dev_docs/reference/omniphysics-realign.md`
- 대체하는 계획: `dev_docs/0.12.0-refactor-plan.md`의 M1-C 이후 분해 및 그 파생 task 분해
- 산출 목표: 아래 task와 exit만으로 Waystone 0.13의 구현 범위를 닫는다. 새 분해가 필요하면 이
  문서의 decision/open question을 먼저 개정한다.

## 0. 전제, 해석, 성공 조건

### 0.1 전제

1. M1-B의 trust kernel, 즉 transactional store, content-addressed artifacts, lease/fencing,
   external-effect reconciliation, detached supervision, cancellation/quiescence, independent
   verification, digest-bound integration decision과 apply 직전 재검증은 보존한다.
2. 그 위의 입력 freeze, assurance 선택, prompt 조립, review/finding 처리, closeout, status,
   ideate와 public skill/CLI는 0.13 목적에 맞게 교체할 수 있다.
3. `PROJECT_BRIEF.md`와 `brief:`가 신규 canonical surface다. `SSOT.md`, `ssot:`, legacy
   `delegate`/`round` runtime을 함께 운용하는 compatibility layer와 data migration은 이 계획의
   범위가 아니다. 필요하면 0.13 구현 뒤 별도의 일회성 migration script 한 개가 처리한다.
4. 0.13은 현재 검증된 one-task = one-job vertical slice 위에서 intent control plane의 여섯 축을
   완성한다. 일반 multi-job DAG scheduler는 이 여섯 축의 e2e가 통과한 뒤 별도 결정으로 시작한다.
   미완성 scheduler를 끼워 넣는 것은 이 계획의 완료 조건이 아니다.
5. 기존 1,099-test suite의 green은 0.13 acceptance가 아니다. 보존 kernel의 직접 회귀 검사와
   §9의 핵심 e2e 7개만 필수다.

### 0.2 mandate 충돌의 잠정 해석

- §4 표는 `Prototype scope`를 acceptance로 변환할 수 있다고 쓰지만 같은 절의 핵심 계약은
  commitment와 non-goal만 authoritative하다고 쓴다. 잠정적으로 **committed frame의
  `Prototype scope` fact를 commitment의 한 종류**로 판독한다. 최종 ruling은 Q1에 남긴다.
- §4의 `ssot:` legacy alias 제안보다 문서 머리말과 §5의 “migration 고려 불요”가 뒤의 사용자
  지시다. 따라서 0.13 runtime은 `brief:`만 읽고 dual-key fallback을 두지 않는다.
- mandate의 최종 권고 3문장 중 첫 문장의 `SSOT`는 verbatim으로 보존한다. 운영상 그 말은
  `PROJECT_BRIEF.md` 전체가 아니라 committed commitment/prototype-scope fact를 뜻한다.

### 0.3 0.13 성공 조건

0.13은 다음이 모두 참일 때 완료다.

- project intent의 확정·가설·미결정이 machine-readable하게 구분되고 fact별 authority가 보존된다.
- 모든 run이 하나의 lifecycle stage와 그 stage에서 유도한 assurance plan을 freeze한다.
- worker는 판단에 충분한 semantic context와 provenance를 받되 Waystone bookkeeping은 받지 않는다.
- 맥락 부족은 무변경 `context_request`와 `waiting_context`로 표현되고 새 brief/spec/attempt로 재개된다.
- reviewer claim, 진위 판정, remediation disposition, task materialization이 서로 분리된다.
- progress의 기본 단위가 task/test/finding 수가 아니라 evidence-bound `OutcomeDelta`다.
- `ideate`가 신규 framing과 기존 프로젝트 realignment를 같은 surface에서 수행한다.
- §9의 핵심 e2e 7개와 변경한 trust seam의 focused regression이 통과한다.

## 1. 제품 정의와 비가역 설계 원칙

0.13의 정의는 다음 문장이다.

> **세션과 worker를 넘어 프로젝트 의도와 실행을 이어가되, 불확실한 가설은 탐색 가능하게
> 유지하고, 검증·리뷰 비용은 승격 위험에 비례시켜, worker에게 목적에 충분한 맥락을 전달하는
> development run engine**

구현 중 판단이 흔들릴 때 다음 세 문장을 그대로 우선한다.

> **SSOT에는 확정된 commitment만 넣고, hypothesis를 requirement로 위장하지 않는다.**

> **Review finding의 존재와 심각도는 remediation priority가 아니다.**

> **Harness prompt를 줄인다는 것은 workflow protocol을 줄인다는 뜻이지, worker가 올바르게
> 판단하는 데 필요한 의미 맥락을 제거한다는 뜻이 아니다.**

omniphysics의 실패는 단순한 test 과다가 아니었다. 제품과 성공 장면이 미결인 상태에서 가장 어려운
연구 축과 전역 assurance를 먼저 고정했고, 실제 capability 변화가 없는 review/fix/docs 작업이 task
완료 수를 늘렸다. 따라서 0.13은 “검증을 약화”하지 않는다. 검증 권한을 `explore → evaluate →
promote` 경계에 맞춰 배치하고, objective delta가 없는 활동을 progress로 오인하지 않는다.

## 2. 대상 아키텍처

### 2.1 전체 흐름

```text
PROJECT_BRIEF.md + owner request + tasks.yaml + accepted ADR/evaluation spec
              │                 fact별 authority/ref
              ▼
      ProjectFrame reader + WorkBrief assembler
              │                 lifecycle stage proposal
              ▼
      CompletionContract + AssurancePlan compiler
              │
              ▼
 RunSpec v2(stage + objective/brief/completion/assurance digests)
              │
              ▼
 M1-B trust kernel(store/artifact/lease/effect/supervisor/transport)
              │
              ├── explore: worker ──► immutable candidate/learning
              ├── evaluate: frozen candidate ──► read-only evaluator ──► evidence
              ├── promote: evaluated candidate ──► verify/review/decision/apply
              │
              └── context_request ──► waiting_context
                                          │
                              revised WorkBrief + RunSpec + attempt
                                          │
                                          └────────── resume
              │
              ▼
 Finding claim → validation → disposition → 선택된 work만 tasks.yaml
              │
              ▼
 OutcomeDelta + closeout → outcome ledger CAS → status / ideate realignment
```

Control plane은 별도 workflow OS가 아니다. `RunSpec`을 만들기 전 의미 입력을 조립하고, M1-B
kernel이 실행한 뒤 결과의 review·progress 의미를 판정하는 얇은 typed layer다. safety preflight,
lease, effect reconciliation, sandbox, artifact integrity는 stage와 무관하게 항상 적용한다. stage가
바꾸는 것은 **결과를 어떤 수준의 증거로 승격할 수 있는가**이지 execution safety가 아니다.

### 2.2 fact별 authority

| Fact | 단일 authority | 허용 projection/cache | 금지 |
|---|---|---|---|
| project commitment·prototype boundary·non-goal | committed `PROJECT_BRIEF.md`의 해당 fact bytes | `ProjectFactRef`의 commit/path/id/digest | 문서 전체를 global SSOT로 취급, DB payload를 두 번째 authority로 사용 |
| working hypothesis·open question | `PROJECT_BRIEF.md`의 해당 nonbinding fact bytes | WorkBrief의 provenance-preserving ref | hard acceptance로 자동 변환 |
| owner의 이번 run 요청 | host가 보존한 owner-source artifact | RunSpec의 digest/ref | coordinator summary를 owner 말로 승격 |
| accepted architecture decision | accepted ADR bytes | WorkBrief source ref | hypothesis 또는 reviewer 제안을 ADR처럼 사용 |
| evaluation metric·threshold·hold-out generation | Git-tracked `waystone-evaluation-spec-1` bytes | AssurancePlan ref | 결과를 본 뒤 같은 generation의 threshold 수정·재사용 |
| stage completion criterion | content-addressed `waystone-completion-contract-1` bytes와 criterion별 authority ref | RunSpec digest/ref | nonbinding fact를 hard criterion으로 승격, 자유문장 acceptance |
| task intent·dependency·선택된 실행 work | `tasks.yaml` | status의 audit count | 모든 finding을 task registry에 저장 |
| active run/job/attempt/action state | canonical SQLite runtime store | read-only status projection | Git 문서를 active FSM authority로 사용 |
| WorkBrief·AssurancePlan·context request/response | content-addressed artifact bytes | SQLite artifact index/digest | prompt text나 DB 자유문장을 authority로 사용 |
| reviewer가 실제 보고한 claim | digest-bound review feedback + immutable finding claim | finding index | reviewer remediation suggestion을 결정으로 사용 |
| validity/failure mechanism 결정 | append-only finding validation bytes | validation head projection | priority/disposition과 한 artifact에서 동시 판정 |
| impact·relevance·disposition 결정 | confirmed validation에 결속된 append-only disposition bytes | disposition head projection | validation 전 자동 처분·task화 |
| accepted objective progress | `refs/waystone/outcomes` first-parent ledger의 `waystone-outcome-delta-1` bytes | status projection | task/test/finding count 또는 현재 branch history를 progress 권위로 사용 |
| execution·verification evidence | M1-B CAS artifact bytes | digest/reference | worker report를 독립 증거로 승격 |

### 2.3 package와 artifact 배치

새 public concept 하나당 module 하나를 우선하고, 반복 증거 없는 generic framework는 만들지 않는다.

```text
waystone/project/brief.py          ProjectFrame/ProjectFactRef parser와 brief CLI core
waystone/project/context.py        canonical_root/project_id/worktree identity resolver
waystone/jobs/work_brief.py        Semantic WorkBrief 조립·검증·prompt projection
waystone/jobs/completion.py        authority-bound CompletionContract compiler
waystone/jobs/profile.py           canonical role/backend profile과 production RunAssembly
waystone/runs/assurance.py         stage별 action DAG와 AssurancePlan compiler
waystone/runs/worker_result.py     WAYSTONE_RESULT.yaml parser와 completion marker adapter
waystone/reviews/findings.py       immutable claim + validation + disposition
waystone/runs/outcome.py           OutcomeDelta 판정·publication·projection
waystone/cli/{brief,review,status}_group.py
skills/run/SKILL.md                canonical implementation/run loop
```

기존 `waystone/runs/*` trust modules는 이 control-plane module을 호출하도록 개조하되, store/lease/effect
알고리즘을 새 control-plane module 안에 복제하지 않는다.

`assemble_run(start)`는 DB open 전에 ADR-0011 `ProjectContext`를 resolve하고 canonical root/project id,
active worktree, checkout identity를 하나로 결속한다. 그 뒤 canonical 4-role profile의 worker/verifier/
reviewer adapters(coordinator는 host actor), store, artifact store, supervisor, effect executor를 주입한 완전한 `RunAssembly`를
반환한다. public `run start/resume/close`는 이 factory만 사용한다. 현재의 `RunEngine(root)` 빈 default나
test-only `_engine_factory` injection이 production 성공 경로가 되어서는 안 된다.

Git-tracked 신규 layout은 다음으로 고정한다.

```text
PROJECT_BRIEF.md
docs/evaluations/<evaluation-id>/spec.yaml    waystone-evaluation-spec-1
docs/reviews/runs/<run-id>/
  request.md
  request.binding.json
  feedback.md
  findings/<finding-id>/
    claim.yaml                     add-only
    validations/0001.yaml          add-only, 이후 0002…
    dispositions/0001.yaml         add-only, 이후 0002…
```

`docs/runs/<run-id>/{closeout.yaml,outcome.yaml}` pair는 일반 branch가 아니라 engine-owned
`refs/waystone/outcomes`의 한 add-only commit에만 기록한다. 숫자 suffix는 표시·충돌 검출용 순서다.
최신 validation/disposition은 각 `supersedes_digest` 단일 chain의 head이며 timestamp나 directory
enumeration order를 authority로 쓰지 않는다. 둘 이상의 head가 생기면 typed conflict로 멈추고
main/owner가 새 revision으로 결속한다.

## 3. 여섯 축의 상세 설계

### 3.1 Project Brief / `ProjectFrame`

Public 파일은 `PROJECT_BRIEF.md`, config key는 `brief:`다. 내부 parser는 문서 전체를 하나의 truth로
반환하지 않고 다음 value object를 만든다.

```text
ProjectFrame(status, status_ref, facts[])
ProjectFact(id, kind, binding, raw_bytes, source_span, digest)
ProjectFactRef(commit, path, fact_id, fact_digest, binding)
```

`binding`은 문서 status와 fact section에서 결정한다.

| status/section | 실행 의미 | hard acceptance source |
|---|---|---|
| `provisional`의 모든 section | 현재 framing 방향, exploration context | 금지 |
| `committed` / Commitments | owner가 확정한 방향 | 가능 |
| `committed` / Prototype scope | 잠정적으로 commitment subtype | 가능(Q1) |
| `committed` / Non-goals | 금지·범위 경계 | 가능 |
| Purpose | 설명 맥락 | 금지 |
| Long-term direction | 미래 방향 | 금지 |
| Working hypotheses | 반증 가능한 탐색 입력 | 금지 |
| Open questions | 명시적 미결정 | 금지 |
| Revision triggers | realignment 조건 | 직접 변환 금지 |

규칙은 다음과 같다.

- `provisional`에서도 `explore`는 가능하다. 다만 project-wide promotion, public API 고정, durable
  architecture lock-in은 거부한다.
- `committed` 전환은 agent가 대화 종료를 근거로 자동 수행하지 않는다. owner adoption evidence가
  필요하며 그 형태는 Q2의 ruling을 따른다.
- canonical path의 현재 문서가 선언할 수 있는 status는 `provisional | committed`뿐이다. 이전 Git
  revision의 brief는 현재 revision에서 대체된 사실을 reader가 `superseded`로 **파생**하며, 과거 bytes를
  다시 편집하지 않는다. 파생 `superseded` frame은 provenance 조회만 가능하고 신규 run source가 아니다.
- hypothesis 반증은 실패가 아니라 `OutcomeDelta(kind=validated-decision)` 또는
  `no-objective-delta`가 아닌 project learning으로 기록할 수 있다.
- 다른 fact를 편집했다고 무관한 run이 drift하지 않도록 RunSpec은 문서 전체 digest가 아니라 실제로
  사용한 `ProjectFactRef`와 frame status ref만 freeze한다.

### 3.2 Lifecycle stage와 stage별 assurance

`lifecycle_stage`는 run 단위의 닫힌 enum이다. 한 run에 stage를 섞지 않는다. 서로 다른 stage가
필요하면 별도 run으로 분리한다.

| Stage | 목적 | 허용 completion | 필수 assurance | 명시적으로 요구하지 않는 것 |
|---|---|---|---|---|
| `explore` | 답을 찾거나 candidate를 만든다 | `candidate`, `measurement`, `refuted-hypothesis`, `learned-constraint` | execution safety, frozen brief, 실제 수행 self-check의 claim 표기, 유한 attempt/budget | independent verifier, macro review, permanent ADR, permanent regression test, apply |
| `evaluate` | frozen candidate가 frozen evaluation spec을 만족하는지 판정한다 | `validated-decision` 또는 정직한 rejection | candidate SHA/config/spec/seed freeze, read-only independent verifier, result digest binding | mainline apply, 모든 risk에 대한 macro review, permanent test |
| `promote` | durable supported behavior로 승격한다 | applied result + supported scope/accepted risk + outcome delta | committed frame, independent verifier, 필요한 최소 regression contract, risk-gated adversarial review, integration/apply 재검증 | finding별 matrix test, unrelated future hardening |

Explore의 code result는 live/mainline에 apply하지 않고 engine-owned immutable
`refs/waystone/candidates/<run-id>`에 publish한다. Evaluate는 그 candidate SHA를 read-only로 판정하고
evidence만 낸다. Promote는 RunSpec에 freeze된 explicit integration target ref와 expected old OID에
M1-B의 CAS apply를 수행한다. 따라서 “apply 비필수”는 candidate를 잃는다는 뜻이 아니며,
“promotion”은 검증만 끝내고 delivery를 암묵적으로 남긴다는 뜻도 아니다.
Candidate ref publication은 expected-old=`absent`인 creation-only CAS다. 같은 run ref 갱신은 거부하고
수정 candidate는 새 run/ref와 `supersedes_candidate_digest`를 갖는다.

현재 engine의 unconditional `runner → verifier → decision → apply` orchestration은 보존 대상이 아니다.
`AssurancePlan.actions`가 아래 세 DAG 중 하나를 완전하게 선택하고 kernel은 선택된 action만 실행한다.

| Stage | exact action DAG | executor와 mutation 경계 |
|---|---|---|
| `explore` | `worker(base) → result-adapter → candidate-publish → completion` | worker만 candidate tree를 변경한다. candidate publish는 `GitRefEffect` CAS이고 verifier/review/integration apply는 plan이 요구할 때만 추가한다. live target은 변경하지 않는다. |
| `evaluate` | `candidate+evaluation-spec freeze → read-only evaluator → evaluation-evidence → completion` | implementer/worker와 apply action이 없다. evaluator action은 independent `verifier` role binding으로 실행하며 candidate/config를 read-only mount하고 evidence CAS artifact만 쓴다. |
| `promote` | `evaluated candidate freeze → independent verify → risk-gated review → integration decision → target-ref apply → completion` | candidate tree를 변경하는 implementer가 없다. 수정이 필요하면 새 explore candidate와 새 evaluate generation부터 시작한다. apply만 expected-old-OID CAS로 live target을 변경한다. |

각 node는 기존 action/effect row, lease/fence, typed artifact transition을 사용한다. 생략 가능한 node도
`AssurancePlan`에서 `not-required`로 freeze하며 실행 중 임의로 추가·삭제하지 않는다. 0.13 candidate는
같은 canonical project의 locally reachable immutable ref만 허용한다. cross-machine candidate transport는
새 network effect/reconciliation evidence가 없으므로 비목표이며 silent fetch/push하지 않는다. explore
closeout 자체는 remote reachability를 요구하지 않도록 ADR-0006의 E-04를 “live apply/promotion code
result”에만 한정해 amend한다.

stage 제안의 deterministic default는 다음이다.

```text
spike / research / unclear solution     -> explore
frozen candidate judgment              -> evaluate
merge / release / public contract       -> promote
ambiguous                               -> owner 또는 main ruling 전 dispatch 금지
```

`AssurancePlan` compiler가 stage, declared risk surface, evaluation spec, owner policy로 필요한 action을
선택한다. `preflight.py`는 선택된 action의 executable capability/toolchain/digest를 검증할 뿐, “모든
run에는 check 한 개 이상”을 정책으로 하드코딩하지 않는다.

Test lifecycle도 같은 경계를 따른다.

- **Probe:** explore에서 가설을 확인하는 임시 check. candidate worktree/CAS artifact에 남길 수 있으나
  mainline suite 편입 의무가 없다.
- **Evaluation check:** frozen candidate generation을 판정한다. evaluation artifact에 결속되고 다음
  generation에서 교체될 수 있다.
- **Regression contract:** promote된 public behavior, stable trust invariant, 또는 반복 가능한 실제
  production failure만 영구 보호한다. 한 finding을 여러 예제로 복제하기보다 가장 작은 property/
  invariant를 고른다.

review/fix는 frozen `max_review_cycles`를 갖는다. engine의 암묵적 무제한 기본값은 없고 compiler가
0 또는 양의 정수를 freeze한다. Evaluate/Promote의 기본 제안값은 2이지만 owner/project policy가
dispatch 전에 바꿀 수 있다. 같은 promotion intent의 result digest가 바뀌어도 cycle counter를
초기화하지 않는다. 소진하면 `waiting_user(reason=review-cycle-exhausted)`로 전이하고 다음 네 선택만
제시한다: accept risk, supported scope 축소, architecture 단순화, 별도 research track 승인. 마지막
선택을 coordinator가 자동 채택하지 않는다.

### 3.3 Finding record와 disposition

Reviewer는 high-recall defect sensor다. reviewer가 소유하는 것은 target digest에 대한 claim과 그
evidence뿐이다. validity, priority, remediation architecture, milestone 중단 권한은 없다.

논리적 finding은 세 종류의 immutable artifact로 구성한다.

1. `waystone-review-finding-1`: reviewer 원 claim, target/result/review digest, reviewer assessment와
   remediation suggestion을 verbatim provenance로 보존한다.
2. `waystone-finding-validation-1`: coordinator가 실제 코드와 evidence를 재현해 validity와 failure
   mechanism을 판정한다. truth 판정만 가지며 priority나 처분은 갖지 않는다.
3. `waystone-finding-disposition-1`: **confirmed validation digest**와 objective/stage를 입력으로
   impact/exposure/relevance/disposition/scope/cost를 결정한다. validation과 disposition 변경은 각각 새
   revision으로만 기록한다.

필수 축은 다음과 같다.

| 축 | 값 |
|---|---|
| validity | `confirmed`, `rejected`, `unresolved` |
| impact | `blocker`, `major`, `minor` |
| exposure | `common`, `edge`, `adversarial`, `unknown` |
| relevance | `current-objective`, `promotion-bound`, `future`, `out-of-scope` |
| disposition | `fix-now`, `fix-before-promotion`, `backlog`, `accept-risk`, `no-action` |
| remediation_scope | `local`, `bounded`, `architectural` |
| estimated_cost | `low`, `medium`, `high`, `unknown` |

상태 규칙은 다음과 같다.

- `confirmed major`는 자동 `fix-now`가 아니다.
- `unresolved`와 `rejected` validation에는 disposition artifact 자체가 없고 task/blocker를 만들지 않는다.
- `confirmed`여도 아직 처분하지 않은 상태를 허용한다. 다만 promote의 required review scope 안에 있는
  confirmed finding은 모두 disposition head가 있어야 promotion readiness가 성립한다.
- `fix-now`만 현재 objective의 실행 task를 materialize한다.
- `fix-before-promotion`은 해당 objective/result lineage의 `promote`만 막으며 explore/evaluate를 막지
  않는다. 새 candidate를 만들었다는 이유만으로 풀리지 않으며, descendant candidate에서 원 failure
  mechanism을 겨냥한 independent fix evidence가 clearance로 결속될 때만 그 candidate 이후에 풀린다.
  필요할 때만 실행 task를 materialize한다.
- `backlog`, `accept-risk`, `no-action`은 finding catalog에만 남고 `tasks.yaml`에 넣지 않는다.
- `architectural` remediation은 owner ruling 없이 `fix-now`로 만들 수 없다.
- remediation은 root cause 확인 → current objective에 영향을 주는 범위 → local/bounded fix 순으로
  제안한다. 반복 evidence 없는 새 harness subsystem은 별도 owner decision이다.

### 3.4 Semantic Work Brief와 `context_request`

WorkBrief는 task registry에 의미 문맥을 복제하지 않는다. coordinator가 dispatch 직전에 조립하고
CAS artifact로 publish하며, RunSpec은 exact digest를 freeze한다. 각 context item은 다음 provenance
중 하나를 잃지 않는다.

- `owner-source`: 사용자 발화 보존본 또는 Git-tracked owner fact.
- `harness-observation`: Git/store/artifact에서 재도출한 관측 사실.
- `coordinator-summary`: main의 해석·압축. owner authority가 아님을 prompt에 표시한다.

Prompt projection은 다음 section만 갖는다.

```text
Why this matters
Current context
Goal and expected outcome delta
Fixed decisions / worker freedom / escalation boundaries
Acceptance or learning criteria
Relevant sources
Report / context request
```

registry path, round, overlay, artifact filename, transition, retry, lease, report 수집 절차 같은 state-machine
bookkeeping은 engine/runner adapter가 소유한다. 단, worker가 결과를 정직하게 반환하는 단일 report
contract에는 `completed | context-requested` union을 둔다.

production runner는 child process 종료 뒤 result snapshot 전에 workspace root의 reserved
`WAYSTONE_RESULT.yaml`을 `WorkerResultAdapter`로 정확히 한 번 읽는다. adapter는
`waystone-worker-result-1`의 `completed | context-requested` union을 검증해 CAS artifact로 publish하고,
이 control file을 code/config result snapshot에서 제외한 뒤 additive `RunnerCompletionMarker v2`에
`worker_result_digest`를 결속한다. 누락·schema 오류·digest 불일치는 typed runner failure이며 stdout
자유문장을 추측하지 않는다. engine은 completion effect가 observed된 뒤에만 digest를 따라 union을
읽는다.

`context_request`는 다음 계약을 갖는다.

1. worker가 decision-critical context 부족을 발견하면 `question`, `blocked_decision`, `why_required`를
   반환하고 실행을 멈춘다.
2. engine은 frozen base와 worker result를 대조한다. code/config delta가 있으면
   `context_request_with_changes`로 거부한다. 부분 변경을 숨겨서 request 성공으로 만들지 않는다.
3. 정당한 request면 runner action은 effect observation까지 정상 종료하고 attempt는 terminal
   `context_requested`, run/job은 `waiting_context`로 전이한다. verifier, integration decision, apply는
   실행하지 않는다.
4. coordinator가 authority source를 찾아 `waystone-context-response-1`과 새 WorkBrief revision을
   발행한다. owner decision이 필요한 경우에는 `waiting_user(reason=owner-context-required)`로
   정직하게 승격한다.
5. response CAS는 run/job이 `waiting_context`이고 request가 current head이며 cancel intent가 없고
   attempt budget이 남았을 때만 성공한다. 성공하면 response, 새 WorkBrief, `supersedes_spec_digest`와
   `resolves_context_request_digest`를 가진 새 RunSpec을 한 transaction에 결속하고 attempt id
   `<run-id>:attempt:<sequence>`를 새로 만들어 `dispatch_ready → running`으로 간다.
6. context request attempt도 attempt budget을 소비하며 attempt/review budget을 초기화하지 않는다.
   cancel intent와 response는 같은 store CAS 경계에서 경쟁한다. cancel intent가 먼저 commit되면
   response는 `context_not_current`로 실패한다. response가 먼저면 새 attempt에 cancel을 적용한다.
   canceled/completed run 또는 이전 request에 대한 late response는 artifact/state를 만들지 않는다.

Routing compiler는 task 크기와 함께 context transfer loss를 본다. main의 tacit intent가 핵심이고
brief 작성 비용이 실행보다 크거나 손실을 닫을 수 없으면 `in-session`을 고른다. source refs로
충분히 전달 가능하고 scope가 bounded하면 `subagent`/`external`을 고른다. stage나 비용 판단이
profile의 declared binding을 다른 backend로 조용히 대체하지는 않는다.

### 3.5 OutcomeDelta와 status projection

Worker는 outcome을 제안할 수 있지만 progress authority를 갖지 않는다. coordinator는 stage별 completion
evidence와 exact result digest를 확인한 뒤 `waystone-outcome-delta-1`을 기록한다. Evaluate/Promote의
성질 주장은 independent verifier evidence가 없으면 delta로 승인할 수 없다. Explore의 candidate 존재
같은 기계적 사실은 harness observation으로 승인할 수 있고, 측정값 주장은 실제 measurement artifact를
요구한다.

`kind`는 다음 다섯 값이다.

```text
executable-capability
measured-improvement
validated-decision
simplification
no-objective-delta
```

각 terminal run은 정확히 하나의 primary delta를 기록한다. review 수정, docs 정리, gate 추가처럼 실행은
완료됐지만 objective가 달라지지 않은 작업도 `no-objective-delta`를 생략하지 않는다. 이 명시적 0이
review-remediation spiral을 status가 볼 수 있게 한다.

`run close`는 final attempt 아래 engine-owned `outcome-publication` action을 만든다. 그 action은
`closeout.yaml`과 `outcome.yaml`을 같은 add-only Git tree에 준비하고, 두 CAS input/artifact ref를
action transition에 결속한 뒤 기존 `GitRefEffect`로 `refs/waystone/outcomes`의 expected old OID를
first-parent CAS update한다. ref effect가 observed된 뒤에만 publication action과 run을 `completed`로
전이한다. prepared commit이나 한쪽 파일만 있는 commit, non-ledger branch의 파일은 progress가 아니다.
pair의 run id, final RunSpec/result/evidence digest가 모두 맞아야 accepted progress이며 불일치는
`closeout-incomplete` typed audit로 남긴다. OutcomeDelta를 closeout body에 편의 필드로 복제하지 않는다.

Status의 primary projection은 다음 순서다.

1. Project Brief status와 current objective fact.
2. active run의 lifecycle stage와 `waiting_context`/promotion readiness.
3. `refs/waystone/outcomes` tip에서 first-parent로 읽은 가장 최근 ledger commit의 OutcomeDelta. 한
   publication commit에는 run pair 하나만 허용하므로 timestamp/merge topology tie-break가 없다.
4. unresolved owner rulings와 promotion-blocking disposition.
5. `project-direction-may-be-stale` advisory와 근거.
6. task/job/test/finding count는 `Audit` 아래의 보조 수치.

advisory는 gate가 아니며 보편적 비율 threshold를 두지 않는다. 다음 직접 관측을 evidence로 표시할 수
있다: active work가 superseded/open fact에 의존, 여러 review-remediation ledger entry에
`no-objective-delta`가 반복됨, task closeout은 쌓였지만 그 사이 positive delta가 없음, 같은 assumption이
다시 열림, owner feedback와 current commitment 충돌. 단일 `no-objective-delta`만으로는 발행하지 않고
둘 이상의 관련 observation 또는 coordinator가 인용한 explicit owner conflict가 필요하다. status는
`/waystone:ideate` realignment를 제안할 뿐 자동 시작하거나 실행을 막지 않는다.

### 3.6 `ideate`: framing과 realignment

public surface는 `/waystone:ideate` 하나만 유지한다.

```text
PROJECT_BRIEF.md 없음  -> framing
PROJECT_BRIEF.md 있음  -> realignment
```

Framing은 commitments를 추측해 닫지 않는다. 대화 결과는 항상 `status: provisional`로 쓰며 owner가
별도 채택해야 `committed`가 된다.

Realignment는 질문 전에 다음 bounded evidence를 본다.

- positive OutcomeDelta와 `no-objective-delta`의 최근 흐름.
- feature/output과 review-remediation/docs 활동의 관계.
- 마지막 executable capability 변화 이후의 task·test·gate 변화.
- active objective와 committed fact의 연결.
- 반복 reopen/PARK와 같은 hypothesis/open question.
- current roadmap 임계경로가 실제 commitment에서 파생되는지.

질문은 한 turn에 하나씩 하며 3–6회 같은 고정 횟수를 두지 않는다. 종료 조건은 “남은 질문이
architecture detail일 뿐, 답에 따라 purpose/prototype/non-goal이 달라지지 않는다”이다. 기존 brief는
in-place 역사 삭제 대신 새 Git revision으로 갱신하고, 바뀐 commitment 때문에 active RunSpec이
stale해지면 자동 재해석하지 않고 새 run을 요구한다.

### 3.7 public CLI/skill mapping

새 개념을 숨은 파일 편집 절차로 남기지 않는다. canonical command는 다음처럼 좁힌다.

| Surface | 책임 |
|---|---|
| `waystone brief check [root]` | front matter, fact marker, status, duplicate와 binding 규칙 검증 |
| `waystone brief show [root] [--fact <id>]` | ProjectFrame 또는 한 fact의 authority/status/digest read-only 표시 |
| `waystone brief adopt [root] --evidence <file>` | exact owner adoption bytes를 CAS에 보존·결속한 뒤에만 provisional → committed 변경 |
| `waystone run start <task-id> --work-brief <file> [--owner-request <file>] [--stage ...]` | typed WorkBrief와 필요한 owner-source bytes를 import/검증하고 CompletionContract·AssurancePlan·RunSpec을 freeze한 뒤 production assembly로 실행 |
| `waystone run context show <run-id>` | pending request와 필요한 decision을 read-only 표시 |
| `waystone run context provide <run-id> --response <file>` | response/brief/spec revision을 발행하고 새 attempt로 resume |
| `waystone run close <run-id> --outcome <file>` | coordinator의 typed OutcomeDelta를 frozen completion/evidence에 대조하고 publication action을 발행 |
| `waystone review ingest <run-id>` | feedback byte-exact 보존과 immutable finding claim 생성 |
| `waystone review validate <finding-id> --file <file>` | validity/failure mechanism revision 기록; priority와 task 생성 없음 |
| `waystone review disposition <finding-id> --file <file>` | confirmed validation에 결속된 impact/relevance/disposition revision 기록 |
| `waystone review materialize <finding-id>` | latest disposition이 허용한 selected work만 `tasks.yaml`에 추가 |
| `waystone status [--project ...]` | objective/stage/delta/advisory primary projection |

`/waystone:init`, `/waystone:ideate`, `/waystone:run`, `/waystone:review`, `/waystone:status`는 이
command를 orchestration하는 얇은 skill이다. skill이 별도 state machine이나 사실 권위를 만들지 않는다.
`brief adopt`, `context provide`, risk acceptance처럼 owner authority가 필요한 action은 skill의 대화
결과만으로 우회하지 않고 해당 typed gate를 통과한다.

CLI ingress는 path를 권위로 저장하지 않는다. file bytes를 먼저 CAS에 import하고 schema와 declared
digest를 검증한 뒤 reference id/digest만 RunSpec 또는 action에 결속한다. `--owner-request`는
WorkBrief의 `objective_ref.kind: owner-request`일 때 필수이며 exact digest가 맞아야 한다. project-fact/
milestone objective에는 금지한다. `--stage`는 WorkBrief stage를 override하는 knob가 아니라 동일성을
확인하는 optional assertion이다. `run close --outcome`의 objective/run/spec/result/evidence가 frozen
lineage와 다르면 publication 전에 거부한다. skill은 이 typed file을 대화에서 조립할 수 있지만 CLI가
없는 hidden in-memory authority를 만들 수 없다.

## 4. 스키마와 포맷 초안

아래 예시는 field semantics를 고정하는 v1/v2 초안이다. 실제 JSON Schema는 해당 구현 task가 이
문서와 1:1로 작성한다. canonical JSON은 M1-B의 sorted-key compact UTF-8 규칙을 재사용한다.

### 4.1 `waystone-project-brief-1`

```markdown
---
schema: waystone-project-brief-1
status: provisional
---
# <Project>

## Purpose
<설명 맥락. hard acceptance source가 아님>

## Commitments
- [commitment/primary-user] ...
- [commitment/desired-outcome] ...

## Prototype scope
- [prototype/first-supported-result] ...

## Long-term direction
- [long-term/future-scale] ...

## Non-goals
- [non-goal/not-a-platform] ...

## Working hypotheses
- [hypothesis/solver-family] ...

## Open questions
- [question/required-accuracy] ...

## Revision triggers
- [trigger/user-feedback-conflict] ...
```

계약:

- front matter exact key는 `schema`, `status`이며 canonical current file의 status는
  `provisional | committed`다. historical revision의 `superseded`는 Git ancestry와 current brief ref로
  reader가 파생한다.
- fact marker grammar는
  `(commitment|prototype|long-term|non-goal|hypothesis|question|trigger)/[a-z0-9-]+`이고 문서 내
  unique다.
- parser는 marker가 붙은 list item의 raw UTF-8 byte span을 digest한다. Unicode/case를 normalize하지
  않는다.
- section과 marker prefix가 맞지 않거나 duplicate/missing marker가 있으면 typed refusal한다.
- narrative paragraph는 context일 수 있으나 `ProjectFactRef` 대상이 아니다.

### 4.2 authority ref, completion contract, candidate/evaluation

Objective는 한 종류의 ref로 위장하지 않고 다음 discriminated union을 쓴다.

```yaml
objective_ref:
  kind: project-fact              # project-fact | owner-request | milestone
  commit: "<sha>"                 # project-fact/milestone
  path: PROJECT_BRIEF.md
  fact_id: hypothesis/solver-family
  fact_digest: "sha256:..."
  binding: nonbinding             # nonbinding | binding
# owner-request이면 {kind, artifact_reference_id, digest, binding}
# milestone이면 {kind, commit, path, item_id, item_digest, binding}
```

`project-fact`는 fact bytes, `owner-request`는 host가 보존한 exact owner-source artifact,
`milestone`은 Git-tracked selected-work item을 authority로 한다. accepted ADR은 fixed decision source가 될
수 있지만 objective를 대신하지 않는다. task title/acceptance 자유문장은 독립 objective authority가
아니다. Explore는 nonbinding hypothesis/question을 objective로 쓸 수 있지만 learning completion만
허용한다. Promote objective는 binding commitment/prototype 또는 exact owner request여야 한다.

`AuthorityRef`는 이 `ObjectiveRef` 세 variant에 criterion 전용
`accepted-adr {commit,path,decision_id,digest}`, `evaluation-spec {commit,path,generation,digest}`,
`evaluation-evidence {reference_id,candidate_digest,generation,digest}`를 더한 closed union이다. 모든 Git
variant는 full commit/path/item id/bytes digest를, CAS variant는 reference id/digest를 요구한다.
CompletionContract의 `source`만 전체 `AuthorityRef`를 쓸 수 있고 `objective_ref`는 앞의 세 variant로
제한한다.

```yaml
schema: waystone-completion-contract-1
lifecycle_stage: promote
objective_ref: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
                fact_id: commitment/desired-outcome, fact_digest: "sha256:...", binding: binding}
criteria:
  - id: supported-result
    mode: promotion               # learning | evaluation | promotion
    text: "supported scope에서 결과를 재현한다"
    source: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
             fact_id: prototype/first-supported-result, fact_digest: "sha256:..."}
    binding: binding
    evidence: {kind: regression-contract, reference_id: "regression:<id>", digest: "sha256:..."}
compiled_by: {rule_id: stage-completion-v1, compiler_digest: "sha256:..."}
```

각 criterion은 `mode/text/source/binding/evidence`를 모두 가진다. compiler는 Explore의 nonbinding
hypothesis/open question을 `mode: learning`으로만 허용하며 “답 또는 candidate를 기록했다”를 완료로
삼는다. Evaluate는 frozen evaluation spec criterion만 `mode: evaluation`으로 쓴다. Promote의
`mode: promotion`은 committed commitment/prototype/non-goal, exact owner request, accepted ADR의
constraint, 또는 같은 candidate generation의 passed evaluation evidence에서만 컴파일한다. task의
자유문장과 hypothesis/open question은 hard criterion으로 거부한다.

```yaml
schema: waystone-evaluation-spec-1
evaluation_id: "<uuidv7>"
generation: 3
objective_ref: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
                fact_id: commitment/desired-outcome, fact_digest: "sha256:...", binding: binding}
criteria:
  - {id: accuracy, metric: exact-match, operator: gte, threshold: 0.90}
datasets:
  - {id: holdout, artifact_reference_id: "evaluation-dataset:<id>:3", digest: "sha256:...",
     visibility: harness-only}
seed: 42
supersedes_spec_digest: "sha256:..."
```

```yaml
schema: waystone-candidate-1
candidate_id: "<uuidv7>"
producer: {run_id: "<uuid>", run_spec_digest: "sha256:...", result_digest: "sha256:..."}
code_sha: "<full-sha>"
config_digest: "sha256:..."
target_ref: "refs/waystone/candidates/<run-id>"
target_oid: "<full-sha>"
supersedes_candidate_digest: null
repair_of_finding_refs: []
```

```yaml
schema: waystone-evaluation-evidence-1
candidate_digest: "sha256:<waystone-candidate-1>"
evaluation_spec_digest: "sha256:<waystone-evaluation-spec-1>"
evaluation_generation: 3
evaluator_action_id: "<action-id>"
result: pass                       # pass | fail
metric_artifacts: [{criterion_id: accuracy, reference_id: "...", digest: "sha256:..."}]
```

candidate ref OID, descriptor `code_sha`, result/config digest가 모두 일치해야 evaluate 입력이 된다.
Evaluation spec은 결과를 보기 전에 Git commit/path/digest로 freeze한다. `harness-only` dataset bytes는
trusted deterministic evaluation harness만 opaque input로 읽을 수 있고 implementer, model evaluator,
reviewer prompt에는 노출하지 않는다. 그 경계를 넘어 노출되면 generation은 `invalidated` evidence로
닫고 같은 threshold를 고쳐 재사용하지 않는다. 새 dataset/spec digest와 증가한 generation으로만
재평가할 수 있다. Promote는
candidate descriptor와 그 **동일 generation**의 passed evidence digest를 freeze한다.

### 4.3 `waystone-work-brief-1`

```yaml
schema: waystone-work-brief-1
brief_id: "<uuidv7>"
task_id: "<task-id>"
revision: 1
supersedes_digest: null
resolves_context_request_digest: null
lifecycle_stage: explore
objective:
  ref:
    kind: project-fact
    commit: "<full-sha>"
    path: PROJECT_BRIEF.md
    fact_id: hypothesis/solver-family
    fact_digest: "sha256:..."
    binding: nonbinding
  desired_delta: "후보 두 개의 실제 차이를 측정한다"
  why_now:
    text: "..."
    provenance: coordinator-summary
    sources: [{kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
               fact_id: hypothesis/solver-family, fact_digest: "sha256:...", binding: nonbinding}]
current_state:
  - text: "현재 baseline은 X에서 실패한다"
    provenance: harness-observation
    source: {kind: git, commit: "<sha>", path: "...", digest: "sha256:..."}
  - text: "실패 원인은 Y로 보인다"
    provenance: coordinator-summary
    sources: [{kind: evidence, digest: "sha256:..."}]
decisions:
  fixed: [{text: "...", provenance: owner-source,
           source: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
                    fact_id: commitment/example, fact_digest: "sha256:...", binding: binding}}]
  worker_may_choose: [{text: "...", provenance: coordinator-summary, sources: []}]
  requires_escalation: [{text: "대규모 architecture fork", provenance: owner-source,
                         source: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
                                  fact_id: non-goal/no-fork, fact_digest: "sha256:...",
                                  binding: binding}}]
constraints:
  - {text: "public API는 바꾸지 않는다", provenance: owner-source,
     source: {kind: project-fact, commit: "<sha>", path: PROJECT_BRIEF.md,
```

Explore의 `candidate/evaluation`은 null이다. Evaluate의 `candidate`는
`{reference_id, digest, target_ref, target_oid, code_sha, config_digest, producer_result_digest}` exact
descriptor이고 `evaluation.spec`은 `{commit, path, digest, generation}`이며 evidence는 null이다.
Promote는 같은 candidate/spec tuple에 `evaluation.evidence: {reference_id, digest, generation}`를 추가하고
passed evidence 일치를 요구한다. `review_decision`은 더 이상 RunSpec에 독립적으로
손작성하지 않고 `assurance_plan` 안의 stage/risk 결과로 결속한다. context resume revision은 같은
run/job을 유지하되 새 brief/assurance/spec digest와 attempt를 가진다. `completion_contract`는
Explore에서는 learning/candidate completion signal, Evaluate에서는 frozen evaluation criterion,
Promote에서는 committed fact·owner request·accepted ADR/evaluation spec에서 유도한 hard acceptance를
담는다. hypothesis/open question을 Promote hard criterion으로 자동 승격할 수 없다. `result_policy`는
stage에 따라 `candidate-ref | evidence-only | integration-ref` 중 하나이며, `integration-ref`는
`target_ref`와 `expected_oid`가 모두 필수다. ref update는 기존 effect/CAS 경로만 사용한다.

각 completed review action은 `{promotion_lineage_id, cycle, target_result_digest, review_digest,
supersedes_digest}`인 `waystone-review-cycle-1` CAS evidence를 append한다. 새 candidate/RunSpec도 같은
promotion intent를 계속하면 root objective/target tuple, lineage id와 두 chain head를 승계하고 compiler가
`consumed_cycles`를 재계산한다. Explore는 독립 research라면 `promotion_lineage: null`일 수 있지만,
promotion blocker 수리를 위한 Explore는 기존 lineage와 `supersedes_candidate_digest`를 필수로 가진다.
owner가 별도 research track을 승인한 경우에만 새 lineage를 만들 수 있다. 숫자를 caller가 낮추거나
result digest를 바꿔 budget을 reset할 수 없다.

`run_spec_digest`는 self-referential body field가 아니다. ArtifactStore reference envelope가 canonical
RunSpec bytes의 digest를 소유하며 모든 consumer가 그 envelope와 bytes를 대조한다. `job_input.input_digest`
는 current v1과 같이 `input_digest` field를 제외한 `job_input` canonical subobject bytes의 digest다.

### 4.9 store v2 최소 확장

M1-B의 entity 구조와 CAS 알고리즘은 유지하고 fresh-install schema를 v2로 올린다. migration 구현은
하지 않는다.

- `ArtifactReferenceKind`에 `input`, `outcome`을 추가한다. imported owner request, WorkBrief,
  CompletionContract, AssurancePlan, evaluation spec, context response와 소비되는 candidate descriptor는
  해당 planning/evaluator transition의 `input`이다. worker result, context request, 생산된 candidate
  descriptor/result, evaluation evidence, verification, review-cycle record는 생산 action의 `evidence`, integration decision은
  `decision`, OutcomeDelta는 `outcome-publication` action의 `outcome`이다. 같은 candidate bytes가 producer
  action에서는 evidence이고 evaluator action에서는 input이므로 reference identity/transition은 별도다.
  Git-tracked finding record는 이 local artifact enum에 중복 등록하지 않는다.
- Run/job state에 `waiting_context`를 추가하고 transport `IdleReason`에
  `run_waiting_context`를 추가한다.
- attempt terminal state/reason에 `context_requested`, run/job `TransitionReason`에
  `context-requested`, `context-provided`를 추가한다. `RunnerCompletionMarker v2`는
  `worker_result_digest`를 필수로 가진다.
- final attempt에 `outcome-publication` action을 추가한다. `input/outcome` refs는 그 action의 planned/observed
  transition에 결속하고 `refs/waystone/outcomes` effect observation 전 run completion을 금지한다.
- RunSpec reference id는 revision을 포함한 `run-spec:<run-id>:<revision>`으로 바꾼다. v1 ref를 새
  revision에 덮어쓰지 않는다.
- v1 database를 만났을 때 silent upgrade/fallback하지 않고 `schema_version_unsupported`로 거부한다.

## 9. Exit와 검증 전략

### 9.1 핵심 e2e 7개

아래 7개가 0.13 product contract다. 각 항목은 대표 happy path 한 개와 그 문장을 판별하는 최소
negative assertion만 둔다. stage/model/profile 조합 matrix를 만들지 않는다.

1. **Semantic prompt:** architecture task worker가 why/current state/fixed decisions/freedom/source
   provenance를 받지만 tasks.yaml, round, overlay, lease, artifact naming protocol은 받지 않는다.
2. **Context resume:** worker가 decision-critical context 부족을 보고하면 code/config delta 없이
   `waiting_context`가 되고, response 뒤 새 WorkBrief digest + RunSpec revision + attempt로 재개한다.
   cancel/late-response race의 패자는 `context_not_current`이며 state/artifact를 남기지 않는다.
3. **Truth ≠ priority:** `confirmed major` finding을 `accept-risk` 또는 `backlog`로 처분할 수 있고
   explore run이나 unrelated work를 자동 차단하지 않는다.
4. **Promotion boundary:** 같은 finding이 committed promotion contract를 위반해
   `fix-before-promotion`이면 exact promotion lineage의 promote만 실제 차단한다. 새 candidate/run으로
   우회해도 review-cycle count와 blocker는 유지되고 failure mechanism의 independent fix evidence로만
   clearance된다.
5. **Cheap exploration:** explore worker run이 locally reachable immutable candidate와 temporary probe 또는
   refuted hypothesis로 완료되며 permanent test, ADR, independent verifier, macro review를 강제하지 않는다.
6. **Expensive promotion:** read-only evaluate가 같은 candidate/spec generation evidence를 만든 뒤 promote
   run은 implementer mutation 없이 committed frame, frozen evaluation evidence, 최소 regression
   contract, 필요한 reviewer evidence, supported scope/accepted risk record 없이는 apply/complete되지
   않는다. hypothesis/open question/task 자유문장을 promotion criterion으로 compile하면 typed refusal한다.
7. **Outcome progress:** task/finding/test 수가 늘어도 latest delta가 `no-objective-delta`면 status가
   그것을 progress라 부르지 않고 outcome ledger의 current objective, stage, last positive delta와 반복
   evidence가 있을 때만 advisory를 보여준다.

### 9.2 위험 비례 검증

- Store/lease/effect/supervisor/cancel을 건드리지 않은 task는 해당 전체 module suite를 반복 실행하지
  않는다. store v2 enum/state seam처럼 실제로 건드린 부분만 기존 fault fixture를 재실행한다.
- `context_request`가 새 state/effect 경계를 만들므로 crash-before/after publication 두 지점과 stale
  role `binding_digest` 한 건은 kernel-level focused regression을 둔다. 그 이상 조합은 실제 failure evidence가
  생길 때 추가한다.
- Finding/Outcome Git publication은 add-only CAS, digest tamper, divergent validation/disposition head 각 한 건으로
  authority를 검증한다.
- 실제 backend smoke는 semantic architecture brief 한 건만 실행한다. model/effort/backend matrix를
  만들지 않는다.
- 이 재설계 자체는 trust/completion/review authority를 바꾸므로 gate에서 independent adversarial
  review 한 번이 필수다. finding은 새 disposition flow로 처리하며 review가 새 task 목록을 자동
  생성하지 않는다.
- legacy 1,099 suite의 pass count는 보고할 수 있으나 exit 조건도 회귀 목표도 아니다.

### 9.3 Gate 산출 evidence

Gate report는 다음만 남긴다.

- e2e 1–7 각각의 command, exit code, 관측된 result/artifact digest.
- touched trust seam의 focused regression command와 결과.
- real backend smoke의 RunSpec/WorkBrief/OutcomeDelta digest.
- 독립 review finding 수와 각 latest disposition.
- 삭제 대상 파일이 존재하지 않고 보존 kernel 파일에 unrelated rewrite가 없다는 diff audit.

테스트 수, assertion 수, review finding 0개 자체는 성공 지표로 사용하지 않는다.

## 10. 열린 질문 — owner/main ruling 필요

각 항목은 구현을 멈추게 하지 않도록 잠정안을 함께 둔다. 해당 task의 RunSpec freeze 전에는 ruling을
확정해야 한다.

### Q1. `Prototype scope` fact는 독립 authority인가, commitment subtype인가?

- **왜 설계자가 확정하지 않았나:** mandate §4의 표는 acceptance 변환을 허용하지만 같은 절의 핵심
  계약은 commitment/non-goal만 authoritative하다고 적어 문언 범위가 다르다.
- **잠정 선택:** committed `Prototype scope` fact를 `commitment` subtype으로 보고 hard acceptance
  source로 허용한다. provisional 상태에서는 금지한다.
- **필요 ruling:** 이 해석을 확정하거나, prototype fact마다 owner가 별도 `binding`을 선언하게 한다.

### Q2. `PROJECT_BRIEF.md`의 `committed` 전환을 무엇이 owner adoption으로 증명하는가?

- **왜 설계자가 확정하지 않았나:** mandate는 명시적 사용자 채택을 요구하지만 Git author, host
  consent event, 대화 artifact 중 어느 것이 owner authority인지 정하지 않는다. actor principal
  결속도 현행 별도 open issue다.
- **잠정 선택:** host가 보존한 exact owner confirmation digest를 `brief commit` action에 입력하고,
  그 digest를 status-field 변경 commit의 sidecar에 결속한다. 단순 agent edit나 Git author 문자열은
  충분하지 않다.
- **필요 ruling:** machine-local consent로 충분한지, Git-tracked adoption sidecar가 필요한지.

### Q3. promotion-bound `accept-risk`를 coordinator가 단독 결정할 수 있는 최대 범위는 어디까지인가?

- **왜 설계자가 확정하지 않았나:** mandate는 coordinator disposition을 허용하지만 architectural
  확대는 owner decision으로 두며, major trust/public-contract risk의 최종 수용 주체는 명시하지 않는다.
- **잠정 선택:** `blocker`, `current-objective`, `promotion-bound`, trust/public-contract risk의
  `accept-risk`는 owner만 결정한다. future/out-of-scope edge/minor는 coordinator가 근거와 함께 결정할
  수 있다.
- **필요 ruling:** owner-only 경계의 severity/relevance/risk 조건 확정.

### Q4. finite review-cycle 기본 제안값 2를 Waystone 기본으로 둘 것인가?

- **왜 설계자가 확정하지 않았나:** mandate의 2회는 예시이고 omniphysics에서는 채택값이지만,
  universal hard number는 다시 Goodhart를 만들 수 있다.
- **잠정 선택:** engine implicit default는 없고 Assurance compiler가 Evaluate/Promote에 2를 제안해
  freeze한다. project policy/owner가 dispatch 전에 변경할 수 있다.
- **필요 ruling:** canonical proposal 2를 유지할지, project가 매번 명시하도록 할지.

### Q5. OutcomeDelta의 `recorded_by`를 coordinator로 고정할 것인가?

- **왜 설계자가 확정하지 않았나:** mandate는 outcome schema와 evidence는 정하지만 worker proposal,
  verifier 판정, coordinator progress 결정 사이의 최종 actor를 명시하지 않는다.
- **잠정 선택:** worker는 proposal만 하고 coordinator가 기록한다. Evaluate/Promote의 성질 주장은
  independent verifier evidence가 필수이며 verifier 자신은 progress를 결정하지 않는다.
- **필요 ruling:** owner가 직접 delta를 기록하는 별도 path와 reviewer의 관여 여부.

### Q6. 0.13에서 durable actor principal 인증까지 닫을 것인가?

- **왜 설계자가 확정하지 않았나:** M1-B의 actor separation과 profile binding은 역할/실행 경계를
  검증하지만 `recorded_by`, `validated_by`, `decided_by`, `issued_by` 문자열을 실제 사람·agent identity에
  cryptographically 결속하지 않는다. 이를 닫으려면 host identity/서명 authority라는 새 trust surface가
  필요하며 mandate는 그 체계를 지정하지 않았다.
- **잠정 선택:** 0.13은 single-coordinator trust domain으로 한정하고 actor record를
  `{role, binding_digest, principal: null}`로 저장한다. null을 인증된 principal처럼 표시하지 않으며
  multi-actor/remote authority에는 사용하지 않는다.
- **필요 ruling:** 이 한계를 accepted risk로 채택할지, 0.13 범위를 늘려 host-signed principal과 key
  lifecycle을 설계할지.

## 11. 구현 중 금지선

- open question을 worker가 추측해 닫지 않는다.
- stage를 낮춰 failed promotion을 explore success로 재분류하지 않는다.
- review requirement를 피하려고 scope/result를 잘게 쪼개거나 finding을 `no-action`으로 자동 처분하지
  않는다.
- context request에 partial patch를 숨기거나 이전 attempt budget을 초기화하지 않는다.
- status advisory를 자동 task, blocker, realignment 실행으로 승격하지 않는다.
- legacy alias/read fallback으로 canonical format failure를 성공처럼 보이지 않는다.
- 새 schema field, status enum, finding taxonomy마다 matrix test를 추가하지 않는다.
- 반복 evidence 없이 generic capability graph, policy DSL, metric engine, review workflow subsystem을
  새로 만들지 않는다.

이 금지선과 D1–D18, §7의 7 task, §9의 e2e 7개가 0.13 구현의 전체 분해 권위다.
